Chapter 3. The Relational Model

Data independence

One of the most practical gifts the relational model gave database engineering is a guarantee you probably already rely on without realizing it: you can reorganize how data is stored, or restructure how it is presented to one group of users, without rewriting the applications that use it. That guarantee has a name — data independence — and it is one of the primary reasons relational databases became the dominant model for business data.

Two Kinds of Independence

Edgar Codd, who formalized the relational model in 1970, identified two distinct levels of independence. Each one protects a different audience from a different kind of change.

Kind What is shielded What can change underneath
Physical data independence Applications and queries How data is stored on disk (files, indexes, page layout)
Logical data independence External views and applications The logical schema (adding tables, splitting a table, renaming columns)

Think of it as two insulating layers. Physical independence sits at the bottom, separating the storage engine from the relational schema. Logical independence sits above that, separating the schema from the views and queries that users write.

Physical Data Independence

Physical data independence means that the way data is physically stored — which files it lives in, what page size is used, which indexes exist — can change without affecting the SQL you write.

You saw a glimpse of this in the Logical vs. Physical Structure page. Adding or dropping an index is the clearest example: the SELECT statement and its result stay exactly the same; only the internal access path changes. A DBA can add an index on a busy production table to fix a slow query, and not a single line of application code needs to touch.

Other physical changes that should not break your SQL:

  • Moving a database to faster SSD storage
  • Changing the internal page size or buffer pool size
  • Partitioning a large table across multiple physical files
  • Switching the storage engine (e.g., from one B-tree format to another)

The database engine's query optimizer is the mechanism that makes this work. When you submit a query, the optimizer consults the current physical layout and generates an efficient execution plan. As the layout changes, the optimizer generates new plans — automatically, transparently.

Logical Data Independence

Logical data independence is the harder of the two. It means that changes to the relational schema — the tables and their columns — should not break the views and queries that sit on top of it.

The usual mechanism is a view: a named, stored query that looks like a table to any code that uses it.

Imagine a users table that is later split into two tables for organizational reasons:

-- Original schema
CREATE TABLE users (
  id      INTEGER PRIMARY KEY,
  name    TEXT,
  email   TEXT,
  country TEXT
);

-- After refactoring: split into two tables
CREATE TABLE user_accounts (id INTEGER PRIMARY KEY, name TEXT, email TEXT);
CREATE TABLE user_locations (user_id INTEGER PRIMARY KEY, country TEXT);

-- A view that reconstructs the original interface
CREATE VIEW users AS
  SELECT a.id, a.name, a.email, l.country
  FROM user_accounts a
  JOIN user_locations l ON l.user_id = a.id;

Any application that was querying SELECT name, email FROM users WHERE country = 'TH' keeps working without modification. The view acts as an adapter between the new schema and the old interface.

Note: Logical data independence is harder to achieve in full because views over complex schema changes (like splitting a column or merging tables) become difficult to keep writable. Read-only views are easy; updatable views that preserve the original behavior require more care.

Why This Matters in Practice

Data independence is not just a theoretical elegance — it solves real problems:

  1. Schema evolution without big-bang deployments. As a system grows, the schema needs to change. With views and a well-designed schema boundary, you can migrate tables incrementally and keep old application code running while the new code adopts the new structure.

  2. Performance tuning without coordination. A DBA can add indexes, reorganize storage, or enable table partitioning during off-peak hours without scheduling a coordinated release with every development team.

  3. Multiple audiences, one source of truth. Different teams (analytics, operations, external partners) can be given views tailored to their needs — different column names, filtered rows, aggregated data — while the underlying tables remain unchanged.

The relational model earns this by strictly separating what data means (the schema and the algebra) from how it is stored (the engine's internals). That separation is not accidental; Codd put it at the center of the design. It is why a query you write today against a well-designed schema can still run correctly years later, even after the hardware, the storage engine, and several rounds of schema refactoring have come and gone.
