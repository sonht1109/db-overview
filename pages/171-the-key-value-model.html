Part V. Database Families

Chapter 18. Key-Value Databases

The key-value model

At its core, a key-value database is the simplest data model imaginable: a giant, persistent dictionary. You hand it a key, it hands back a value. That is the entire contract. No schemas to define, no joins to write, no query planner to second-guess. This simplicity is both its superpower and its boundary.

The Model in One Sentence

Every record is a key → value pair stored in a hash-map-like structure. The key is a unique identifier (a string or binary blob). The value is whatever you want — a string, a number, a serialized JSON object, a binary blob, a counter. The database stores it and retrieves it without inspecting its contents.

Concept What it is Example
Key Unique lookup handle — you choose the format session:u4291:token
Value Opaque payload — any bytes you want to store {"userId":4291,"expires":1751000000}
Namespace / prefix A convention (not enforced by all engines) for grouping keys cache: · session: · rate:

Note: "Opaque" means the database does not parse or index the value's contents. It stores bytes in, bytes out. If you need to search inside values, you must handle that in your application or choose a different model (e.g., a document store).

Structure and Key Design

Because there is no schema and no query language beyond GET key / SET key value, key naming becomes your schema. Teams adopt conventions like colon-separated namespaces to create logical groupings:

session:user:4291          → {"token":"abc123","created":1750900000}
rate_limit:user:4291:day   → 47
product:sku:NB-001         → {"name":"Notebook","price":2.99}
cache:homepage:v3          → <rendered HTML>

This is a deliberate design choice. The database is blind to the colon; it is pure convention. But a consistent scheme lets you reason about key space, set expiry by prefix in some engines, and avoid collisions across different parts of an application.

Core Operations

The API is deliberately tiny. Most key-value stores expose only a handful of verbs:

Operation Meaning
GET key Retrieve the value for a key (or null if absent)
SET key value Write or overwrite a key-value pair
DEL key Delete a key
EXISTS key Check whether a key is present
EXPIRE key seconds Schedule automatic deletion after a TTL

Some engines add richer primitives (atomic counters, lists, sorted sets), but the lookup-by-exact-key model stays at the center. There is no WHERE price < 5 — you either know the key or you do not.

Note: The EXPIRE / TTL (time-to-live) feature is what makes key-value stores ideal for caches and sessions: data evicts itself automatically, so you never accumulate stale records.

Where the Model Fits (and Where It Does Not)

Strong fits:

  • Session storage — look up session:<id> on every request; expire automatically after inactivity.
  • Caching — store pre-computed results under a predictable key; invalidate or let them expire.
  • Rate limiting — increment an integer counter under rate:<user>:<window>; check before serving.
  • Feature flags — feature:<flag_name> → true/false, read on every render.

Poor fits:

  • Queries that filter or search inside values (a document store or relational DB handles this better).
  • Relationships between records that you need to traverse (a graph DB or relational FK handles this).
  • Ad-hoc analytics across the full dataset (a column store or data warehouse handles this).

The key-value model is intentionally narrow. It trades expressiveness for speed and operational simplicity — and in the right problem domain, that trade is an excellent one.
