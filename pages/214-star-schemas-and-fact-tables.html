Part V. Database Families

Chapter 22. Analytical Databases and Warehouses

Star schemas and fact tables

Analytical queries ask different questions than transactional ones. Instead of "fetch order #4821," they ask "what was total revenue by region last quarter?" To answer that efficiently, the data must be organized around measurements and their context — not around the entities of daily operations. The star schema is the most widely used approach to this kind of organization, and understanding it unlocks almost everything about how data warehouses are designed.

The Star Schema

A star schema places one large fact table at the center, surrounded by several smaller dimension tables that radiate outward like points of a star. Every row in the fact table records a single business event — a sale, a page view, a support call — along with numeric measurements and foreign keys into the dimension tables that describe who, what, when, where, and why.

Fact Tables

The fact table is the heart of the schema. Every row represents a single grain — the finest unit of measurement the schema tracks. Choosing the grain is one of the most important design decisions you will make.

Column Type Role
sale_id BIGINT Surrogate primary key
date_key INT FK → dim_date
product_key INT FK → dim_product
customer_key INT FK → dim_customer
store_key INT FK → dim_store
sale_amount DECIMAL(12,2) Additive measure
quantity INT Additive measure
discount_pct DECIMAL(5,2) Semi-additive measure

Additive measures (revenue, quantity) can be summed across any dimension. Semi-additive measures (inventory balance, discount percentage) can be summed across some dimensions but not all — summing a discount percentage across customers makes no sense.

Fact tables are typically very wide (50–200 columns) and very deep (hundreds of millions to billions of rows). They dominate warehouse storage and are the primary target of columnar compression.

The Grain

The grain is stated as a sentence: "One row per line item on a sales order." If the grain is ambiguous or mixed, aggregations produce wrong answers. Common grains:

  • One row per daily product inventory snapshot
  • One row per web session event
  • One row per financial transaction

Finer grain = more flexibility but more storage. Coarser grain = less storage but less analytical depth.

Dimension Tables

Dimension tables hold the descriptive context around each fact. They are typically small — thousands to a few million rows — and deliberately denormalized. Instead of a separate city table joined through a region table joined through a country table, all geography is flattened into a single dim_geography table.

-- Example dimension table: all location attributes in one flat table
CREATE TABLE dim_store (
  store_key   INT PRIMARY KEY,       -- surrogate key (not operational ID)
  store_id    VARCHAR(20),           -- natural / business key
  store_name  VARCHAR(100),
  city        VARCHAR(100),
  state       VARCHAR(50),
  region      VARCHAR(50),
  country     VARCHAR(50),
  sq_footage  INT,
  opened_date DATE
);

Surrogate Keys

Dimension tables use surrogate keys — system-generated integers — rather than natural business keys. This insulates the warehouse from operational system changes (a product SKU can change; the surrogate key never does) and enables slowly changing dimensions.

Slowly Changing Dimensions (SCDs)

When a customer moves from New York to London, which city should historical sales show? SCD strategies handle this:

Type Strategy Trade-off
Type 1 Overwrite old value Simple; loses history
Type 2 Insert new row with new key Full history; table grows
Type 3 Add "previous value" column Limited history; compact

Type 2 is the most common: the dimension table gets valid_from and valid_to dates, and the fact table's foreign key points to the specific version of the row that was current at event time.

Why Denormalization is Intentional

In OLTP design, normalization reduces redundancy and update anomalies. In OLAP design, the calculus flips:

Joins are expensive on billions of rows. Denormalization replaces runtime joins with storage — and storage is cheap. The goal is to touch as few column segments as possible per query.

A normalized "snowflake" variant does exist (discussed below), but the flat star schema is often preferred because:

  1. Fewer joins means simpler query plans
  2. Columnar engines can filter and aggregate a dimension column with a single segment read
  3. Query writers don't need to know the operational normalization structure

The Snowflake Schema

In a snowflake schema, dimension tables are themselves normalized into sub-dimensions. dim_product might have a foreign key to dim_category, which joins to dim_department.

fact_sales
  └── dim_product
        └── dim_category
              └── dim_department

Advantages: Less storage redundancy, easier dimension maintenance. Disadvantages: More joins at query time, more complex SQL, harder for BI tools to auto-generate queries.

Most modern warehouses prefer the star schema or a hybrid. Snowflake schemas are more common when dimensions are extremely large or when the hierarchy has many levels.

Key Takeaways

  • A star schema has one fact table at the center surrounded by flat dimension tables — shaped like a star.
  • Fact tables store measurements (additive or semi-additive) plus foreign keys; they are wide and deep, often billions of rows.
  • Dimension tables store descriptive attributes; they are small, denormalized, and accessed via surrogate keys.
  • The grain defines what each fact row represents — getting it right is the first step in warehouse design.
  • Denormalization is intentional in OLAP: fewer joins means faster full-scan aggregations on columnar storage.
  • The snowflake schema normalizes dimension tables for reduced redundancy, at the cost of more joins.
  • Slowly changing dimensions (SCD Type 2) preserve history by inserting new rows rather than overwriting old ones.
