Part VI. Engineering Practice

Chapter 23. Designing for Performance

Latency, throughput, and concurrency

Performance conversations collapse quickly into three intertwined numbers: latency, throughput, and concurrency. Confuse them and you'll optimise the wrong thing — reducing latency when you need more throughput, or raising concurrency when the bottleneck is single-thread speed. Getting precise about each term is the foundation of every performance decision in this chapter.

Latency: How Long One Request Takes

Latency is the elapsed time between a client sending a request and receiving the final byte of the response. It is measured per-request and usually reported as a distribution, not a mean.

The mean hides the tail. A query that averages 5 ms but has a 99th-percentile of 800 ms will feel broken to 1 in 100 users. Database engineers track:

Percentile Common shorthand Why it matters
p50 median Typical experience
p95 — Most users' worst case
p99 "nines" SLA boundary in most contracts
p99.9 p999 Worst-case under load

Latency Budget

Every user-facing request has a budget — say 200 ms. That budget is split across network hops, application code, and database queries. A single slow query blows the whole budget. That's why even a "fast" 50 ms average query is unacceptable if it sits inside a budget of 100 ms.

Throughput: How Many Requests Per Second

Throughput is the rate of completed work — requests per second, rows read per second, transactions per minute. It is a system-level property, not a per-request one.

Latency and throughput are related but not the same:

  • A system can have low latency and low throughput — each query is fast but the server handles only one at a time (no concurrency).
  • A system can have high throughput and high latency — a batch ETL job processes millions of rows per second but each individual query takes seconds.

The relationship is approximated by Little's Law:

Throughput = Concurrency / Latency
  (requests/second = concurrent requests / seconds per request)

If your database handles 50 concurrent queries and each takes 0.1 s, throughput is 500 req/s. To double throughput you can either halve latency or double concurrency — but concurrency has a ceiling set by your hardware.

Concurrency: How Much Work Happens in Parallel

Concurrency is the number of requests in flight simultaneously. The database manages concurrency through:

  • Connection pooling — reusing a fixed set of connections rather than opening a new one per request.
  • Thread / worker pools — each connection typically maps to an OS thread or async task.
  • Lock management — concurrent writers must be serialised on the same row; concurrent readers can proceed in parallel under MVCC.

The Concurrency Ceiling

Every system has a point where adding more concurrency hurts throughput. Past that point, the overhead of context-switching, lock contention, and cache thrashing grows faster than the gain from parallelism. Profiling tools expose this as CPU saturation or lock-wait time climbing steeply.

Putting It Together

Metric Unit Tuning lever
Latency ms per request Indexes, query rewrites, caching
Throughput requests / second Horizontal scaling, pooling, batching
Concurrency simultaneous requests Pool size, async I/O, partitioning

Key takeaway: Always measure latency as a distribution (p95/p99), not a mean. Use Little's Law to reason about throughput vs concurrency trade-offs. A fast-average, slow-tail query is a production incident waiting to happen.
