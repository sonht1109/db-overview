Part IV. Distributed Systems

Chapter 14. From One Machine to Many

Cross-node queries

Once your data is spread across multiple nodes — whether by sharding, federation, or a mix of both — a new class of problem appears: what happens when a single query needs data that lives on more than one machine? That's a cross-node query, and handling it correctly is one of the harder engineering challenges in distributed databases.

Why Cross-Node Queries Are Hard

On a single machine, a join is straightforward: the engine reads both tables from local disk, builds a hash table or merge buffer in RAM, and returns the result. When the two tables are on different servers, none of that applies. The engine must:

  1. Decide which node coordinates the query (the query coordinator).
  2. Send sub-queries or data requests to the relevant nodes (remote scan).
  3. Receive partial results over the network.
  4. Combine (merge, join, aggregate) those partial results locally.
  5. Return the final answer to the client.

Every one of these steps adds latency, and the network — unlike a local memory bus — can be slow, lossy, and unpredictable.

The Scatter-Gather Pattern

The most common execution model for cross-node queries is scatter-gather:

  • Scatter: The coordinator broadcasts a sub-query to all (or several) shards in parallel.
  • Gather: Each shard executes its piece and sends back partial results.
  • Merge: The coordinator combines the partial results into a final answer.

Scatter-gather works well for simple aggregations (COUNT, SUM, MAX) because each shard computes a partial aggregate and the coordinator combines them cheaply. It works poorly for large cross-shard joins, because the coordinator may have to receive millions of rows from multiple shards and join them in memory — doing the heavy lifting that each individual node could have done locally.

Cross-Shard Joins in Practice

Consider an e-commerce system where orders is sharded by order_id and customers is sharded by customer_id. A query like:

SELECT c.name, COUNT(o.order_id) AS order_count
FROM   customers c
JOIN   orders o ON o.customer_id = c.customer_id
GROUP  BY c.name;

...is a cross-shard join. A given customer's row lives on one shard; their orders may be scattered across several others. Distributed databases handle this in a few ways:

Strategy How it works Best when
Broadcast join Coordinator sends one (small) table to every shard; each shard joins locally One side of the join is small enough to fit in memory
Repartition join Both tables are reshuffled so matching keys land on the same node, then joined Neither side is small; keys are well-distributed
Co-location Data is designed upfront so related rows share a shard key (e.g., customer_id) Query patterns are known at design time

Co-location is the most efficient — it eliminates network traffic entirely for the join — but it requires thinking about your access patterns before you shard.

Note: Systems like Google Spanner, CockroachDB, and YugabyteDB have built-in query planners that choose between these strategies automatically. Older hand-rolled sharding setups leave this to the application layer.

Distributed Aggregations

Aggregates split cleanly across shards when they are decomposable — meaning the coordinator can combine partial results from each shard without seeing every row:

  • COUNT(*) → sum the per-shard counts.
  • SUM(amount) → sum the per-shard sums.
  • MAX(price) → take the max of per-shard maxes.
  • AVG(score) → not directly decomposable. Each shard must return (sum, count) separately; the coordinator divides the totals. A naive per-shard average averaged together gives the wrong answer.

COUNT(DISTINCT …) is especially painful: exact distinct counts require shipping all values to one place (or using an approximation like HyperLogLog).

The Cost to Keep in Mind

Cross-node queries introduce three costs that don't exist locally:

  1. Network latency — even fast data-center networks add milliseconds per hop.
  2. Data transfer volume — shuffling millions of rows between nodes for a join is expensive.
  3. Coordination overhead — the coordinator must track partial results, handle partial failures, and retry timed-out shards.

The practical takeaway: design your shard key so that the most frequent, most latency-sensitive queries touch as few shards as possible. Cross-node queries are sometimes unavoidable, but they should be the exception, not the rule.

Reveal: Why can't you just run ORDER BY … LIMIT 10 on each shard and merge the tops?
Because the true global top 10 might not appear in any single shard's top 10. Imagine three shards each returning their top 10 rows — that's 30 rows total. The coordinator must sort all 30 and take the top 10. The safe rule: each shard must return the top N × number_of_shards rows (or the coordinator must do a full merge-sort). This is why OFFSET-based pagination is especially expensive in sharded systems — deep pages require fetching and discarding huge numbers of rows from every shard.
