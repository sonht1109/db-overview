Dirty reads and non-repeatable reads

When multiple transactions run at the same time, they can step on each other in subtle ways. Two classic problems are dirty reads and non-repeatable reads — both arise from insufficient isolation, and both can silently corrupt application logic without triggering any error.

Dirty Reads: Seeing Uncommitted Data

A dirty read happens when Transaction A reads a row that Transaction B has modified but not yet committed. If B later rolls back, A was working with data that never actually existed in the database.

Example scenario:

Step Transaction A (bank teller) Transaction B (transfer job)
1 UPDATE accounts SET balance = 0 WHERE id = 1
2 SELECT balance FROM accounts WHERE id = 1 → sees 0
3 ROLLBACK — balance reverts to original
4 A acts on a balance of 0 that never existed

Transaction A made a decision based on phantom data. This is a dirty read.

Note: Most databases prevent dirty reads by default. In SQL standard terms, this requires at least Read Committed isolation level. The lowest level, Read Uncommitted, explicitly allows dirty reads — it is rarely used in production.

Non-Repeatable Reads: Data That Changes Under You

A non-repeatable read is subtler. Transaction A reads a row, then reads it again later in the same transaction — and gets a different value because Transaction B committed a change in between.

Example scenario:

Step Transaction A (report job) Transaction B (admin update)
1 SELECT price FROM products WHERE id = 7 → $100
2 UPDATE products SET price = 150 WHERE id = 7
3 COMMIT
4 SELECT price FROM products WHERE id = 7 → $150
5 A sees two different prices in one logical operation

Transaction A asked the same question twice and got two different answers. This breaks the assumption that a transaction sees a stable snapshot of the database.

Why It Matters

In a pricing report, two reads of the same product during one transaction should agree. Inconsistency here can produce wrong totals, incorrect audit logs, or subtle application bugs that are hard to reproduce.

Preventing non-repeatable reads requires Repeatable Read isolation level, which guarantees a transaction always sees the same value for a row it has already read — even if a concurrent transaction commits a change.

Isolation Levels at a Glance

Isolation Level Dirty Reads Non-Repeatable Reads Phantom Reads
Read Uncommitted possible possible possible
Read Committed prevented possible possible
Repeatable Read prevented prevented possible
Serializable prevented prevented prevented

Each level adds stronger guarantees at the cost of reduced concurrency (more locking or version tracking). Choosing the right level is always a tradeoff.
