Part VII. Building Understanding

Chapter 28. Case Studies

Mistakes and redesigns

Every database has a history of decisions that seemed right at the time but turned out to be wrong. Studying those mistakes is one of the fastest ways to build good instincts. This page catalogues the most common database design errors — drawn from real-world post-mortems — and the redesigns that fixed them.

Mistake 1: Storing Multiple Values in a Single Column

What happened: A developer needed to tag articles with multiple categories. The quick solution: store tags as a comma-separated string in a tags TEXT column.

-- Wrong
CREATE TABLE articles (id INTEGER PRIMARY KEY, title TEXT, tags TEXT);
INSERT INTO articles VALUES (1, 'Intro to SQL', 'database,tutorial,beginner');

Why it breaks: Every operation becomes painful. Finding all "database" articles requires WHERE tags LIKE '%database%' — a full table scan that also matches "non-database". Counting uses of a tag is a string parsing exercise. Renaming a tag means string replacement across millions of rows.

The redesign: A proper junction table.

CREATE TABLE article_tags (
  article_id INTEGER REFERENCES articles(id) ON DELETE CASCADE,
  tag_id     INTEGER REFERENCES tags(id),
  PRIMARY KEY (article_id, tag_id)
);

Now finding all "database" articles is a single indexed join, and renaming a tag is one UPDATE.

Mistake 2: Using OFFSET for Pagination

What happened: An API endpoint loaded page N of results with LIMIT 20 OFFSET (N-1)*20. Works fine at page 1; becomes slower with every page.

-- Slow at large offsets
SELECT * FROM events ORDER BY created_at DESC LIMIT 20 OFFSET 10000;

Why it breaks: The database must read and discard 10 000 rows to reach page 501. At OFFSET 1 000 000, the query takes seconds.

The redesign: Cursor-based pagination. The client receives the last seen ID (or timestamp) and uses it as a filter:

-- Fast regardless of depth
SELECT * FROM events
WHERE created_at < :last_seen_ts
ORDER BY created_at DESC
LIMIT 20;

The index on created_at allows the database to jump directly to the right position without scanning.

Mistake 3: Storing Money as REAL (Float)

What happened: A billing system stored prices in a REAL column. Totals accumulated rounding errors. An invoice for $19.99 × 12 = $239.88 sometimes computed as $239.87999999…

-- Wrong
CREATE TABLE invoices (id INTEGER PRIMARY KEY, amount REAL);

The redesign: Store money as integer cents (or the smallest currency unit):

CREATE TABLE invoices (id INTEGER PRIMARY KEY, amount_cents INTEGER);
-- $239.88 = 23988

Format for display in the application layer. Integer arithmetic is exact; floating-point is not.

Mistake 4: Using NULLs to Model State Machines

What happened: An order's lifecycle was modeled as a series of nullable timestamp columns:

CREATE TABLE orders (
  id INTEGER PRIMARY KEY,
  placed_at   INTEGER,
  confirmed_at INTEGER,  -- NULL until confirmed
  shipped_at   INTEGER,  -- NULL until shipped
  delivered_at INTEGER   -- NULL until delivered
);

Why it breaks: Determining current status requires checking which columns are non-NULL. Invalid states are possible (e.g., delivered_at set but shipped_at still NULL). Adding a new state requires a schema migration.

The redesign: A single status column and a separate status_history table for auditing:

CREATE TABLE orders (
  id     INTEGER PRIMARY KEY,
  status TEXT NOT NULL DEFAULT 'pending'
  -- 'pending' | 'confirmed' | 'shipped' | 'delivered' | 'cancelled'
);

CREATE TABLE order_status_history (
  id        INTEGER PRIMARY KEY,
  order_id  INTEGER REFERENCES orders(id),
  status    TEXT NOT NULL,
  changed_at INTEGER NOT NULL,
  changed_by INTEGER REFERENCES users(id)
);

Current state is always unambiguous. History is queryable. New states require only inserting into order_status_history, not a schema change.

Mistake 5: Missing an Index on a Foreign Key

What happened: An app stored comments with a post_id foreign key. Loading comments for a post worked fine during development with 100 rows. In production with 10 million rows, it took 4 seconds.

-- Missing index!
CREATE TABLE comments (
  id      INTEGER PRIMARY KEY,
  post_id INTEGER REFERENCES posts(id),
  body    TEXT
);
-- SELECT * FROM comments WHERE post_id = 42; -- full table scan!

The redesign: Add the index:

CREATE INDEX idx_comments_post ON comments (post_id);

The query drops from 4 seconds to 2 milliseconds. In SQLite (and most databases), a foreign key constraint does not automatically create an index — you must do it explicitly.

Key Takeaways

  • Never store multiple values in one column — junction tables exist for this reason.
  • Cursor pagination (WHERE id < ?) is always faster than OFFSET for deep pages.
  • Integer cents are the correct way to store money in any relational database.
  • State machines belong in an explicit status column with a history table, not in a field of nullable timestamps.
  • Foreign keys do not automatically create indexes — always add one explicitly on the child side.
