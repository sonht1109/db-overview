Chapter 6. How Query Processing Works

Rule-based and cost-based planning

Once the optimizer has a logical plan, it faces a decision: which of the many equivalent ways to execute that plan is the fastest? Historically, databases answered this question in two very different ways — one by following hand-crafted rules, the other by estimating the cost of every realistic option. Most modern engines blend both approaches, so understanding the distinction helps you read query plans and write queries that the optimizer can handle well.

Rule-Based Optimization (RBO)

Early query optimizers — most famously Oracle before version 7 — used a fixed catalog of rules to rewrite and rank plans. The rules were deterministic: given this kind of query shape, always do X.

Common examples of rule-based transformations:

Rule What it does
Push selections down Move WHERE filters as close to the table scan as possible
Eliminate redundant projections Drop columns that are not needed further up the tree
Flatten subqueries Rewrite a correlated subquery as a join when safe to do so
Use an available index Prefer an index scan over a full scan whenever a matching index exists

The big advantage of RBO is predictability: the same query always produces the same plan, which makes behavior easy to reason about. The fatal weakness is that rules ignore data. A rule might say "use the index on customer_id," but if the orders table has only 50 rows, a full scan is faster than chasing index pointers. RBO has no way to know that.

Cost-Based Optimization (CBO)

Cost-based optimizers, pioneered by System R at IBM in the 1970s and now the default in PostgreSQL, MySQL, SQL Server, and virtually every serious engine, take a different approach. Instead of following fixed rules, the optimizer:

  1. Enumerates candidate plans — different join orderings, different access methods (index vs. full scan), different join algorithms (nested loop, hash join, sort-merge join).
  2. Estimates the cost of each plan using statistics: table row counts, column cardinality, histograms of value distributions, and correlation between columns.
  3. Picks the plan with the lowest estimated cost, measured in units like I/O operations or CPU cycles.

The statistics that drive this are stored in the catalog and updated by commands like ANALYZE (PostgreSQL/SQLite) or UPDATE STATISTICS (SQL Server). Stale statistics are one of the most common causes of bad query plans in production.

Note: SQLite's optimizer is a simplified CBO. It tracks row counts and some column statistics, and it can choose between index scans and full table scans based on those numbers. Running ANALYZE in SQLite updates these statistics in the sqlite_stat1 table.

How Cost Estimates Drive Plan Choice

Consider a join between orders (large) and customers (small). The optimizer considers options like:

Option A: Hash-join(orders, customers)  -- build hash table on customers (small side)
Option B: Nested-loop(customers, orders) -- for each customer, scan orders
Option C: Index-lookup on orders.customer_id -- follow index pointers per customer

Without statistics, a rule might blindly pick the index. With statistics, the optimizer sees that the customer_id filter is highly selective and the customers table is small, making the hash join cheapest. That is cost-based reasoning.

RBO and CBO Together

In practice, modern engines layer both approaches. Rule-based transformations happen first — the optimizer applies algebraic rewrites (push predicates down, eliminate unnecessary joins) that are always beneficial regardless of data. Only after that structural cleanup does the cost model kick in to choose among the remaining physical options.

This hybrid approach means:

  • Deterministic rewrites (RBO) handle structural improvements cheaply and predictably.
  • Cost estimation (CBO) handles data-sensitive choices like join order and access paths.

Understanding which layer is responsible for which decision is the key to diagnosing slow queries: if a structural rewrite did not fire, check whether your query can be expressed more cleanly; if the wrong physical plan was chosen, check whether your statistics are fresh.

Reveal: Why might a query suddenly run slower after a data load?
The cost-based optimizer made its plan choice using statistics collected when the table was smaller. As data grows, old statistics describe a different distribution — a column that was once highly selective may now return a large fraction of rows, making an index scan more expensive than a full scan. Running ANALYZE (or the equivalent for your database) refreshes these statistics so the optimizer can make an accurate new decision. Routine statistics maintenance is a standard DBA task in production systems for exactly this reason.
