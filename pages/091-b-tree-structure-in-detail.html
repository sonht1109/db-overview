Chapter 10. Trees, Logs, and Internal Structures

B-tree structure in detail

B-trees are the dominant index structure in relational databases — PostgreSQL, MySQL/InnoDB, SQLite, Oracle, and SQL Server all use them as the default for primary keys and secondary indexes. Understanding their shape explains why indexed lookups are fast, why range scans are cheap, and why unordered bulk inserts can stress a database.

The Shape of a B-tree

A B-tree organizes data into a balanced hierarchy of pages (also called nodes). Every path from the root to any leaf is exactly the same number of hops — that is what "balanced" guarantees. There are three kinds of nodes:

                     [ 30 | 60 ]          ← root (internal node)
                    /     |     \
          [10|20]  [40|50]  [70|80]       ← internal nodes
          /  |  \   ...
    [1-9][11-19][21-29] ...               ← leaf nodes (hold actual rows / row pointers)
Node type What it stores Purpose
Root Keys + child pointers Entry point; always kept in memory
Internal Keys + child pointers Routing — no row data
Leaf Keys + row data (or row pointers) Where actual lookups land

The order (or branching factor) of a B-tree is the maximum number of children per internal node — typically 100–500 in practice, because each node maps to one disk page (~8 KB) and keys are small. With a branching factor of 200 and 3 levels, a tree can index 200³ = 8 million rows while reading only 3 pages to find any one of them.

Note: The variant used by most databases is actually a B+ tree: internal nodes store only keys (no row data), and all row data lives in the leaf nodes. Leaf nodes are also linked together in a doubly-linked list, making range scans very efficient.

Finding a Row: The Search Algorithm

A point lookup walks the tree top-down. At each internal node the engine binary-searches the keys to find the right child pointer, loads that child page, and repeats until it reaches a leaf.

Search for key = 45:
  Root [30 | 60]   → 30 < 45 < 60, follow middle child
  Internal [40|50] → 40 < 45 < 50, follow middle child
  Leaf [41|43|45]  → found at slot 2

Cost: O(log_B n) page reads, where B is the branching factor and n is the number of rows. For a million-row table with B = 200, that is about log₂₀₀(1 000 000) ≈ 2.9 — so at most 3 page reads, regardless of table size.

Range scans are equally efficient: find the starting leaf, then follow the leaf-level linked list forward until the range is exhausted — no need to climb back up the tree.

Insertions and Page Splits

Inserting a new key always targets a leaf. If the leaf has room, the key is inserted in sorted order and the page is written back. If the leaf is full, a page split occurs:

  1. A new sibling page is allocated.
  2. The keys are divided roughly 50/50 between the old and new page.
  3. The median key is promoted to the parent as a new separator.
  4. If the parent is also full, it splits too — the process can cascade all the way to the root.
Leaf [10|20|30|40] is full. Insert 25:
  → Split into [10|20] and [25|30|40]
  → Promote 25 to parent

Splits are the main reason heavily write-loaded tables periodically need index maintenance (REINDEX, REBUILD, or the equivalent): many splits over time can leave pages that are only half-full, wasting space and adding extra I/O per scan.

Note: Deletions can cause the mirror operation — an underflow where a page falls below a minimum fill threshold. Engines handle this by either merging two half-empty siblings or redistributing keys between them, then removing the separator from the parent.

Fill Factor and Real-World Tuning

Production engines expose a fill factor setting (PostgreSQL: fillfactor, SQL Server: FILL FACTOR). A fill factor of 80% tells the engine to leave 20% of each leaf page empty at build time. This reserves space for future inserts on already-existing pages, reducing splits on write-heavy tables at the cost of a slightly larger index. For read-mostly tables, 100% packs data tighter and reduces I/O per scan.

The key intuitions to carry forward:

  • Height stays tiny — branching factor keeps trees 2–4 levels deep for virtually any realistic dataset.
  • Leaves are linked — range scans never backtrack through internal nodes.
  • Splits propagate upward — a write-heavy workload with many inserts in key order (auto-increment IDs) always appends to the rightmost leaf and rarely splits, while random-key inserts (UUIDs) cause frequent splits throughout the tree.
