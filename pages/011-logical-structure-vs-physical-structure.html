Chapter 2. How Data Is Organized

Logical structure vs physical structure

When you write a query, you think in terms of tables, rows, and columns. The database engine, meanwhile, is thinking about bytes, pages, and disk seeks. These two worlds — what the data means and how it is stored — are deliberately kept separate. Understanding that separation is one of the most clarifying ideas in all of database engineering.

The Logical Structure: What You See

The logical structure is the data model as the user perceives it. It answers the question: what does the data look like and how does it relate?

In a relational database, the logical structure is defined by your schema — the tables, columns, data types, primary keys, and foreign-key relationships you declare. When you write SELECT name, email FROM users WHERE active = 1, you are reasoning entirely at the logical level. You do not care whether users lives in one file or ten, or whether the rows are stored in alphabetical order or at random.

Layer Vocabulary Who cares?
Logical Tables, rows, columns, keys, views, joins Application developers, data analysts
Physical Pages, blocks, B-trees, heap files, indexes, WAL Database engine, DBAs tuning performance

This split is sometimes called data independence: changing how data is stored should not break the SQL that reads it.

The Physical Structure: What Actually Happens on Disk

The physical structure is how the database engine materializes your logical tables into bytes. Every major relational engine uses a similar set of primitives:

  • Pages (or blocks): Storage is divided into fixed-size chunks — typically 4 KB to 16 KB. Reading or writing always happens at the page boundary, even if you only need one row.
  • Heap files: The simplest physical layout — rows are written to pages in insertion order, with no guarantee about their sequence.
  • B-tree indexes: A sorted tree structure built alongside a heap. Lookups by an indexed column skip most pages and jump straight to the right leaf. This is physical machinery; it is invisible to a SELECT statement.
  • Write-ahead log (WAL): Before changing a page, the engine records the change in a sequential log file. This is how crash recovery works — again, entirely hidden from SQL.

Note: SQLite, which powers the interactive examples in this guide, stores an entire database in a single file divided into 4 KB pages. Every table and index lives somewhere in that file. You never see pages when you write SQL, but they explain why an INSERT can be slow on a nearly full page.

Seeing the Separation in Action

The clearest demonstration is an index. An index is a purely physical artifact — it does not change what rows the database contains, only how fast it can find them. From a logical standpoint, the table before and after CREATE INDEX is identical. From a physical standpoint, a new B-tree structure now exists on disk.

Why the Distinction Matters

Keeping logical and physical concerns separate pays off in three concrete ways:

  1. Portability. The same SQL works on SQLite, PostgreSQL, and MySQL even though each engine uses a very different physical layout under the hood. Your application code does not change when you switch engines.

  2. Tuning without breakage. A DBA can add an index, change a storage engine, or move a table to faster disks without touching a single line of application SQL. The logical interface stays stable.

  3. Reasoning clearly about bugs. When a query returns the wrong rows, the problem is almost always logical (wrong join, wrong filter). When a query returns correct rows but runs slowly, the problem is almost always physical (missing index, table scan on millions of rows). Knowing which layer to inspect saves hours of debugging.

The rest of Part I focuses on the logical level — schemas, relationships, and queries. Later chapters on indexing and storage engines will pull back the curtain on the physical side. For now, the key takeaway is simple: your SQL describes what you want, and the database engine decides how to get it.
