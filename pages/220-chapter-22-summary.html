Part V. Database Families

Chapter 22. Analytical Databases and Warehouses

Chapter 22 Summary: What We Learned

Chapter Contents

Page Topic Takeaway in One Sentence
211 OLTP vs OLAP OLTP = many small, fast transactions (order entry). OLAP = few large, complex queries (business reporting). Different engines for each.
212 Columnar storage Values for each column are stored together — scanning a few columns on billions of rows reads only relevant data.
213 Compression and vectorized execution Columnar data compresses extremely well (run-length, dictionary, delta); the engine processes batches of values with SIMD instructions.
214 Star schemas and fact tables A fact table contains measurements (sales, clicks); dimension tables describe the context (time, product, customer). Querying is straightforward.
215 Batch loading Analytical databases are optimized for bulk ingestion (ETL/ELT pipelines), not real-time row-by-row inserts.
216 Materialized views Pre-computed query results that are updated incrementally or on refresh — turn hours-long queries into sub-second lookups.
217 Data lakes and lakehouses Data lakes store raw files (Parquet, Iceberg) in object storage; lakehouses add ACID transactions and SQL on top.
218 Why analytics engines feel different No indexes, no single-row updates, massive parallelism, different SQL dialects — they are built for scanning, not seeking.
219 Good use cases Business intelligence dashboards, periodic reports, machine learning feature engineering, log analysis, financial reconciliation.
