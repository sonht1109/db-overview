Part V. Database Families

Chapter 20. Column-Family Databases

Wide rows and sparse data

Column-family databases take a fundamentally different approach to organizing data than the relational model. Instead of storing data in rigid tables where every row has exactly the same columns, they allow wide rows — rows that can carry hundreds or even thousands of columns — and they are sparse, meaning a row only stores the columns it actually has. This page explores what that means in practice, why it matters for certain workloads, and how the data model maps onto systems like Apache Cassandra and Apache HBase.

What Is a "Wide Row"?

In a relational table, every row has exactly the same set of columns. If a column has no value for a given row, you store NULL. The schema is fixed: add a column and you touch every row.

A wide row works differently. Think of a row as an ordered map of column name → value pairs. One row might have 3 columns; the next row in the same table might have 3,000 columns. The columns themselves can differ between rows. There is no NULL stored for missing entries — a column simply does not exist for that row.

This is a crucial distinction:

Concept Relational Column-Family
Schema Fixed — same columns for every row Flexible — each row defines its own columns
Missing data NULL stored (wastes space) Column absent (nothing stored)
Row width Narrow, bounded by schema Wide, potentially thousands of columns
Column definition DDL (ALTER TABLE ADD COLUMN) Implicit — just write to it
Suitable for Structured, uniform entities Event streams, sensor data, timelines

The Sparsity Advantage

Imagine you are modeling user activity on a platform. A user can perform 500 different action types. In a relational model you might add 500 boolean or timestamp columns. Most users perform a handful of action types, so 98% of those columns would be NULL.

In a column-family database, you write only the columns for the actions a user actually performed. A user who did five things has a row with five columns. A power user who did 400 things has a row with 400 columns. Storage is proportional to the actual data, not the schema footprint.

Key insight: Sparsity is the default, not an exception. Column-family stores are built to store tens of millions of rows each with a different column set efficiently. NULLs are not "stored as NULL" — they are simply absent.

The Hierarchy: Keyspace → Table → Partition → Row → Column

Column-family systems organize data in a consistent hierarchy. Using Cassandra's terminology:

Keyspace  (≈ database / schema in SQL)
  └── Table  (≈ table, but with a wide-row data model)
        └── Partition  (a logical group of rows with the same partition key)
              └── Row  (identified by the primary key within a partition)
                    └── Column  (a name-value pair, only present if data exists)

This hierarchy determines both how data is stored on disk and how it is distributed across nodes. A partition is the unit of distribution — all rows in the same partition live on the same node(s).

Column Families

In HBase (and Cassandra's early design), columns are grouped into column families — a fixed set of column groups defined at table creation. Within a column family, individual column qualifiers can vary per row.

Table: user_activity
  Column Family: "events"
    Row "user:1001"  →  events:login=2024-01-15, events:purchase=2024-01-20
    Row "user:1002"  →  events:login=2024-01-16, events:search=2024-01-16
  Column Family: "profile"
    Row "user:1001"  →  profile:name=Alice, profile:country=US
    Row "user:1002"  →  profile:name=Bob

Column families matter physically: each family is stored in a separate file (HFile in HBase, SSTable in Cassandra). Reads that only need one column family don't touch the files of another, which can dramatically reduce I/O.

In modern Cassandra (version 3+), the data model is closer to SQL in syntax — you define columns in a CREATE TABLE statement — but the storage model remains sparse and wide rows are still possible through collection types and user-defined types.

Modeling Wide Rows: A Practical Example

Consider a system that stores time-series IoT sensor readings. Different sensors report different metrics. In a relational model:

-- Relational: every sensor gets NULLs for metrics it doesn't report
CREATE TABLE sensor_readings (
  sensor_id    TEXT,
  ts           TIMESTAMP,
  temperature  FLOAT,  -- NULL for sensors that don't report this
  humidity     FLOAT,  -- NULL for sensors that don't report this
  pressure     FLOAT,  -- NULL for sensors that don't report this
  voltage      FLOAT,  -- NULL for sensors that don't report this
  PRIMARY KEY (sensor_id, ts)
);

In a column-family model using CQL (Cassandra Query Language):

-- Cassandra: only store what you have
CREATE TABLE sensor_readings (
  sensor_id TEXT,
  ts        TIMESTAMP,
  metric    TEXT,
  value     FLOAT,
  PRIMARY KEY (sensor_id, ts, metric)
);
-- Each "row" in the partition is effectively a column in the wide-row model

Or using a map type to embed the sparse columns directly:

CREATE TABLE sensor_readings (
  sensor_id TEXT,
  ts        TIMESTAMP,
  readings  MAP<TEXT, FLOAT>,   -- sparse: only present keys stored
  PRIMARY KEY (sensor_id, ts)
);

What Column-Family Systems Are Not

It is easy to conflate "column-family" with "column-store" (OLAP). They are different things:

  • A column-store (like Parquet, DuckDB, Redshift) stores data column-by-column on disk, enabling compression and vectorized reads across all rows for a given column. It is optimized for analytics.
  • A column-family store (Cassandra, HBase) groups data by row and partition key. Within a partition, columns are stored together. It is optimized for low-latency reads and writes by key.

The word "column" in each name refers to completely different ideas.

Key Takeaways

  • Wide rows allow a single row to have thousands of columns, with different rows having different columns.
  • Sparsity means only present columns are stored — there are no NULLs consuming space.
  • The hierarchy is: keyspace → table → partition → row → column.
  • Column families group physically related columns into separate files, enabling selective I/O.
  • Column-family databases are distinct from column-stores (OLAP engines) — the word "column" has different meanings in each context.
  • Best fit: event streams, sensor data, time-series, user activity tracking — workloads where schema uniformity across rows is not a requirement.
