Part III. Storage and Execution

Chapter 9. Pages, Records, and Files

Fixed-length and variable-length records

Every row in a table must eventually live somewhere on disk. How the database engine lays out the bytes for that row is determined by whether its fields have fixed or variable lengths — a design decision with real consequences for storage efficiency, read speed, and update cost.

Fixed-Length Records

A fixed-length record is one where every field occupies exactly the same number of bytes, regardless of the actual data stored. If a CHAR(20) column is declared, the engine always writes 20 bytes — padding with spaces (or zeros) when the actual value is shorter.

Column Declared type Bytes on disk Example value
employee_id INTEGER 4 1042
department_id INTEGER 4 7
salary REAL 8 82500.00
status_code CHAR(1) 1 'A'

Total record size: 17 bytes, always. Every record in this table is the same width.

This uniformity is a big win for the storage engine:

  • Random access by row number — to find record n, seek to byte offset n × 17. No scanning needed.
  • Simple updates in place — a new value for salary fits in the same 8 bytes, so the engine overwrites without moving anything.
  • No metadata overhead — no need to store lengths or offsets per field.

The cost is wasted space when values are short. A CHAR(200) column storing mostly short names wastes hundreds of bytes per row.

Variable-Length Records

A variable-length record contains at least one field whose on-disk size changes with the data — typically columns declared as VARCHAR(n), TEXT, or BLOB. The engine stores only the bytes actually needed, plus a small amount of bookkeeping so it can find each field.

A common layout stores a slot directory (or "record header") at the start of the record: a compact list of (offset, length) pairs — one per variable field — followed by the actual data bytes:

[ offset_1 | len_1 | offset_2 | len_2 | ... | data_1 | data_2 | ... ]

For example, storing name = 'Ali' and bio = 'Loves hiking' might look like:

[ 4 | 3 | 7 | 12 | A | l | i | L | o | v | e | s |   | h | i | k | i | n | g ]
  ^offset  ^len  ^offset  ^len   ^---name---^  ^-----------bio-----------^
Advantage Trade-off
Compact storage — short strings use few bytes Cannot seek to a field by offset alone
Handles unbounded text naturally Updates that grow a value may force a record to move
Works well for sparse, irregular data Small per-field header overhead

Note: Most real engines use a hybrid: fixed-length fields (integers, dates, floats) are stored at known offsets at the start of the record, and variable-length fields are appended after them. This keeps fast access for common numeric fields while still supporting text columns.

Why This Matters for Query Performance

The record layout affects more than storage size. When the engine scans a table page looking for rows that match a WHERE clause:

  • With fixed-length records, it can jump directly to any row's field using arithmetic — no parsing required.
  • With variable-length records, it must consult the slot directory for each row before it can read a specific field.

This is one reason why columns used in hot filter conditions (WHERE status = 'A') are often kept as fixed-width types, while free-text columns (bio, notes) are allowed to vary. The trade-off between compactness and access speed shapes table design from the very first CREATE TABLE.
