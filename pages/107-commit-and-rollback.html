Part III. Storage and Execution

Chapter 11. Transactions

Commit and rollback

Every transaction lives in one of two states: in progress or ended. Ending a transaction is a deliberate choice — you either tell the database to make all your changes permanent (COMMIT) or to erase them entirely (ROLLBACK). Those two commands are the on/off switch for every guarantee that transactions provide.

What Commit and Rollback Actually Do

When you open a transaction with BEGIN, the database engine starts tracking every change you make — which rows you inserted, updated, or deleted. Until you commit, those changes live in a private workspace that only your session can see. Other sessions see the old data; you see your in-progress version.

COMMIT tells the engine: I am done, make this real. The engine flushes the changes to durable storage (typically by writing a record to the write-ahead log first), releases any locks, and makes the new data visible to every other session. Once a commit returns successfully, the changes survive crashes.

ROLLBACK tells the engine: Discard everything since BEGIN. The engine undoes every change in the transaction — new rows are deleted, modified rows are restored to their previous values — and the database is left exactly as it was before the BEGIN. No trace of the aborted work remains.

Implicit vs. Explicit Transactions

Most databases support two modes:

Mode How it works When to use
Explicit You write BEGIN … COMMIT/ROLLBACK yourself Multi-statement operations that must be atomic
Auto-commit Every statement is its own one-statement transaction Quick, standalone reads and single-row writes

Note: Auto-commit is on by default in most database clients. Every INSERT, UPDATE, or DELETE you run without an explicit BEGIN is automatically committed the moment it succeeds. This is convenient but means there is no safety net — a mistaken DELETE without a WHERE clause is permanent as soon as it runs.

Always use explicit transactions when multiple statements need to succeed or fail together.

Automatic Rollback on Error

You do not always have to call ROLLBACK yourself. If a statement inside a transaction causes an error (a constraint violation, a type mismatch, a deadlock), the database marks the transaction as aborted. Any further statements in the same transaction will be rejected, and when you eventually call COMMIT — or when the connection drops — the engine rolls back automatically.

BEGIN;

INSERT INTO orders (id, customer_id, total) VALUES (101, 9, 250.00);

-- This statement fails: customer_id 999 does not exist (foreign key violation).
INSERT INTO orders (id, customer_id, total) VALUES (102, 999, 75.00);

-- The transaction is now aborted. Even calling COMMIT here rolls back.
COMMIT;  -- behaves like ROLLBACK — neither row is saved

This all-or-nothing behavior is what "atomicity" means in practice: a half-finished transaction cannot be committed.

Savepoints: Partial Rollback

Full rollback is all-or-nothing, but SQL also provides savepoints — named checkpoints within a transaction that you can roll back to without aborting the whole thing.

BEGIN;

INSERT INTO audit_log (msg) VALUES ('started');

SAVEPOINT before_risky_step;

UPDATE inventory SET qty = qty - 1000 WHERE product_id = 5;  -- might be wrong

-- Oops, that was too many. Roll back just to the savepoint:
ROLLBACK TO SAVEPOINT before_risky_step;

UPDATE inventory SET qty = qty - 10 WHERE product_id = 5;   -- correct amount

COMMIT;  -- the audit log insert AND the corrected update are committed

Savepoints are useful in long transactions where one optional step can fail without invalidating earlier work — common in batch imports and stored procedures.
