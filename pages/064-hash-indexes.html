Chapter 7. Indexes

Hash indexes

A B-tree index is great for range queries, but it is doing more work than necessary when all you ever ask is "give me the row where id = 42." A hash index is built for exactly that case — pure equality lookups — and it does so with O(1) average-case time instead of O(log n).

How a Hash Index Works

The idea is borrowed directly from hash tables in programming. When a row is inserted, the database applies a hash function to the indexed column's value. The hash function maps that value to a bucket — a slot in a fixed array. The bucket stores a pointer to the actual row on disk.

hash("alice@example.com")  →  bucket 7  →  [row pointer]
hash("bob@example.com")    →  bucket 2  →  [row pointer]
hash("carol@example.com")  →  bucket 7  →  [row pointer, row pointer]  ← collision

At query time the engine hashes the search value, jumps directly to that bucket, and returns the matching row pointers — no tree traversal required.

Operation B-tree index Hash index
Equality (= 42) O(log n) O(1) average
Range (BETWEEN 10 AND 50) O(log n + k) Not supported
ORDER BY / sorting Supported Not supported
LIKE 'abc%' prefix Supported Not supported
Space overhead Moderate Low to moderate

Note: Hash indexes trade range capability for raw equality speed. If your workload is overwhelmingly point lookups (session tokens, UUIDs, email addresses), a hash index can be the right choice. If you need ordering or ranges even occasionally, a B-tree is safer.

Collisions and Growth

No hash function is perfect — two different values can land in the same bucket (a collision). The common fix is chaining: each bucket holds a short linked list, and the engine scans the list for an exact match. With a healthy load factor (entries ÷ buckets) kept below ~0.75, chains stay short and lookups remain fast. When a table grows past that threshold, the index is rehashed — all entries redistributed into a larger bucket array.

Database Support and Caveats

Not every database offers hash indexes as a first-class citizen:

  • PostgreSQL — explicit CREATE INDEX ... USING hash is supported and WAL-logged since PostgreSQL 10 (earlier versions had durability limitations).
  • MySQL / InnoDB — InnoDB builds an adaptive hash index automatically in memory for frequently accessed B-tree pages; you cannot create one explicitly.
  • SQLite — does not support hash indexes natively; all indexes are B-trees. The widget below uses a standard index but demonstrates the same equality-lookup pattern a hash index is optimized for.

When to Choose a Hash Index

Reach for a hash index when:

  1. You only query by exact equality — no ranges, no prefix matches, no ORDER BY on that column.
  2. The column has high cardinality — UUIDs, session tokens, email addresses. Low-cardinality columns (like a boolean) gain nothing from any index.
  3. Your workload is insert-heavy — hash indexes skip tree rebalancing, so maintenance cost can be lower than a B-tree (though rehashing has its own cost).

When in doubt, a B-tree is the safer default — it handles equality nearly as fast and keeps all other query patterns open.
