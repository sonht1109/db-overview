Part II. Querying and Language

Chapter 7. Indexes

What an index really is

Every time a query filters, sorts, or joins on a column, the database has to find the matching rows. Without any help, the only safe strategy is to read every single row in the table and check each one. That works fine for hundreds of rows. For millions, it becomes the kind of problem that makes engineers get paged at 2 a.m. An index is the structure that makes that problem go away — but only if you understand what it actually is.

The Naive Approach: Full Table Scan

Imagine a users table with one million rows. You run:

SELECT * FROM users WHERE email = 'alice@example.com';

Without an index, the database engine executes a full table scan (also called a sequential scan): it reads every page of the table from disk, examines every row, and returns the ones where email matches. For one row out of a million, that is an enormous amount of wasted work.

The cost scales linearly with table size — O(n). Double the rows, double the time.

What an Index Actually Is

An index is a separate data structure maintained by the database engine alongside your table. It stores a subset of the table's data — typically one or more column values — organized in a way that makes searching fast.

The most common form is a B-tree index (balanced tree). Conceptually, it looks like this:

         [M]
        /   \
     [D–L]  [N–Z]
     /   \   /   \
  [D–G][H–L][N–R][S–Z]
   ...  ...  ...  ...

Each leaf node holds the indexed column value paired with a pointer (a row identifier) back to the actual row in the table. The tree stays balanced so that any lookup takes at most O(log n) steps — for a million rows, that is roughly 20 comparisons instead of 1,000,000.

Note: The ~20 comparisons figure assumes a binary tree (log₂ 1M ≈ 20). Real database B-trees have much higher fanout — each node stores hundreds of keys. That means a B-tree for 1M rows is typically only 3–4 nodes deep, requiring even fewer I/O operations than the binary-tree model suggests.

Note: An index is not magic — it is a trade-off. The engine must update every index on every INSERT, UPDATE, and DELETE. More indexes means slower writes and more disk space. The win is on reads.

The table itself does not change when you create an index. The index is a parallel structure the optimizer can choose to use — or ignore — depending on the query.

What the Engine Does With an Index

When a query has a WHERE clause on an indexed column, the optimizer weighs two paths:

Strategy How it works When it is chosen
Full table scan Read all rows sequentially No usable index, or most rows match anyway
Index scan Walk the B-tree to find matching row pointers, then fetch those rows Selective filter — few rows match
Index-only scan Walk the B-tree; all needed columns are already in the index All SELECTed columns are in the index

"Selective" is the key word. If a query for status = 'active' matches 95% of the table, a full scan is actually faster — the overhead of bouncing back and forth between the index and the table pages costs more than just reading everything in order.

The Key Mental Model

Think of a database index exactly like a book's index: the book's content (your table data) does not move, but there is a sorted list at the back that tells you exactly which pages contain the term you are looking for. You go straight to those pages instead of reading cover to cover.

Three things to remember as you move forward:

  1. An index is a separate structure — it mirrors part of your table data in a search-friendly layout.
  2. The optimizer chooses whether to use it — writing a query that could use an index does not guarantee it will.
  3. Indexes cost something — write overhead and storage are real; you add them deliberately, not by default.
