Part VII. Building Understanding

Chapter 27. A Tiny Database from Scratch

Add concurrency control

So far, our toy engine is entirely single-threaded. One transaction runs to completion before the next begins. Real databases handle dozens to thousands of concurrent transactions, and making them behave correctly is one of the hardest problems in systems design. Without concurrency control, two transactions can read stale data, overwrite each other's writes, or produce results that could never have happened in any sequential order. This page adds the minimal machinery needed to get correctness back.

Why Concurrency Breaks Things

Consider two transactions running simultaneously against an accounts table:

T1: read(Alice.balance)   → 1000
                            T2: read(Alice.balance)  → 1000
T1: write(Alice.balance)  ← 800   (deducted 200)
                            T2: write(Alice.balance) ← 700   (deducted 300)
                            T2: COMMIT
T1: COMMIT

Alice started with 1000. Two deductions should leave 500. Instead she has 800 — T2's write was lost. This is the classic lost update anomaly, and it happens whenever two transactions read the same value and write back independently.

The Standard Anomalies

Anomaly What happens Prevented by isolation level
Dirty read Read an uncommitted value from another txn Read Committed
Non-repeatable read Re-read same row, get different value Repeatable Read
Lost update Two txns read-then-write same row; one overwrites the other Repeatable Read
Phantom read Re-scan same range, get different set of rows Serializable

Approach 1: Two-Phase Locking (2PL)

The classical solution. Every transaction must acquire a lock before accessing a row. Locks are released only at commit or rollback. No other transaction can acquire a conflicting lock until the first releases it.

import threading
from collections import defaultdict

class LockManager:
    """Shared / exclusive row-level lock table."""
    def __init__(self):
        self._locks = defaultdict(lambda: {"holders": set(), "mode": None})
        self._cv    = threading.Condition()

    def acquire(self, txn_id: int, row_num: int, mode: str = "S"):
        """mode: 'S' (shared/read) or 'X' (exclusive/write). Blocks until granted."""
        with self._cv:
            while True:
                entry = self._locks[row_num]
                if entry["mode"] is None:                # free
                    entry["mode"]    = mode
                    entry["holders"] = {txn_id}
                    return
                if mode == "S" and entry["mode"] == "S": # shared compat
                    entry["holders"].add(txn_id)
                    return
                # Conflict — wait
                self._cv.wait()

    def release_all(self, txn_id: int):
        """Release every lock held by txn_id (called at commit/rollback)."""
        with self._cv:
            for row_num, entry in list(self._locks.items()):
                entry["holders"].discard(txn_id)
                if not entry["holders"]:
                    entry["mode"] = None
                    del self._locks[row_num]
            self._cv.notify_all()

2PL prevents lost updates and ensures serializable schedules, but it can cause deadlocks when two transactions each hold a lock the other needs.

T1 holds lock on row 3, waits for row 7
T2 holds lock on row 7, waits for row 3
→ deadlock — neither can proceed

Real databases detect deadlocks by scanning the wait-for graph and aborting one transaction.

Approach 2: Optimistic Concurrency Control (OCC)

Instead of blocking, OCC lets transactions proceed without locks, then validates at commit time that no conflict occurred:

class OptimisticTransaction:
    def __init__(self, table: TableFile, version: dict):
        self.table        = table
        self._reads       = {}    # row_num -> version_at_read
        self._writes      = []
        self.global_ver   = version  # shared dict: row_num -> current version

    def read(self, row_num: int) -> dict:
        self._reads[row_num] = self.global_ver.get(row_num, 0)
        return self.table.read_row(row_num)

    def write(self, row: dict):
        self._writes.append(row)

    def commit(self, lock: threading.Lock) -> bool:
        with lock:
            # Validate: have any read rows been modified since we read them?
            for row_num, ver in self._reads.items():
                if self.global_ver.get(row_num, 0) != ver:
                    return False   # conflict — abort
            # Install writes
            for row in self._writes:
                row_num = self.table.write_row(row)
                self.global_ver[row_num] = self.global_ver.get(row_num, 0) + 1
            return True

OCC shines when conflicts are rare (read-heavy workloads). Under high contention, the abort rate climbs and throughput collapses.

MVCC: The Modern Compromise

Multi-Version Concurrency Control (used by PostgreSQL, Oracle, MySQL InnoDB) keeps multiple versions of every row. Readers get a snapshot of the database as it existed when their transaction started — they never block writers, and writers never block readers.

# Each row in the table gets a created_at and expired_at transaction ID
# Readers see rows where created_at <= my_txn_id < expired_at
rows = [
    {"id": 1, "name": "Alice", "salary": 70_000, "xmin": 1, "xmax": None},
    {"id": 1, "name": "Alice", "salary": 75_000, "xmin": 5, "xmax": None},
    # Transaction 5 updated the salary; txn 1's version still visible to older readers
]

MVCC is what makes SELECT fast in PostgreSQL even when a long-running transaction is writing furiously — reads go to the snapshot, not the current "dirty" state.

Key Takeaways

  • Without concurrency control, concurrent transactions can produce dirty reads, lost updates, and phantoms — results that are impossible in any serial execution.
  • Two-phase locking (2PL) prevents anomalies by blocking conflicting transactions, at the cost of potential deadlocks.
  • Optimistic concurrency control (OCC) avoids locks but aborts transactions that conflict at commit time — best when conflicts are rare.
  • MVCC keeps multiple row versions so readers never block writers and vice versa; it is the dominant choice in modern OLTP databases.
  • All three strategies trade off throughput, latency, and implementation complexity differently. Understanding all three lets you read a database's isolation documentation with real comprehension.
