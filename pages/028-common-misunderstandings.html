Chapter 3. The Relational Model

Common misunderstandings

The relational model is deceptively simple to describe but surprisingly easy to misuse. A few stubborn myths trip up newcomers — and even experienced developers — leading to slow queries, fragile schemas, and hard-to-fix bugs. This page names them plainly so you can avoid them from the start.

Misunderstanding 1: "A table is just a spreadsheet"

At first glance, a table and a spreadsheet look identical: rows and columns, values in cells. But the relational model makes a guarantee that spreadsheets do not: rows in a table are an unordered set. There is no row 1, row 2, row 3 in any meaningful sense unless you ask for an explicit order.

This trips people up in two common ways:

  1. Assuming SELECT * returns rows in insertion order. It might — but only by accident of the current execution plan. Add an index, vacuum the table, or scale the data and that order can change silently.
  2. Using a row's physical position to mean something. Spreadsheet users often rely on "this is the 5th row so it's the 5th entry." In a relation, position carries no meaning.

The fix is simple: if order matters to your result, write ORDER BY. That is the only reliable contract.

-- Wrong: relies on undefined ordering
SELECT * FROM events;

-- Right: declare the order you need
SELECT * FROM events ORDER BY occurred_at;

Note: The relational model inherits this from set theory. A mathematical set has no inherent sequence — and neither does a SQL table. ORDER BY is not a hint; it is the authoritative way to impose sequence.

Misunderstanding 2: "NULL means zero (or empty string, or false)"

NULL is one of the most misunderstood concepts in SQL. It does not mean zero, blank, or false. It means the value is unknown or inapplicable. That distinction has real consequences because NULL follows a three-valued logic: a comparison with NULL yields neither TRUE nor FALSE — it yields NULL.

Expression Result
NULL = NULL NULL (not TRUE)
NULL = 0 NULL (not TRUE)
NULL <> 'hello' NULL (not TRUE)
1 + NULL NULL

This means WHERE column = NULL never matches any row, even rows where the column is NULL. The correct test is IS NULL or IS NOT NULL.

Misunderstanding 3: "More tables means slower queries"

New schema designers sometimes denormalize aggressively — stuffing everything into one giant table — because they fear joins are slow. This is backwards.

Joins are not inherently expensive. A well-indexed join between two tables is often faster than a full scan of a single bloated table, because:

  • The optimizer can use indexes on the join columns to skip most rows.
  • Smaller, focused tables fit more rows per disk page.
  • Redundant data in a denormalized table means more bytes to read.

The real cost drivers are missing indexes and returning more data than needed — not the join itself.

That said, normalization has real trade-offs. Heavy aggregation workloads (analytics, reporting) sometimes benefit from pre-joined, denormalized structures. The right choice depends on your workload — not a blanket fear of joins.

Misunderstanding 4: "The primary key must be an auto-increment integer"

Auto-increment integers (id SERIAL, id INTEGER PRIMARY KEY AUTOINCREMENT) are convenient defaults, but they are not a requirement of the relational model. A primary key must be unique and non-NULL — that is all.

Natural keys — values that already exist in the domain — are perfectly valid:

Table Natural key candidate
countries ISO country code ('US', 'DE')
currency_pairs ('USD', 'EUR') composite
employees Employee number assigned by HR

Surrogate keys (auto-increment integers) win when no natural key is stable or guaranteed unique. Natural keys win when you want human-readable references and the value truly won't change. Neither is universally correct — the choice is about stability and legibility of the data, not a rule of the model itself.

Reveal: Can a table have no primary key at all?
Technically yes — SQL allows tables without a primary key (they are called heap tables in some engines). But the relational model assumes every relation has a key: without one, you can have duplicate rows, which breaks set semantics and makes updates and deletes ambiguous. In practice, always define a primary key.
