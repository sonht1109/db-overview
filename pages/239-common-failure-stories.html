Part VI. Engineering Practice

Chapter 24. Designing for Reliability

Common failure stories

Real database outages rarely happen for novel reasons. They happen for the same ten reasons, over and over, at different companies with different stacks — because the underlying dynamics are architectural, not implementation-specific. Studying failure stories teaches pattern recognition: the ability to look at a set of symptoms and recognise the category of problem before the incident drags into its second hour. This page distills the most instructive recurring failure patterns.

The Cascade of Slowness

What happened: A poorly-indexed query on a hot table began taking 8 seconds instead of 80 milliseconds after a data volume milestone was crossed. Each slow query held a database connection for the full 8 seconds. The application had 20 database connections in its pool. At modest traffic, all 20 connections were occupied with slow queries, and new requests waited. Wait times accumulated, threads backed up, and within 3 minutes the service was returning 503 errors to 100% of users — for what was fundamentally a missing index.

Why it cascades: Connection pools decouple request volume from database connections. Under normal conditions this is good. Under abnormal latency, it means a single slow query class can exhaust the pool because each connection is held for far longer than expected.

Detection: Connection wait time metrics spike before error rates do. If you only alert on error rates, you lose the warning window.

Prevention:

  • Alert on p99 query latency, not just errors
  • Set statement_timeout to bound worst-case query duration
  • Use EXPLAIN ANALYZE on all queries that touch large tables before deploying

The Replication Slot That Ate the Disk

What happened: A team set up logical replication for a data pipeline. The downstream consumer was later decommissioned, but no one dropped the replication slot. PostgreSQL holds WAL segments for all active replication slots — so every WAL segment produced since the consumer died was retained. Over 6 weeks, the WAL directory grew to fill the disk. The database halted all writes with "no space left on device."

-- Replication slots and their lag (bytes retained)
SELECT slot_name, plugin, active,
       pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS retained_wal
FROM pg_replication_slots
ORDER BY pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) DESC;

Prevention:

  • Alert when any replication slot retains > 10 GB of WAL
  • Document and automate slot cleanup when consumers are decommissioned
  • Set max_slot_wal_keep_size (PostgreSQL 13+) to cap WAL retained per slot

The Autovacuum Fallen Behind

What happened: A high-throughput orders table received thousands of UPDATEs per second. Autovacuum was configured with default settings (tuned for small databases) and could not keep pace. Dead tuples accumulated; the table's pg_class.relpages count ballooned. Queries that had historically taken 50 ms began taking 4 seconds as the planner's row estimates became wildly wrong due to stale statistics. The fix — manually running VACUUM ANALYZE — completed in 40 minutes and immediately restored query performance.

Why autovacuum falls behind: The defaults assume low-churn tables. A table with 10 million rows being updated at 5,000 rows/second needs autovacuum to run nearly continuously, not once every few hours.

Tuning for hot tables:

ALTER TABLE orders SET (
  autovacuum_vacuum_scale_factor = 0.01,    -- vacuum when 1% of rows are dead (default 20%)
  autovacuum_analyze_scale_factor = 0.005,  -- analyze when 0.5% of rows change
  autovacuum_vacuum_cost_delay = 2          -- ms; lower = more aggressive (default 20ms)
);

The Migration That Locked the Table

What happened: A developer ran ALTER TABLE users ADD COLUMN last_login TIMESTAMP NOT NULL DEFAULT now(); in production. The statement acquired an ACCESS EXCLUSIVE lock and held it for 14 minutes while rewriting every row in a 90M-row table. Every query touching users — login, profile update, session validation — queued behind that lock. The application effectively went down for 14 minutes.

What should have happened:

-- Step 1: Add nullable column (near-instant)
ALTER TABLE users ADD COLUMN last_login TIMESTAMP;

-- Step 2: Backfill in small batches (background job, no exclusive lock)
UPDATE users SET last_login = now() WHERE id BETWEEN 1 AND 50000 AND last_login IS NULL;
-- ... repeat for all batches ...

-- Step 3: Add NOT NULL constraint (PostgreSQL 12+ can verify without full rewrite if null check already done)
ALTER TABLE users ALTER COLUMN last_login SET NOT NULL;

The 3 AM Pager With No Runbook

What happened: A replica-lag alert fired. The on-call engineer, unfamiliar with the replication stack, spent 45 minutes reading documentation and experimenting with commands. Several of those commands had unintended effects. What turned out to be a straightforward disk I/O bottleneck became a 2-hour incident — not because the problem was hard, but because the response process was improvised.

The same team subsequently wrote a replication-lag runbook covering five common causes and their specific resolutions. The next time the alert fired (4 months later), time-to-resolution was 8 minutes.

Lesson: The hardest part of incident response is not technical — it is that stress, sleep deprivation, and time pressure severely impair diagnostic thinking. Runbooks remove the thinking burden and let the engineer execute.

The Backup That Was Never Restored

What happened: After a ransomware attack encrypted the primary database and all local backups, the team initiated a restore from S3. The backup files were present. But the restore had never been tested, and the procedure in the runbook referenced a deprecated pg_restore flag. The flag caused the tool to silently skip all data, producing an empty database. The team spent 6 hours debugging the restore process before successfully recovering data — 19 hours after the attack began.

The lesson: A backup is a hypothesis. Only a tested restore confirms it. Run quarterly restore drills to a staging environment. Measure and record the actual restore time.

The Query That Worked Yesterday

What happened: A query that read from a reports table took 90 ms for 18 months. After a new month's data was loaded — growing the table from 8 million to 11 million rows — the same query took 45 seconds. The cause: the query planner had been using an index, but after the row count crossed a threshold, the planner estimated that a sequential scan would be cheaper (incorrectly, due to stale statistics). Running ANALYZE reports; restored the correct plan within seconds.

Prevention:

  • Run ANALYZE after large data loads (or let autovacuum do it — if it is configured aggressively enough)
  • Monitor query latency trends over time, not just point-in-time
  • Use pg_stat_statements to track query plans and flag dramatic latency regressions

Key Takeaways

  • Most database outages are repeats of known patterns: slow queries cascading through connection pools, replication slot bloat, missed vacuum, lock-storm migrations, untested backups
  • The recurring nature of these failures means each one is a learning opportunity for the entire industry
  • Connection pool exhaustion is the most common user-visible symptom — but the root cause is almost always query latency, not the pool itself
  • Backups are hypotheses until proven by a successful restore drill
  • Pattern recognition — knowing "this looks like a replication slot problem" — is what separates experienced database operators from novices, and it is built entirely through studying past failures
