Part II. Querying and Language

Chapter 6. How Query Processing Works

Physical plans

When you run a SQL query, the database engine compiles it into a physical plan — a tree of concrete operations specifying exactly how data will be fetched, filtered, joined, and assembled. Understanding physical plans is what lets you reason about why a query is fast or slow, and why adding an index can change everything.

From Logical to Physical

SQL describes what you want, not how to get it. The optimizer first builds a logical plan — an abstract tree of relational operations like "filter rows" or "join two relations" — then maps each node to one or more physical operators: concrete algorithms that run against real storage.

Logical operation Possible physical operators
Scan a table Full table scan, index scan, index-only scan
Filter rows Applied inline during the scan
Join two tables Nested-loop join, hash join, sort-merge join
Sort rows In-memory sort, external merge-sort
Aggregate Hash aggregate, sort-then-aggregate

The optimizer chooses operators based on statistics: estimated table sizes, column cardinality, available indexes, and filter selectivity. The goal is to minimize total work — measured in I/O and CPU.

Reading a Physical Plan

Most databases let you inspect the chosen physical plan with EXPLAIN or EXPLAIN ANALYZE. SQLite's EXPLAIN QUERY PLAN shows a simplified version:

EXPLAIN QUERY PLAN
SELECT e.name, d.dept_name
FROM employees e
JOIN departments d ON e.dept_id = d.id
WHERE e.salary > 80000;

You might see output like:

SCAN employees AS e
SEARCH departments AS d USING INTEGER PRIMARY KEY (rowid=?)

Two things stand out here. First, SCAN means a full table scan — SQLite will read every row in employees to evaluate the salary filter. Second, SEARCH ... USING INTEGER PRIMARY KEY means for each matching employee the engine does a fast primary-key lookup into departments. This is a nested-loop join: the outer loop scans employees, and the inner loop looks up the matching department row by its primary key.

Note: EXPLAIN QUERY PLAN shows the intended plan before execution. EXPLAIN ANALYZE (in PostgreSQL) or EXPLAIN with VERBOSE runs the query and reports actual row counts and timing alongside estimates — invaluable for spotting optimizer mistakes.

The Three Common Join Algorithms

Joins are where physical planning has the biggest impact on performance, so it is worth knowing the three algorithms most engines implement:

Nested-loop join — for each row in the outer table, probe the inner table for matches. Without an index this is O(n × m). With an index on the inner side the probe is O(log m) per outer row — fast when the outer set is small.

Hash join — build a hash table from the smaller relation, then stream the larger relation through it. Very efficient for large equi-joins with no useful index. Memory-hungry: if the hash table spills to disk, cost rises sharply.

Sort-merge join — sort both sides on the join key, then merge the sorted streams in one pass. Efficient when inputs are already sorted (both have an index on the join column), or when the output itself must be ordered.

Why This Matters

A physical plan is the engine's bet on the cheapest execution path given what it knows. When a query is slow, the culprit is almost always a suboptimal plan — a full scan where an index scan was possible, or a hash join spilling to disk because statistics were stale. Learning to read EXPLAIN output is the single most practical performance skill you will develop as a database user.
