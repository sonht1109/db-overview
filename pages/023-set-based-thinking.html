Chapter 3. The Relational Model

Set-based thinking

Most programmers learn to solve problems one item at a time: loop over a list, check each element, do something to it, move on. Relational databases flip that instinct on its head. SQL is built on set-based thinking — you describe what you want from a whole collection of rows, and the database engine figures out how to get it. Embracing this shift is the single biggest mental upgrade for writing fast, clear SQL.

Rows Are Members of a Set

In mathematics, a set is an unordered collection of distinct elements. A database table is close to that idea: it is a collection of rows, and you operate on all the matching rows at once rather than one by one.

When you write:

UPDATE accounts SET status = 'inactive' WHERE last_login < '2024-01-01';

You are not looping through rows in your head — you are declaring a condition that selects a subset, and the engine applies the change to every member of that subset in one logical operation. The database might touch thousands of rows; you wrote one statement.

This is the essence of set-based thinking: express the desired state of a set, not a sequence of steps.

The Contrast with Row-by-Row (Cursor) Logic

To make the difference concrete, imagine you want to give every employee in the engineering department a 10 % raise.

Row-by-row approach (procedural pseudocode):

for each employee:
    if employee.department == 'engineering':
        employee.salary = employee.salary * 1.10
        save(employee)

Set-based SQL:

UPDATE employees
SET salary = salary * 1.10
WHERE department = 'engineering';

The SQL version sends one statement to the engine. The engine applies it to the entire matching set — atomically, and using whatever indexes or parallel strategies it has available. The pseudocode sends N round-trips (or cursor fetches) and gives the engine no opportunity to optimise across rows.

Note: SQL does have cursors for row-by-row processing, but they are a last resort. Experienced database developers reach for them only when a problem genuinely cannot be expressed as a set operation — which is rare.

Thinking in Filters, Projections, and Joins

Set-based thinking maps naturally onto three operations you will use constantly:

Operation What it does SQL keyword
Filter Narrow the set to rows matching a condition WHERE
Project Select only certain columns from the set SELECT col1, col2
Join Combine two sets by matching rows JOIN ... ON

Each of these produces a new set — a result set — that you can then filter, project, or join further. The whole query is a pipeline of set transformations.

Why It Matters Beyond Style

Set-based queries are not just cleaner — they unlock the engine's optimizer. When you hand the database a full set operation, it can:

  • Choose between a full table scan or an index lookup based on statistics
  • Execute in parallel across partitions
  • Avoid redundant disk reads by caching pages across rows in the same pass

A row-by-row cursor forces the engine into a rigid execution path you dictated. A set-based query gives it freedom to find the fastest plan.

This is why two queries that return identical results can differ in speed by orders of magnitude. The data is the same; what changes is whether you let the optimizer work as intended.

Key insight: SQL's declarative nature — saying what, not how — is not a limitation. It is the feature. Set-based thinking is how you write SQL that stays fast as your data grows.
