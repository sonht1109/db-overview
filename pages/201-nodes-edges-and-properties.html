Part V. Database Families

Chapter 21. Graph Databases

Nodes, edges, and properties

A graph database stores data as a network of nodes (entities) and edges (relationships). Instead of spreading related data across multiple tables joined by foreign keys, graph databases make relationships first-class citizens — each edge is a named, directed, typed connection that can carry its own properties. This model is a natural fit for any domain that is fundamentally relational in the non-SQL sense: social networks, supply chains, knowledge graphs, fraud rings, and recommendation engines.

The Labeled Property Graph Model

The dominant graph model today is the Labeled Property Graph (LPG), used by Neo4j, Amazon Neptune (Gremlin mode), and Memgraph. It has four building blocks:

Concept Description Example
Node An entity (vertex) A person, a company, a product
Label A type tag on a node (multiple allowed) :Person, :Employee
Edge A directed, typed relationship between two nodes KNOWS, WORKS_AT
Property A key-value attribute on a node or edge name: "Alice", since: 2019

Note: Edges are always directed (they have a start node and an end node), but graph engines typically let you traverse them in either direction at query time. The direction is semantic — (Alice)-[:FOLLOWS]->(Bob) means Alice follows Bob, not the other way around.

Nodes

A node is a record with an identity, zero or more labels, and zero or more properties. Labels are like table names but lighter-weight: a node can carry multiple labels simultaneously (:Person:Employee), and labels carry no schema constraints by default.

Node id=1
  labels: [:Person]
  properties:
    name: "Alice"
    age: 34
    city: "Singapore"

Node id=2
  labels: [:Person]
  properties:
    name: "Bob"
    age: 29
    city: "Hanoi"

Node id=3
  labels: [:Company]
  properties:
    name: "Acme Corp"
    founded: 2010

Edges

An edge connects exactly two nodes with a mandatory type and optional properties. Every edge has:

  • A start node (tail)
  • An end node (head)
  • A relationship type (uppercase by convention in Cypher)
  • Zero or more properties
Edge id=10
  type: KNOWS
  start: node 1 (Alice)
  end:   node 2 (Bob)
  properties:
    since: 2021

Edge id=11
  type: WORKS_AT
  start: node 1 (Alice)
  end:   node 3 (Acme Corp)
  properties:
    role: "Engineer"
    since: 2022

Edge id=12
  type: WORKS_AT
  start: node 2 (Bob)
  end:   node 3 (Acme Corp)
  properties:
    role: "Designer"
    since: 2023

How This Differs from Relational Tables

In a relational database, relationships are implicit. Two tables are connected by matching values — a foreign key in one row that equals a primary key in another. The relationship itself has no physical representation; it only comes into being when a query performs a join.

In a graph database, relationships are explicit and stored directly. Each edge record points directly to its start and end nodes. There are three important consequences:

  1. No join computation at query time. Following an edge is a pointer traversal, not an index lookup and comparison across two table scans.
  2. Relationship properties are natural. To store when a friendship started in a relational model you need a junction table with an extra column. In a graph, you just add a since property to the edge itself.
  3. Schema flexibility. Nodes with the same label don't need identical property sets. Alice might have a phone property; Bob might not. No NULL column overhead, no ALTER TABLE.

Relational vs Graph: Storing a Friendship Network

Relational approach:

-- Three tables minimum for a "person knows person with timestamp"
CREATE TABLE persons (id INTEGER PRIMARY KEY, name TEXT, age INTEGER);
CREATE TABLE friendships (
  person_a INTEGER REFERENCES persons(id),
  person_b INTEGER REFERENCES persons(id),
  since    INTEGER,
  PRIMARY KEY (person_a, person_b)
);

Graph approach (Cypher, Neo4j):

CREATE (a:Person {name: "Alice", age: 34})
CREATE (b:Person {name: "Bob",   age: 29})
CREATE (a)-[:KNOWS {since: 2021}]->(b)

The graph version has no join table, no nullable foreign keys, and the relationship is a named, typed first-class object. When you ask "who does Alice know?" the engine walks the edges stored on Alice's node record — it never scans the friends table.

Why Graph Databases Don't Use Foreign Keys

Foreign keys enforce referential integrity, but they don't store physical pointers. When PostgreSQL evaluates JOIN persons ON friendships.person_b = persons.id, it looks up an index on persons.id for each row in friendships. That index lookup has a cost that scales with the size of both tables.

Graph databases use a technique called index-free adjacency: each node stores a direct linked list of its edges. Traversing one hop is an O(1) pointer dereference regardless of how many nodes exist in the graph. A social network with 1 billion users doesn't make "find Alice's friends" slower — only the number of Alice's friends matters, not the global population.

Concept Relational Graph (LPG)
Relationship storage Implicit (matching FK values) Explicit (edge record with pointers)
Relationship properties Junction table column Edge property
Multi-label entity NULL columns or sub-type tables Multiple labels on one node
Hop cost Index lookup per join Pointer traversal (O(1) per hop)
Schema enforcement Strict column schema Optional constraints

Key Takeaways

  • Nodes are entities with labels and properties; edges are directed, typed relationships that can carry their own properties.
  • The Labeled Property Graph model makes relationships first-class objects — stored explicitly, not inferred at query time.
  • Graph databases use index-free adjacency: nodes hold direct pointers to their edges, making each hop O(1) regardless of graph size.
  • The relational equivalent of a graph requires junction tables and multi-table joins — workable for shallow queries but expensive for multi-hop traversals.
  • Graph databases excel when your questions are about the connections themselves — paths, reachability, clusters — rather than aggregate statistics over flat records.
