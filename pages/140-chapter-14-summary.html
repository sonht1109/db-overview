Chapter 14. From One Machine to Many

Chapter 14 Summary: What We Learned

Chapter Contents

Page Topic Takeaway in One Sentence
131 Why distribute a database A single machine hits limits: data size, throughput, or availability — distribution spreads load and tolerates failures.
132 Replication and sharding Replication = copies of the same data (availability, read scale). Sharding = splitting data across nodes (write scale, capacity).
133 Shared-nothing systems Each node owns its own CPU, memory, and disk — scales horizontally but requires careful data placement.
134 Partition keys The partition key determines which node owns a row; choose it to spread load evenly and support your most frequent queries.
135 Cross-node queries Queries that span partitions require scatter-gather or distributed joins — significantly slower than single-node queries.
136 Rebalancing data Adding or removing nodes requires moving partitions; consistent hashing minimizes the amount of data moved.
137 Hot partitions A partition that receives disproportionate traffic becomes a bottleneck — choose partition keys to avoid hotspots.
138 Failure becomes normal With many nodes, some are always down — the system must handle partial failures gracefully.
139 New tradeoffs at scale CAP theorem, latency vs consistency, operational complexity, and network costs all become front-line concerns.
