Chapter 23. Designing for Performance

Read-heavy vs write-heavy workloads

Almost every performance decision in database engineering starts with a single question: is this workload read-heavy or write-heavy? The two cases have opposite bottlenecks, opposite tuning strategies, and — if you pick the wrong storage engine or replication topology — opposite failure modes. Answering the question early shapes every choice that follows.

Defining the Terms

A workload is read-heavy when SELECT queries dominate. The database spends most of its time retrieving pages from disk (or cache), evaluating predicates, and streaming results. E-commerce product pages, analytics dashboards, and content feeds are canonical examples.

A workload is write-heavy when INSERT, UPDATE, and DELETE operations dominate. The database spends most of its time maintaining indexes, writing WAL (write-ahead log) entries, and flushing dirty pages. Time-series telemetry, event ingestion, and financial ledgers are canonical examples.

Most real systems are a mixture, but the dominant pattern dictates the architecture.

Why Indexes Hurt Writes

Every index is a secondary copy of data that must be updated on every write. A table with five indexes on a write-heavy workload pays for six writes (the heap + five index updates) for every one logical row change. Under sustained insert throughput, this multiplier is the difference between a system that keeps up and one that falls behind.

LSM-trees (used by Cassandra, RocksDB, ScyllaDB) flip this trade-off: writes go to an in-memory buffer and are merged and sorted on disk later. Read latency increases because the engine must merge multiple sorted runs, but write throughput can be an order of magnitude higher than a B-tree under the same hardware.

B-trees (PostgreSQL, MySQL InnoDB) favour reads: the sorted structure means point lookups and range scans are a single traversal. Writes pay the cost of re-balancing pages in place.

The Index Count Rule of Thumb

Read-heavy table:   index everything you query
Write-heavy table:  index only what you cannot afford to scan

Replication Topology Choices

Strategy Fits Drawback
Primary + read replicas Read-heavy; distribute SELECTs Replication lag; writes still single-node
Multi-primary Write-heavy, geographically distributed Conflict resolution complexity
Sharding Write-heavy, high cardinality key Cross-shard queries become expensive
Single primary, write-optimised storage Write-heavy, single region No horizontal write scaling

Measuring the Ratio

Before choosing an architecture, measure. Most databases expose counters:

-- PostgreSQL: rows read vs rows inserted/updated/deleted
SELECT relname,
       seq_tup_read   AS rows_scanned,
       n_tup_ins      AS inserts,
       n_tup_upd      AS updates,
       n_tup_del      AS deletes
FROM   pg_stat_user_tables
ORDER  BY (n_tup_ins + n_tup_upd + n_tup_del) DESC;

Mixed Workloads: HTAP

Some systems run HTAP (Hybrid Transactional/Analytical Processing) — heavy writes from an online application and heavy analytical reads from a reporting layer simultaneously. The common architectures are:

  • Separate stores — writes go to an OLTP database; a nightly or streaming ETL moves data to a column-store (e.g., BigQuery, Redshift) for analytics.
  • In-engine column stores — MySQL HeatWave, SQL Server's in-memory columnstore index, Oracle Dual Format — maintain a row store for writes and a column store for reads in the same engine.
  • Change Data Capture — a CDC stream (Debezium, AWS DMS) replicates changes in near-real-time to an analytics store without a batch window.

Key takeaways: Index count, storage engine choice, and replication topology all flip depending on whether your workload is read-heavy or write-heavy.
