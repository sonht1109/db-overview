Part VI. Engineering Practice

Chapter 24. Designing for Reliability

Schema migrations in production

Changing a database schema while the application is running is one of the most dangerous operations in software engineering. A naive ALTER TABLE that would complete instantly on a development laptop can lock millions of rows for minutes on a production table — taking down the application in the process. Schema migrations in production require a completely different mindset from development migrations: every change must be safe to run while traffic flows.

Why Production Migrations Are Hard

On a development database with 1,000 rows, ALTER TABLE orders ADD COLUMN discount NUMERIC; finishes in milliseconds. On a production table with 200 million rows, the same statement may:

  • Acquire an exclusive table lock that blocks all reads and writes for the duration
  • Cause every query touching that table to queue up, exhausting connection pool slots
  • Time out after 30 seconds, leaving the schema unchanged but the application degraded

The problem is compounded by high availability: with replicas, the DDL must also propagate and re-execute on each replica, which can lag behind and cause further issues.

The Lock Hierarchy

Understanding what each operation locks is the key to safe migrations. PostgreSQL's lock levels from weakest to strongest:

Lock Blocks Common operations
ACCESS SHARE Nothing except ACCESS EXCLUSIVE Normal SELECT
ROW SHARE ACCESS EXCLUSIVE SELECT FOR UPDATE
ROW EXCLUSIVE SHARE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE INSERT, UPDATE, DELETE
SHARE UPDATE EXCLUSIVE SHARE UPDATE EXCLUSIVE, SHARE, and above VACUUM, CREATE INDEX CONCURRENTLY
ACCESS EXCLUSIVE Everything ALTER TABLE, DROP TABLE, TRUNCATE

Any ACCESS EXCLUSIVE operation will block all reads and writes. The goal of safe migrations is to avoid ACCESS EXCLUSIVE locks on hot tables, or to hold them for the absolute minimum time.

The Expand-Contract Pattern

The safest way to change a schema that is actively in use is the expand-contract (also called additive migration) pattern. Changes are split into three phases deployed independently:

Phase 1 — Expand:   Add the new column/table (nullable, no constraints)
Phase 2 — Migrate:  Backfill data; deploy code that writes both old and new
Phase 3 — Contract: Drop the old column after all code uses the new one

Example: Renaming a Column

Renaming user_name to display_name in a live system:

Phase 1 (Expand)

-- Safe: adding a nullable column takes near-zero lock time
ALTER TABLE users ADD COLUMN display_name TEXT;

Application code update (dual-write)

# Write to both columns; read from old until backfill complete
user.display_name = user.user_name  # copy on write

Phase 2 (Backfill — in batches)

-- Update in small batches to avoid long-running locks
UPDATE users SET display_name = user_name
WHERE id BETWEEN 1 AND 10000 AND display_name IS NULL;
-- repeat for next batch...

Phase 3 (Contract — after all code uses display_name)

ALTER TABLE users DROP COLUMN user_name;

Concurrent Index Creation

Adding an index the normal way takes a full SHARE lock on the table, blocking writes for as long as the index build takes. For large tables this can be 10–60 minutes.

-- Dangerous: blocks all writes while building the index
CREATE INDEX idx_orders_user ON orders(user_id);

-- Safe: builds index in the background without blocking writes
CREATE INDEX CONCURRENTLY idx_orders_user ON orders(user_id);

CREATE INDEX CONCURRENTLY takes longer and uses more I/O, but only acquires SHARE UPDATE EXCLUSIVE, which does not block normal reads or writes. It cannot be run inside a transaction block.

Caveat: Concurrent index builds can fail partway through, leaving an INVALID index. Check pg_indexes or pg_index.indisvalid after creation. Drop and retry if the index is marked invalid.

Online Schema Change Tools

For the most disruptive changes (adding NOT NULL columns, changing column types), dedicated tools perform the migration transparently by creating a shadow table:

Tool Database Technique
pg_repack PostgreSQL Rebuilds table online, swaps with minimal lock
pglogical PostgreSQL Logical replication-based schema changes
gh-ost MySQL Ghost table + binary log streaming
pt-online-schema-change MySQL Triggers on shadow table
Online DDL MySQL 8+ / InnoDB Built-in: many operations are now lock-free

Migration Tooling and Version Control

Schema changes must be version-controlled like code. Common migration frameworks:

Flyway:     versioned SQL files (V001__add_discount.sql)
Liquibase:  XML or YAML changesets, supports rollback
Alembic:    Python (SQLAlchemy); auto-generates diffs
sqitch:     dependency-aware; works with any SQL dialect

Every migration should have:

  • A forward migration (apply)
  • A rollback migration (undo) — or explicit documentation that the change is non-reversible
  • A tested execution plan against production data volume

Key Takeaways

  • ALTER TABLE acquires ACCESS EXCLUSIVE lock — it blocks everything; never run naively on a hot table
  • Use the expand-contract pattern: add columns first (nullable), backfill in batches, then contract
  • Use CREATE INDEX CONCURRENTLY to build indexes without blocking writes
  • For extreme cases, use online schema tools (gh-ost, pg_repack) that shadow-copy and swap tables
  • Version-control every migration; always have a rollback plan before running against production
