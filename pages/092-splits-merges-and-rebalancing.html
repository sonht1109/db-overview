Part III. Storage and Execution

Chapter 10. Trees, Logs, and Internal Structures

Splits, merges, and rebalancing

B-tree indexes look elegant in diagrams, but the real database takes a constant beating of inserts, updates, and deletes. Every write can break the tree's balance. The engine fixes this automatically through splits, merges, and rebalancing — understanding them explains why index performance holds up over time, and why it sometimes quietly degrades.

The Problem: Pages Fill Up (and Empty Out)

A B-tree stores keys in fixed-size pages. Each page holds some maximum number of keys. Two things go wrong over time:

  • Overflow: An insert targets a page that is already full.
  • Underflow: A delete leaves a page nearly empty, wasting I/O and space.

The B-tree invariant says every non-root node must be at least half full. Splits handle overflow; merges handle underflow; rebalancing is the lighter-weight middle path when a full split or merge is not strictly necessary.

Splits: When a Page Is Full

When an insert arrives at a leaf page that has no room, the engine performs a page split:

  1. Allocate a new sibling page.
  2. Move roughly half the keys from the full page into the new page.
  3. Promote the middle key up to the parent as a separator.
  4. Update the parent's child pointers to include the new page.
Before split (page full, inserting 35):
Parent:  [ 20 | 50 ]
          /    |    \
        ...  [22|28|33|38]  ...   ← leaf full

After split:
Parent:  [ 20 | 33 | 50 ]        ← 33 promoted
          /    |     |    \
        ...  [22|28]  [35|38]  ...

The promotion can cascade: if the parent is also full, it splits too, pushing a key further up. In the worst case this chain reaches the root, which splits into two children and a new root is created — the only moment the tree grows taller.

Note: Splits require writing at least two pages (the original and the new sibling) plus the parent. For safety, the engine journals these writes before applying them so a crash mid-split does not corrupt the tree. That is one reason WAL and B-trees are inseparable in practice.

Merges: When a Page Is Too Empty

When a delete leaves a page below the minimum fill threshold (usually 50%), the engine looks at the adjacent sibling:

  • If the sibling has extra keys to spare, borrow one key from the sibling and adjust the separator in the parent. This is the "rebalance" or "rotate" case — cheaper than a full merge.
  • If the sibling is also near-minimum, merging is the only option: pull all keys from one sibling into the other, and delete the now-empty page plus its separator from the parent.

Deleting a separator from the parent can trigger another underflow, so merges can also cascade upward — shrinking the tree's height when the root is left with a single child.

Before merge (leaf [7] underflows after delete):
Parent:  [ 10 ]
          /    \
        [7]    [12|15]

After merge:
Parent:  (root dissolves, tree shrinks)
         [7|12|15]            ← single root leaf

Rebalancing Without a Split or Merge

A full split or merge touches at least three pages and always modifies the parent. Many engines try a cheaper fix first: redistribution (also called rotation). If a sibling page has slack, the engine shuffles one or two keys across the boundary and updates only the separator in the parent — no new pages allocated, no pages freed.

Situation Engine action Pages written
Page full, sibling has slack Redistribute (rotate) 2 leaves + parent
Page full, sibling also full Split New page + original + parent
Page underflows, sibling has slack Redistribute (rotate) 2 leaves + parent
Page underflows, sibling also minimal Merge 1 page freed + parent

Production engines prefer redistribution when possible because it keeps pages fuller on average, reducing tree height and improving cache hit rates.

Why This Matters for Performance

Splits and merges are the engine's way of staying honest, but they are not free:

  • Write amplification: One logical insert can touch many pages (the target leaf, a new sibling, and every ancestor up to the root in a worst-case cascade).
  • Page fragmentation: After many random inserts and deletes, pages may be only 60–70% full on average. A full sequential scan of the index reads more pages than a freshly built one would.
  • Index rebuilds: REINDEX (SQLite/PostgreSQL) or ALTER INDEX ... REBUILD (SQL Server) rewrite the B-tree from scratch, packing pages to near 100% fill. This is the blunt-but-effective cure for fragmentation after heavy churn.

Understanding splits and merges also sets up the next topic: LSM-trees (Log-Structured Merge-trees) were invented specifically to avoid the high random-write cost of in-place B-tree splits by batching all writes into sequential appends.
