Chapter 7. Indexes

Secondary indexes

Every table has at most one primary key — a built-in way to find a row by its unique identifier. But you rarely look up a user by their internal ID. More often you search by email address, filter products by category, or find orders placed after a certain date. A secondary index is any index built on a column (or columns) other than the primary key. It gives the database a fast path to rows based on the values of those columns.

Primary vs. Secondary: What Changes

With a B-tree primary-key index, the leaf nodes of the tree hold the actual row data — this is called a clustered index (the norm in InnoDB/MySQL; in SQLite every rowid table is effectively clustered). A secondary index is different: its leaf nodes store the indexed column values plus a pointer back to the full row (typically the primary key). Looking up a row via a secondary index therefore involves two steps:

  1. Traverse the secondary index B-tree to find the matching key(s) and their row pointers.
  2. Follow each pointer into the primary index (or heap) to fetch the complete row.

That second lookup is called a row lookup or primary-key lookup. It's cheap for a handful of rows, but if a secondary index matches a large fraction of the table it can be slower than a plain full scan — the optimizer will skip the index in that case.

Property Primary index Secondary index
Uniqueness enforced Always (primary key) Optional (UNIQUE or not)
Leaf stores Full row (clustered) or row pointer Indexed columns + row pointer
Number per table One Many (as many as you create)
Write overhead Unavoidable One extra tree write per DML

Creating and Dropping Secondary Indexes

Secondary indexes are created with CREATE INDEX and dropped with DROP INDEX. They do not change the data — only the access structures the engine maintains alongside it.

-- Non-unique index: one email can appear many times (unusual, but legal)
CREATE INDEX idx_users_email ON users (email);

-- Unique index: enforces uniqueness AND speeds up lookups
CREATE UNIQUE INDEX idx_users_email_unique ON users (email);

-- Index on a non-key column in orders
CREATE INDEX idx_orders_status ON orders (status);

-- Remove an index that is no longer useful
DROP INDEX idx_orders_status;

Note: In SQLite, a UNIQUE constraint on a column implicitly creates a unique index. In most other databases the same is true — you get the lookup speed for free when you declare the constraint.

Selectivity: When a Secondary Index Pays Off

Not all secondary indexes are equally useful. The engine estimates selectivity — roughly, how many rows match a given value. A highly selective index eliminates most rows early; a low-selectivity index does not.

Think about a status column with only three possible values: 'pending', 'shipped', 'cancelled'. An index on status alone is nearly useless — every value matches 30–40% of the table. The engine will table-scan instead. By contrast, an index on email is extremely selective: each value matches at most one row.

Column Distinct values Typical selectivity Index helpful?
user_id (FK) Millions Very high Yes
email Millions Very high Yes
country_code ~250 Low Rarely alone; maybe composite
status 3–5 Very low Rarely alone
is_active 2 Minimal Almost never

Note: Low selectivity is not always a dead end. A composite index that pairs a low-selectivity column with a high-selectivity one (e.g., (status, user_id)) can be very useful when queries filter on both.

Covering Indexes: Skipping the Row Lookup

A secondary index that contains all the columns a query needs is called a covering index. When the engine can satisfy the entire query from the index alone, it never touches the main table — the expensive row-pointer lookup disappears entirely.

-- If we add price to the category index, this query is fully covered:
CREATE INDEX idx_products_cat_price ON products (category, price);

-- The engine reads (category, price) from the index — no row lookup needed
SELECT name, price FROM products WHERE category = 'electronics' ORDER BY price;

Covering indexes are one of the most powerful tuning techniques for read-heavy workloads. The trade-off is a wider index that costs more storage and slightly more overhead on every write.

Reveal: Why does adding more columns to an index have a write cost?
Every time you INSERT, UPDATE, or DELETE a row, the database must also update all indexes that cover columns touched by that operation. A wider index means more data to write and more B-tree pages to potentially rebalance. In a write-heavy workload, too many secondary indexes can slow inserts significantly — each insert into a table with five indexes performs six tree writes instead of one.
