Chapter 24. Designing for Reliability

High availability basics

A single database server is a single point of failure. When that server crashes — and it will, eventually — every application connected to it stops working. High availability (HA) is the set of architectural patterns that allow a database system to survive individual node failures without significant downtime. This page covers the foundational concepts: replication topologies, failover mechanics, and the trade-offs you accept when designing for HA.

The Core Problem: Single Points of Failure

Availability is measured in "nines":

Availability Downtime per year Downtime per month
99% (two nines) 87.6 hours 7.3 hours
99.9% (three nines) 8.76 hours 43.8 minutes
99.99% (four nines) 52.6 minutes 4.4 minutes
99.999% (five nines) 5.26 minutes 26.3 seconds

Moving from two to three nines requires planned maintenance windows and a simple restart procedure. Moving from three to four nines requires redundant hardware and automated failover. Five nines requires every layer of the stack to be redundant, including the database.

Primary–Replica Replication

The most common HA primitive is primary–replica (master–slave) replication: one node (the primary) accepts writes; one or more replicas receive a copy of every change and apply it locally.

Changes flow from primary to replica via the same write-ahead log used for crash recovery (in PostgreSQL this is streaming replication; in MySQL it is the binary log). The replica is always slightly behind — this lag is the cost of asynchronous replication.

Synchronous vs. Asynchronous Replication

The biggest design decision in any HA setup is whether replication is synchronous or asynchronous.

Asynchronous Synchronous
Commit latency Low — primary does not wait Higher — must wait for replica acknowledgement
Data loss on failover Possible — replica may lag Zero — replica has every committed transaction
Replica failure impact None — primary continues Primary stalls if synchronous replica is unavailable
Best for Read scale-out, reporting Financial systems, any zero-RPO requirement

PostgreSQL's synchronous_commit = on (or remote_apply) makes commits synchronous. MySQL Group Replication offers similar guarantees. The practical middle ground is semi-synchronous: the primary waits for at least one replica to acknowledge receipt before confirming the commit.

Automated Failover

Manual failover is too slow for four-nines availability. Automated failover requires three components:

  1. Health monitoring — continuously checks whether the primary is responding (TCP ping, query heartbeat)
  2. Consensus / fencing — ensures only one node promotes itself to primary at a time, preventing split-brain
  3. VIP / DNS update — redirects application connections to the new primary without code changes

Common tools:

  • Patroni (PostgreSQL) — uses etcd/Consul/ZooKeeper for distributed consensus; most production PostgreSQL HA uses this
  • Orchestrator (MySQL) — topology-aware failover with visual dashboard
  • AWS RDS Multi-AZ — fully managed; failover in ~60–120 seconds via DNS update
  • Google Cloud Spanner — built-in multi-region HA, no operator intervention required

The Split-Brain Problem

Split-brain occurs when two nodes both believe they are the primary and accept writes simultaneously. Resolving the resulting divergence is extremely painful. Prevent it with STONITH (Shoot The Other Node In The Head) — the old primary's network is fenced before the new primary is promoted, guaranteeing the old primary cannot accept writes.

Read Replicas for Scale-Out

Beyond HA, replicas also serve read scale-out: route SELECT-heavy analytics and reporting queries to a replica, leaving the primary free for writes. This is the simplest form of horizontal read scaling and requires zero application changes beyond a read-only connection string.

Warning: Because replicas lag, reads from a replica may not reflect the most recent write. Design applications to tolerate eventual consistency on the replica read path, or use a synchronous replica for reads that must be current.

Key Takeaways

  • HA requires redundant nodes; a single server cannot meet four-nines availability
  • Primary–replica replication with WAL streaming is the standard building block for PostgreSQL and MySQL HA
  • Choose synchronous replication for zero RPO; accept the latency cost
  • Automated failover requires health checks, distributed consensus (to prevent split-brain), and VIP/DNS redirection
  • Read replicas provide both HA and read scale-out — but replica lag means reads may be slightly stale
