Chapter 15. Replication

Synchronous vs asynchronous replication

When a leader node accepts a write, it must eventually send that change to its replicas. The critical question is: when does the leader tell the client "done"? Does it wait until at least one replica has confirmed the write, or does it acknowledge immediately and let replication happen in the background? That choice — synchronous versus asynchronous — shapes every durability and latency tradeoff in a replicated system.

The Two Modes

Synchronous replication means the leader waits for the replica to confirm it has received (and usually persisted) the write before acknowledging success to the client. The client's COMMIT does not return until the replica says "got it."

Asynchronous replication means the leader acknowledges the client as soon as the write is durable locally. The replica will catch up eventually — maybe milliseconds later, maybe seconds later if the network is busy — but the client is not held waiting.

Tradeoffs at a Glance

Synchronous Asynchronous
Durability on crash Strong — replica already has the data Weaker — unconfirmed writes can be lost
Write latency Higher — adds one network round-trip Lower — no waiting for replica
Replica availability Required — a slow replica stalls writes Optional — leader proceeds regardless
Read-your-writes on replica Guaranteed immediately after commit Not guaranteed; reads may be stale
Replication lag Zero (by definition) Variable; can be seconds or more

Note: Most production systems use a hybrid approach: one synchronous replica for durability, and additional asynchronous replicas for read scale. PostgreSQL calls this "synchronous_standby_names" — you nominate one standby that must confirm each commit; all others replicate asynchronously.

Why Lag Matters in Practice

With asynchronous replication, a replica lags behind the leader by some number of seconds. This creates real problems:

  • Read-your-writes violation. A user posts a comment, then immediately reloads the page. If the read hits a replica, they may not see their own comment.
  • Failover data loss. If the leader crashes before a write is replicated, promoting the replica means that write is silently gone.
  • Stale metrics. Analytics queries on a lagging replica undercount recent events.

The lag is not constant. Under low load, a healthy replica might be milliseconds behind. During a network hiccup, writes queue up and lag can spike to minutes. A replica that is too far behind is called unavailable for consistent reads in some systems, and can be taken out of the read pool automatically.

Semi-Synchronous Replication

MySQL's default (since 5.7) is semi-synchronous: the leader waits for at least one replica to acknowledge receipt, but not necessarily to flush to disk. This is a middle ground — you avoid the scenario where no replica has the data, without fully blocking on a disk flush. If no replica acknowledges within a timeout, MySQL falls back to asynchronous mode automatically.

The distinction matters:

  • Receive-and-buffer (semi-sync): replica has the data in memory but maybe not on disk. Still survives a leader crash if the replica doesn't crash simultaneously.
  • Receive-and-fsync (fully synchronous): replica has flushed to disk. Survives even if both machines lose power simultaneously.

Reveal: If the leader crashes right now, which replicas can safely be promoted without data loss?
Only replica-1 (sync mode, zero lag). Replica-2 through replica-4 have not yet applied all writes — promoting any of them would silently lose the writes they haven't received. In a real system, the failover controller must check each replica's log position and pick the one closest to the leader, then accept that some commits may be rolled back.
