Chapter 12. Concurrency Control

The lost update problem

Imagine two bank tellers both pulling up the same account at the same moment, each counting the balance, each adding a deposit, and each typing in a new total — without knowing the other exists. One write lands first; the second write replaces it. A deposit has vanished into thin air. That is the lost update problem, one of the most common and costly concurrency bugs in database applications.

What Makes an Update "Lost"

An update is lost when two transactions follow this pattern on the same row:

  1. Both transactions read the current value.
  2. Each computes a new value based on what it read.
  3. Each writes its result back.
  4. The second write silently overwrites the first — as if the first never happened.

The critical point: neither transaction does anything wrong in isolation. The bug emerges only from the interleaving of their steps.

A concrete example

Time Transaction A (deposit $200) Transaction B (deposit $300) balance on disk
t1 READ balance → 1000 1000
t2 READ balance → 1000 1000
t3 WRITE balance = 1000 + 200 1200
t4 WRITE balance = 1000 + 300 1300 ← wrong

Expected result: 1500. Actual result: 1300. Transaction A's $200 deposit was lost because B used the stale value it read at t2.

Why Read-Then-Write Is the Root Cause

The pattern SELECT … then UPDATE is the culprit. It creates a window of vulnerability between the read and the write. Any other transaction that reads during that window will base its own write on stale data.

This pattern appears constantly in real applications:

  • Counters and inventory: stock = stock - qty
  • Likes and view counts: likes = likes + 1
  • Account balances: balance = balance + amount
  • Sequence numbers: next_id = max_id + 1

Note: The problem does not require bad intentions or unusual timing. Under any moderate load, two concurrent requests on the same popular row will trigger it regularly.

Three Ways to Prevent It

1. Atomic update (preferred when possible)

Instead of read-then-write, push the arithmetic into a single SQL statement. The database engine applies it atomically:

-- Safe: the engine reads and writes in one atomic step
UPDATE accounts
SET balance = balance + 200
WHERE id = 1;

No application-level read is needed, so there is no window for another transaction to slip in.

2. Pessimistic locking — SELECT FOR UPDATE

Lock the row at read time so no other transaction can touch it until you commit:

BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE; -- lock acquired
-- application computes new value
UPDATE accounts SET balance = 1200 WHERE id = 1;
COMMIT; -- lock released

Transaction B will block at its own SELECT FOR UPDATE until A commits. Safe, but adds wait time under contention.

3. Optimistic concurrency control (OCC)

Add a version column. Before writing, check that the version you read is still current:

-- Read: version = 7
SELECT balance, version FROM accounts WHERE id = 1;

-- Write: only succeeds if nobody else changed the row
UPDATE accounts
SET balance = 1200, version = 8
WHERE id = 1 AND version = 7;

-- If 0 rows updated → someone else changed it → retry

OCC avoids locks entirely; it detects conflicts only at write time and lets the application retry. It works well when conflicts are rare (optimistic about success).

Use the atomic update approach:

UPDATE posts SET likes = likes + 1 WHERE id = ?;

Optimistic locking requires retries (wasteful when conflicts are frequent) and pessimistic locking serializes all increments (a bottleneck). The atomic update lets the engine batch the arithmetic efficiently. For extreme throughput, some systems batch these writes in memory (e.g. Redis) and flush to the DB periodically, accepting a small margin of error in the displayed count.
