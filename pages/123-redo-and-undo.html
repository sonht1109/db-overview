Part III. Storage and Execution

Chapter 13. Recovery

Redo and undo

When a database restarts after a crash, it faces two distinct problems at once: some committed changes may never have reached disk, and some uncommitted changes may have. Fixing the first requires redo; fixing the second requires undo. These two operations are the twin engines of crash recovery, and understanding exactly how they work — and why they run in the order they do — reveals a lot about how databases guarantee ACID properties under failure.

Redo: Replaying Committed Work

The WAL (write-ahead log) records every change before it touches a data page. When a transaction commits, its log records are flushed to durable storage — but the actual data pages in the buffer pool may not be written to disk yet. A crash at that moment means: the log says "committed," but the heap file on disk disagrees.

Redo fixes this by replaying the log forward. For every log record that belongs to a committed transaction, the engine re-applies the change to the data file:

Log record:  LSN 204 | txn T7 | UPDATE accounts | page 18 offset 120
             old_val = 1000  new_val = 900
             
Redo action: write new_val (900) to page 18, offset 120

A critical detail: redo is idempotent. If a page was already flushed before the crash, replaying the log record writes the same value again — harmless. This means recovery can always start from a known-good checkpoint and replay forward without worrying about double-applying changes.

Note: The page's on-disk LSN (stored in the page header) tells the engine whether a log record has already been applied. If page.pageLSN >= record.lsn, the redo is skipped — the page is already up to date.

Undo: Erasing Uncommitted Work

The opposite problem: a transaction was in flight when the crash occurred. It may have written dirty pages to disk (the buffer pool flusher doesn't know or care about commit status). Those partial writes must be completely removed.

Undo walks the log backward through the transaction's records, applying the before-image (old value) of each change:

Log record:  LSN 198 | txn T9 | UPDATE orders | page 31 offset 44
             old_val = 'pending'  new_val = 'shipped'

Undo action: write old_val ('pending') back to page 31, offset 44

Each undo action is itself logged as a Compensation Log Record (CLR). CLRs are critical for correctness: if the database crashes again during recovery (a real scenario), it must not undo changes that have already been undone. A CLR marks "this undo step is done," so re-running recovery skips it.

The Order Matters: Redo Before Undo

This is the counter-intuitive part. You might expect recovery to skip undo-only transactions entirely during redo — why replay changes you're about to reverse? ARIES (the recovery protocol used by PostgreSQL, SQL Server, DB2, and others) does not do this.

ARIES always redoes first, then undoes. The reason is correctness under repeated crashes. A transaction might have been mid-rollback when the crash happened — some of its changes were already reversed, some were not. ARIES reconstructs the exact crash state (including partial rollbacks recorded as CLRs), then undoes whatever remains. This ensures recovery is always correct no matter how many times it is interrupted.

PostgreSQL vs. ARIES: Two Different Approaches to Undo

The undo mechanism described above (reversing changes via before-images in the WAL) is the ARIES protocol used by MySQL InnoDB, SQL Server, and DB2. The WAL stores both before- and after-images, and recovery walks backward to restore old values.

PostgreSQL takes a fundamentally different approach. PostgreSQL's WAL records contain only redo information (the after-image). There is no undo log and no undo pass during recovery. Instead, PostgreSQL relies on MVCC (Multi-Version Concurrency Control):

  • Every UPDATE or DELETE creates a new row version rather than overwriting the old one. The old row versions remain in the heap pages.
  • Each row version carries visibility markers (xmin/xmax) recording which transaction created and deleted it.
  • After a crash, recovery replays the WAL to restore committed changes. Uncommitted transactions are not physically rolled back — their row versions simply remain invisible because the transaction that created them is marked as aborted/crashed.
  • The old row versions are eventually cleaned up by VACUUM, which removes rows no longer visible to any transaction.

Key difference: ARIES does undo-by-reversing; PostgreSQL does undo-by-ignoring (old data is already there, new data is marked invisible). This is why PostgreSQL recovery has no "undo phase" — it only needs the redo pass, then marks crashed transactions as aborted. The trade-off is that PostgreSQL accumulates dead tuples that need periodic VACUUM.

Key Differences at a Glance

Redo Undo
Direction Forward through log Backward through log
Goal Re-apply committed changes missing from disk Erase uncommitted partial changes
Uses After-image (new value) Before-image (old value)
Logged? No extra record needed Yes — writes a CLR to survive re-crash
Idempotent? Yes (page LSN check skips re-done work) Yes (CLR prevents re-undoing)

Together, redo and undo make crash recovery correct under any failure scenario, including crashes that happen mid-recovery. The log is the source of truth; the data files are just a cache of what the log already durably records.
