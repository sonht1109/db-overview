Part VII. Building Understanding

Chapter 30. Where to Go Next

Open source databases to explore

The open-source database ecosystem is unusually rich: you can read the source, file issues, follow mailing lists, and even contribute to systems that run at internet scale. Knowing which projects are worth studying — and what each one teaches — is itself a skill. This page maps the landscape so you can pick your next deep dive.

How to Use This List

The projects below are organized by what they teach you, not just what category they fall into. Pick one from each major category you want to understand, read the paper (if one exists), build it locally, and run the test suite. That combination — paper + code + tests — is the fastest path to genuine understanding.

Relational (OLTP)

These are traditional SQL databases optimized for transactional workloads.

Project Language What it teaches Paper / Resource
SQLite C B-tree storage, VDBE bytecode interpreter, WAL mode sqlite.org/arch.html
PostgreSQL C Full RDBMS: MVCC, planner, extensions, WAL, vacuum postgresql.org/docs/current/internals.html
MySQL / InnoDB C++ Clustered indexes, doublewrite buffer, redo log InnoDB internals guide
MariaDB C++ PostgreSQL/MySQL alternative; Galera cluster replication mariadb.com/kb/en/documentation/
DuckDB C++ Columnar OLAP embedded in a library; vectorized execution "DuckDB: an Embeddable Analytical Database" (SIGMOD 2019)

Start with SQLite if you want to read every subsystem end-to-end in a manageable codebase. PostgreSQL if you want a full production RDBMS with excellent inline documentation.

Key-Value and Log-Structured

Project Language What it teaches Paper

LMDB is underrated for learning. It is under 10,000 lines of C, implements a full copy-on-write B-tree with MVCC, and has exceptional documentation. Read it after SQLite for a completely different design philosophy.

Where to Start

If you have studied this course in order, a recommended exploration path:

  1. SQLite — study the B-tree and WAL implementation. Read btree.c and wal.c.
  2. PostgreSQL — study MVCC and the query planner. Read heapam.c and optimizer/.
  3. RocksDB — study LSM-tree compaction. Read compaction/compaction_picker.cc.
  4. DuckDB — study vectorized execution. Read execution/operator/.
  5. CockroachDB — study distributed transactions. Read the Raft and MVCC layers.

Each step builds on the previous one. You will recognize patterns from one engine appearing in the next — that recognition is the payoff.

Key Takeaways

  • The open-source ecosystem covers every database category; you can read production code for all of them.
  • SQLite and LMDB are the best starting points for readable, self-contained storage engines.
  • PostgreSQL and RocksDB are the best choices for studying a full production system.
  • DuckDB and CockroachDB represent the modern generation: columnar OLAP and distributed SQL respectively.
  • Follow the recommended path: single-node B-tree → full RDBMS → LSM-tree → columnar → distributed.
