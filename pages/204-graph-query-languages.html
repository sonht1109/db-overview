Part V. Database Families

Chapter 21. Graph Databases

Graph query languages

Graph databases need query languages that can express path patterns, variable-depth traversals, and relationship conditions naturally — things SQL was never designed for. Several languages have emerged, each with a different philosophy. This page covers the major contenders: Cypher, Gremlin, SPARQL, and the emerging ISO standard GQL.

Why SQL Falls Short for Graphs

SQL can express graph queries — recursive CTEs (Common Table Expressions) let you walk edges — but the syntax is verbose, the depth must often be bounded at compile time, and the query planner is not optimized for pointer-chasing. A three-hop path query in SQL is already painful; a variable-length path query is nearly unwritable.

Graph query languages solve this by treating path patterns as first-class syntax.

Cypher: Pattern Matching with ASCII Art

Cypher is the query language of Neo4j (open-sourced as openCypher). Its defining feature is an ASCII-art notation for graph patterns that reads almost like a diagram drawn in text.

(a:Person)-[:KNOWS]->(b:Person)

Round brackets () denote nodes; square brackets [] denote relationships; - and -> indicate direction. You read this exactly as it looks: "a Person node connected via a KNOWS edge to another Person node."

Core Cypher Clauses

Clause Purpose SQL analogue
MATCH Declare a pattern to find FROM + JOIN
WHERE Filter matched results WHERE
RETURN Specify output SELECT
CREATE Insert nodes or edges INSERT
MERGE Upsert — create if not found INSERT OR IGNORE + UPDATE
SET Update properties UPDATE … SET
DELETE Remove nodes or edges DELETE

Cypher Query Examples

Find all people Alice knows:

MATCH (alice:Person {name: "Alice"})-[:KNOWS]->(friend:Person)
RETURN friend.name;

Friend of friend (2 hops) — depth is a parameter, not a join:

MATCH (alice:Person {name: "Alice"})-[:KNOWS*2]->(fof:Person)
WHERE fof.name <> "Alice"
RETURN DISTINCT fof.name;

The *2 means exactly 2 hops. Use *1..3 for variable depth (1 to 3 hops). There is no rewrite needed — just change the number.

Create a new relationship with a property:

MATCH (a:Person {name: "Alice"}), (b:Person {name: "Bob"})
MERGE (a)-[:FOLLOWS {since: "2024-01"}]->(b);

Shortest path between two nodes:

MATCH p = shortestPath(
  (alice:Person {name:"Alice"})-[:KNOWS*]-(target:Person {name:"Eve"})
)
RETURN length(p), [n IN nodes(p) | n.name] AS path;

Gremlin: Imperative Traversal Steps

Gremlin is the query language of Apache TinkerPop, a graph computing framework used by Amazon Neptune, JanusGraph, and others. Where Cypher is declarative (describe the pattern, let the engine plan), Gremlin is imperative: you describe the exact traversal steps.

// Start at Alice, follow outgoing KNOWS edges, return names
g.V().has('name', 'Alice').out('KNOWS').values('name')

// Friend of friend
g.V().has('name', 'Alice').out('KNOWS').out('KNOWS').values('name').dedup()

// Variable depth: 1 to 3 hops
g.V().has('name', 'Alice').repeat(out('KNOWS')).times(2).values('name')

// Shortest path
g.V().has('name','Alice').
  repeat(both().simplePath()).
  until(has('name','Eve')).
  path().
  limit(1)

Gremlin steps chain like a pipeline: each step produces a traverser that the next step filters or transforms. This is familiar to developers comfortable with functional pipelines.

Gremlin vs Cypher: Cypher is easier to read for graph patterns; Gremlin gives finer procedural control and works across multiple backends. Choose based on your database (Neo4j → Cypher; Neptune/JanusGraph → Gremlin).

SPARQL: Querying RDF Graphs

SPARQL targets a different graph model: RDF (Resource Description Framework), used in semantic web and linked data. In RDF, all data is triples: (subject, predicate, object). A graph is a bag of triples.

PREFIX foaf: <http://xmlns.com/foaf/0.1/>

SELECT ?friendName
WHERE {
  ?alice foaf:name "Alice" .
  ?alice foaf:knows ?friend .
  ?friend foaf:name ?friendName .
}

SPARQL is critical for knowledge graphs, ontology reasoning, and linked open data (Wikidata, DBpedia). For property graph use cases, Cypher or Gremlin are more common.

GQL: The ISO Standard

GQL (Graph Query Language) was ratified as ISO/IEC 39075 in 2024 — the first international standard for graph databases, filling the same role SQL plays for relational systems. GQL is heavily influenced by Cypher and incorporates ideas from PGQL and G-CORE.

-- GQL (ISO 39075) — syntax resembles Cypher
MATCH (p:Person)-[:KNOWS]->(friend:Person)
WHERE p.name = 'Alice'
RETURN friend.name;

GQL's significance: vendors can implement it without a license, and tooling can target a single standard. It is still early (2024–2025 adoption), but most major graph databases have announced GQL roadmaps.

Key Takeaways

  • Cypher (openCypher / Neo4j) uses ASCII-art pattern syntax and is the most readable graph query language for property graphs; variable-depth traversal is a first-class feature.
  • Gremlin (Apache TinkerPop) takes an imperative pipeline approach, giving developers fine-grained control over traversal strategy and working across multiple graph backends.
  • SPARQL is the W3C standard for RDF / semantic web graphs and underpins knowledge graphs and linked open data.
  • GQL (ISO/IEC 39075, 2024) is the emerging international standard for property graph queries, designed to be to graphs what SQL is to relational databases.
  • The core advantage of all graph query languages over SQL: variable-depth traversal is a parameter, not a structural rewrite. [:KNOWS*1..5] in Cypher replaces five pairs of joins in SQL.
  • Adoption is still fragmented, but GQL standardization will drive convergence over the next decade.
