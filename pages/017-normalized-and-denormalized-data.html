Chapter 2. How Data Is Organized

Normalized and denormalized data

When you store the same fact in multiple places, you invite trouble: update it in one row and forget the others, and your database starts disagreeing with itself. Normalization is the practice of organizing data to eliminate that redundancy. Denormalization is the deliberate decision to reintroduce some redundancy for the sake of read performance. Both are tools — understanding when to use each is one of the most practical skills in database design.

The Problem Normalization Solves

Imagine a single table that tracks every order a customer has ever placed:

order_id customer_name customer_email product price
1 Alice Wong alice@example.com Headphones 79.99
2 Alice Wong alice@example.com USB Cable 9.99
3 Bob Patel bob@example.com Keyboard 49.99

Alice's email appears twice. If she changes it, you must update every row — miss one, and your data is inconsistent. This is called an update anomaly. There are also insertion anomalies (you can't record a customer until they place an order) and deletion anomalies (deleting the last order erases the customer record entirely).

Normalization fixes these by splitting the table so each fact lives in exactly one place.

Normal Forms: The Key Ideas

Normalization is defined in levels called normal forms (1NF, 2NF, 3NF, and beyond). In practice, reaching Third Normal Form (3NF) is the standard target for transactional databases.

Normal Form Rule (simplified)
1NF Every cell holds one atomic value; no repeating groups
2NF 1NF + every non-key column depends on the whole primary key
3NF 2NF + no non-key column depends on another non-key column

For the orders example, a 3NF design separates customers from orders:

-- Customers: each email stored once
CREATE TABLE customers (
  id    SERIAL PRIMARY KEY,
  name  TEXT NOT NULL,
  email TEXT NOT NULL UNIQUE
);

-- Orders: reference the customer by id
CREATE TABLE orders (
  id          SERIAL PRIMARY KEY,
  customer_id INTEGER REFERENCES customers(id),
  product     TEXT NOT NULL,
  price       REAL NOT NULL
);

Now Alice's email exists in exactly one row. Change it once, and every order automatically reflects the update.

When Denormalization Makes Sense

Normalization is great for writes, but reads can suffer. Every time you need the customer's name alongside an order, you must perform a JOIN — and joins across large tables are expensive.

Denormalization trades storage and write complexity for faster reads. A reporting table might intentionally store customer_name directly on the order row so a dashboard query never needs a join:

-- Denormalized orders table (for read-heavy analytics)
CREATE TABLE orders_flat (
  order_id      SERIAL PRIMARY KEY,
  customer_name TEXT,
  customer_email TEXT,
  product       TEXT,
  price         REAL
);

This pattern is common in data warehouses and analytical (OLAP) workloads, where data is loaded in bulk and queried heavily but updated rarely.

Note: In a denormalized table, updating Alice's email requires touching every order row she has. That's acceptable in a warehouse loaded nightly from a source-of-truth system — but would be a maintenance nightmare in a live transactional database.

Choosing the Right Approach

Normalized Denormalized
Best for Transactional apps (OLTP) Analytics, reporting (OLAP)
Update cost Low — change once High — change many rows
Read cost Higher — requires joins Low — single table scan
Data integrity Strong Relies on ETL discipline

Most real systems use both: a normalized operational database feeding a denormalized data warehouse or read replica. The key insight is that normalization is not an absolute law — it is a default that you deviate from consciously, with a clear reason.
