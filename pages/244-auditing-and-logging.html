Chapter 25. Security and Access Control

Auditing and logging

When a security incident happens, the first question is always: what happened, who did it, and when? Auditing and logging are how databases answer that question. An audit trail is not just a compliance checkbox — it is the difference between a contained breach and one that goes undetected for months. This page covers what to log, how to configure it, and how to query audit data effectively.

What to Log

Not all database activity is equally worth capturing. Over-logging generates noise that buries real signals; under-logging leaves gaps attackers exploit. A useful audit strategy logs at three levels:

Level What Why
Connection events Login attempts, successful connections, disconnections Detect brute-force, credential stuffing, unexpected service accounts logging in
DDL events CREATE, ALTER, DROP, TRUNCATE Schema changes are irreversible; catch accidental or malicious structural changes
DML on sensitive tables SELECT, INSERT, UPDATE, DELETE on PII/financial tables Who read or changed sensitive data and when
Privilege changes GRANT, REVOKE, role membership changes Privilege escalation is a common attack vector
Failed queries / errors Permission-denied errors, syntax errors in bulk Probing and injection attempts often leave error traces

PostgreSQL Audit Configuration

PostgreSQL's built-in logging is configured in postgresql.conf:

# Log all connections and disconnections
log_connections = on
log_disconnections = on

# Log all DDL statements
log_statement = 'ddl'

# Log slow queries (useful for performance and unexpected bulk reads)
log_min_duration_statement = 1000  # milliseconds

# Log failed authentication
log_failed_connections = on  # (available in PG 15+)

For finer-grained DML auditing, the community extension pgAudit (PostgreSQL Audit Extension) logs individual SELECT, INSERT, UPDATE, and DELETE statements with full context:

-- Install the extension (requires superuser)
CREATE EXTENSION pgaudit;

-- Enable pgAudit for a specific role
SET pgaudit.log = 'read,write';
SET pgaudit.log_relation = on;

pgAudit emits structured log lines that include the statement class, object type, object name, and the full SQL text — far more useful than the raw PostgreSQL log.

Audit Tables: Logging Data Changes

For application-level auditing — tracking who changed what value in a business table — a trigger-based audit table is the most common pattern:

-- Audit table: stores old and new values for every change
CREATE TABLE audit_log (
  log_id      SERIAL PRIMARY KEY,
  table_name  TEXT NOT NULL,
  operation   TEXT NOT NULL,          -- INSERT / UPDATE / DELETE
  changed_by  TEXT NOT NULL,          -- current_user
  changed_at  TEXT NOT NULL,          -- timestamp
  old_data    TEXT,                   -- JSON of old row
  new_data    TEXT                    -- JSON of new row
);

-- Trigger function
CREATE OR REPLACE FUNCTION audit_trigger_fn()
RETURNS trigger LANGUAGE plpgsql AS $
BEGIN
  INSERT INTO audit_log(table_name, operation, changed_by, changed_at, old_data, new_data)
  VALUES (
    TG_TABLE_NAME,
    TG_OP,
    current_user,
    to_char(now(), 'YYYY-MM-DD"T"HH24:MI:SS'),
    CASE WHEN TG_OP = 'DELETE' THEN row_to_json(OLD)::text ELSE NULL END,
    CASE WHEN TG_OP IN ('INSERT','UPDATE') THEN row_to_json(NEW)::text ELSE NULL END
  );
  RETURN NULL;
END;
$;

The trigger fires AFTER INSERT OR UPDATE OR DELETE on each monitored table and writes a row to audit_log. The audit log itself should be insert-only: revoke UPDATE and DELETE privileges from all application roles.

What Makes a Good Audit Log

An audit log entry should contain enough context to reconstruct exactly what happened:

  • Who — the database user (current_user), plus the application-level user if your app tracks that in a session variable
  • When — timestamp with time zone (not just a date)
  • What — the operation type and the affected table
  • Old state / New state — for UPDATE and DELETE, the previous values; for INSERT, the new values
  • Where from — client IP address (available in inet_client_addr() in PostgreSQL)

Application-Level User Tracking

The database only knows the database role. If five hundred users share one app_user role, the audit log shows app_user for all of them. Pass the application user ID through a session variable:

-- Set at the start of each session
SELECT set_config('app.user_id', '12345', false);

-- Read in the trigger
current_setting('app.user_id', true)

Protecting the Audit Log

An audit log is only useful if it cannot be tampered with. Apply the following:

  1. Revoke modification rights — application roles should have INSERT only on audit_log, never UPDATE or DELETE.
  2. Ship logs off the database — stream logs to a separate system (Splunk, Datadog, S3) in real time. If an attacker compromises the database, they should not also be able to erase the trail.
  3. Immutable log stores — cloud storage with Object Lock (S3 Object Lock, GCS retention policies) makes log files legally immutable for a configured retention period.
  4. Rotate and archive — partition the audit log by month; archive old partitions to cold storage to keep the active table fast.

Key Takeaways

  • Log connections, DDL, privilege changes, and DML on sensitive tables at minimum; avoid logging everything (it creates noise and storage pressure).
  • Use trigger-based audit tables to capture old and new values for changes — the database log alone does not record this.
  • Always record who, what, when, and from where — including the application-layer user ID if multiple users share a database role.
  • Protect the audit log itself: insert-only for application roles, shipped to an external immutable store.
  • An audit log that exists only inside the database is a liability: a compromised database can be used to erase the trail.
