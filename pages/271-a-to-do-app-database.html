Chapter 28. Case Studies

A to-do app database

A to-do app is the "hello world" of database design — small enough to understand completely, yet rich enough to introduce the patterns that appear at every scale. This case study builds a production-quality schema step by step, explaining every design decision along the way.

Requirements and Access Patterns

Before touching a schema, list what the app must do:

  • Users sign up and own lists of tasks.
  • Tasks have a title, an optional due date, a completion flag, and optional tags.
  • Tasks can belong to one list; lists belong to one user.
  • The app frequently renders "all incomplete tasks in this list, ordered by due date."
  • Tag filtering ("show tasks tagged #work") must also be fast.

These requirements drive every decision below.

Core Schema

CREATE TABLE users (
  id        INTEGER PRIMARY KEY,
  email     TEXT    NOT NULL UNIQUE,
  name      TEXT    NOT NULL,
  created_at INTEGER NOT NULL  -- Unix epoch seconds
);

CREATE TABLE lists (
  id         INTEGER PRIMARY KEY,
  user_id    INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title      TEXT    NOT NULL,
  created_at INTEGER NOT NULL
);

CREATE TABLE tasks (
  id          INTEGER PRIMARY KEY,
  list_id     INTEGER NOT NULL REFERENCES lists(id) ON DELETE CASCADE,
  title       TEXT    NOT NULL,
  notes       TEXT,
  due_at      INTEGER,          -- NULL means no due date
  completed   INTEGER NOT NULL DEFAULT 0,  -- 0 or 1
  position    REAL    NOT NULL DEFAULT 0,  -- fractional for drag-sort
  created_at  INTEGER NOT NULL
);

CREATE TABLE tags (
  id   INTEGER PRIMARY KEY,
  name TEXT NOT NULL UNIQUE
);

CREATE TABLE task_tags (
  task_id INTEGER NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  tag_id  INTEGER NOT NULL REFERENCES tags(id)  ON DELETE CASCADE,
  PRIMARY KEY (task_id, tag_id)
);

Why These Columns?

Choice Reasoning
completed INTEGER not BOOLEAN SQLite has no boolean type; 0/1 integer is the idiom and compresses well
position REAL Fractional (gap) indexing lets you insert between two items without renumbering: insert between 1.0 and 2.0 → 1.5
due_at INTEGER Storing Unix epoch as integer is timezone-agnostic and sortable with a plain < compare
Separate tags table Avoids tag duplication and makes rename-a-tag trivial
task_tags junction Many-to-many; the composite PRIMARY KEY doubles as a unique index

Key Indexes

The query "all incomplete tasks in this list, ordered by due date" runs on every page load:

CREATE INDEX idx_tasks_list_incomplete
  ON tasks (list_id, completed, due_at);

This composite index satisfies WHERE list_id = ? AND completed = 0 ORDER BY due_at with a single B-tree scan — no sort step needed because due_at is already in index order.

For tag filtering:

CREATE INDEX idx_task_tags_tag ON task_tags (tag_id, task_id);

A query like "tasks in list 5 with tag 'work'" becomes:

SELECT t.*
FROM tasks t
JOIN task_tags tt ON tt.task_id = t.id
WHERE t.list_id = 5
  AND t.completed = 0
  AND tt.tag_id = (SELECT id FROM tags WHERE name = 'work')
ORDER BY t.due_at;

Design Decisions Worth Discussing

Soft Delete vs. Hard Delete

Many apps add deleted_at TIMESTAMP instead of actually removing rows. That gives you an "undo" feature and audit trail. The trade-off: every query must add WHERE deleted_at IS NULL, your indexes must include that column, and the table grows forever. For a simple to-do app, hard delete with ON DELETE CASCADE is usually the right call — simpler, smaller, faster.

Fractional Position for Drag-Sort

Storing sort order as a REAL column avoids the O(n) renumbering problem. If a user drags task C between A (position=1.0) and B (position=2.0), you write one row: UPDATE tasks SET position = 1.5 WHERE id = ?. The only edge case is floating-point exhaustion after many re-sorts — a rare event easily handled by a periodic "re-gap" operation that resets positions to 1, 2, 3 … n.

One Database per User vs. Shared

At small scale, all users share one database (as shown). At large scale, some apps shard by user (each user gets their own SQLite file or PostgreSQL schema). This eliminates cross-user query interference but complicates "global" queries like "top 100 most-used tags across all users."

Key Takeaways

  • Model access patterns first: the index on (list_id, completed, due_at) was chosen because it mirrors the most frequent query exactly.
  • Fractional positions solve the drag-sort problem elegantly — one write, not N.
  • Junction tables (task_tags) are the clean many-to-many solution; avoid storing tags as comma-separated strings in a column.
  • Cascade deletes keep referential integrity automatic when you hard-delete.
  • Even a simple app produces non-trivial design decisions; the right choices depend on your scale, query patterns, and feature roadmap.
