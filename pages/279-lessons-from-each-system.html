Part VII. Building Understanding

Chapter 28. Case Studies

Lessons from each system

Each case study in this chapter was a real design problem, and each one taught something that generalises beyond its specific domain. This closing page distills the cross-cutting lessons — the patterns and principles that appear in every well-designed database system, regardless of scale or use case.

Lesson 1: Design for Reads, Not for Writes

Applications almost always read far more than they write. The to-do app, the e-commerce store, and the chat system all write once and read hundreds of times. The lesson is to design your schema and indexes around your most frequent read queries, and accept that writes take slightly more work.

This means:

  • Put the right columns in indexes (leading with the most selective filter first).
  • Pre-compute values that are expensive to aggregate at read time (the analytics pipeline's agg_daily_traffic).
  • Accept denormalization when it eliminates a slow join on the hot path.

Counter-balance: Time-series and analytics pipelines invert this: writes are the bottleneck. In those systems, design for write throughput (sequential appends, no secondary indexes on raw data) and accept slower ad-hoc reads in exchange.

Lesson 2: Model Your Access Patterns Before Your Entities

The most common mistake in schema design is drawing an entity-relationship diagram based on the business domain, then wondering later why queries are slow. The better order:

  1. List your queries — every distinct read pattern the app will run.
  2. List your writes — inserts, updates, deletes and how frequently each occurs.
  3. Then design the schema and indexes to serve those patterns.

The chat system's server_seq column exists solely because of the access pattern "load the last 50 messages in a conversation ordered by sequence." The multi-tenant SaaS's leading tenant_id in every index exists because every query is scoped to one tenant. The entity-relationship diagram alone wouldn't have told you either of those things.

Lesson 3: Timestamps Are Cheap — Audit Trails Are Priceless

Every table in every system should have created_at. Most tables should have updated_at. High-stakes tables (orders, payments, status changes) should have a separate _history table with full row snapshots.

The e-commerce order that records its address and unit price at purchase time, the chat message that records sent_at and server_seq, the analytics pipeline's raw events that are never modified — these are all applications of the same principle: immutable historical records prevent a category of bugs that are otherwise nearly impossible to debug.

Lesson 4: Keep the Write Path Simple

Every application layer between a write and durable storage is a potential failure point. The analytics pipeline's principle of idempotent ETL (delete + re-insert, not update-in-place) is a general rule: make your write operations safe to re-run. This applies to:

  • Database migrations (should be runnable more than once without breaking).
  • ETL jobs (should produce the same output given the same input).
  • API endpoints that create resources (should be idempotent when given the same request ID).

Lesson 5: Normalize Until It Hurts, Then Denormalize Deliberately

Start normalized. The to-do app's tags table is separate from tasks. The time-series schema's metric_names is separate from metrics. The search schema's documents is separate from doc_search.

Then, when you profile real queries and find that a join is responsible for a significant share of read latency, denormalize that specific join — store a redundant copy of the data in the row that needs it. Document the denormalization and the procedure for keeping the copies in sync.

Never pre-emptively denormalize. It adds complexity before you have evidence the complexity is necessary.

Lesson 6: Choose the Right Tool for the Workload

The case studies deliberately span different database types:

System Dominant characteristic Best tool
To-do app OLTP, low volume SQLite / PostgreSQL
E-commerce OLTP, transactions, concurrency PostgreSQL / MySQL
Chat High-frequency writes + reads, ordering PostgreSQL + Redis for fan-out
Analytics pipeline Batch reads, columnar scans DuckDB / BigQuery / ClickHouse
Time-series metrics Append-only, retention, rollups TimescaleDB / InfluxDB
Search index Inverted index, relevance ranking Elasticsearch / Typesense
Multi-tenant SaaS OLTP, row-level isolation PostgreSQL with RLS

None of these tools is universally best. PostgreSQL is an excellent default that covers most OLTP workloads. But when your access pattern fundamentally conflicts with a relational B-tree (e.g., full-text ranking, sub-second time-series ingestion, graph traversal), switching tools pays off faster than you expect.

Final Reflection

Good database design is not about memorizing rules. It is about asking the right questions:

  • What are my read queries? (Design indexes for them.)
  • What must never be lost? (Make it append-only and immutable.)
  • What grows without bound? (Build in retention and pagination.)
  • What must be consistent? (Put it in a transaction.)
  • What is expensive to compute? (Pre-aggregate it.)

Every schema you write is a set of bets about how the data will be used. Make those bets explicit, document them, profile them in production, and update them when reality proves them wrong.

Key Takeaways

  • Design for reads first — your indexes are predictions about which queries matter most.
  • Access patterns before entities — understand how data flows before you draw the schema.
  • Timestamps and history tables are cheap insurance against a category of hard-to-debug errors.
  • Idempotent writes make systems resilient to retries, restarts, and partial failures.
  • Normalize first, denormalize with evidence — premature denormalization is complexity without benefit.
  • Choose the right tool — no single database technology is optimal for every workload class.
