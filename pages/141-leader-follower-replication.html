Part IV. Distributed Systems

Chapter 15. Replication

Leader-follower replication

When a single database server fails, everything that depends on it fails too. Replication solves this by keeping copies of your data on multiple machines. Leader-follower replication (also called primary-replica or master-slave replication) is the most widely deployed pattern — used by PostgreSQL, MySQL, MongoDB, and many others.

How It Works

The idea is simple: one node is designated the leader (primary). All writes go to the leader. The leader records every change and streams those changes to one or more followers (replicas). Followers apply the changes in the same order and stay in sync.

The mechanism used to stream changes varies by database:

Method How it works Used by
WAL shipping Send the raw write-ahead log segments PostgreSQL (streaming replication)
Row-based binlog Log each row's before/after values MySQL (row-based replication)
Logical log Log high-level operations decoded from WAL PostgreSQL logical replication
Trigger-based Application triggers write to a change table Older or heterogeneous setups

Synchronous vs. Asynchronous Replication

The crucial question is: when does the leader confirm a write to the client?

  • Asynchronous replication — the leader commits and replies immediately, then sends the log to followers in the background. Fastest, but if the leader crashes before followers catch up, the un-replicated writes are lost.
  • Synchronous replication — the leader waits for at least one follower to acknowledge before replying. The write is durable on two machines, but every commit pays the network round-trip latency.

Most systems use a hybrid: one synchronous follower (guaranteeing durability) and the rest asynchronous (for throughput). PostgreSQL calls this synchronous_standby_names.

Note: "Semi-synchronous" replication in MySQL similarly waits for at least one replica to write the event to its relay log — not necessarily apply it — before the leader acks the client.

Failover

When the leader fails, one follower must be promoted to become the new leader. This process — failover — involves:

  1. Detecting that the leader is gone (heartbeat timeout).
  2. Electing the most up-to-date follower as the new leader.
  3. Reconfiguring clients to send writes to the new leader.
  4. Old followers starting to replicate from the new leader.

Failover sounds straightforward but hides serious edge cases:

  • The old leader may still be alive (network partition, not a crash) — now you have two nodes that both think they're leader, called split-brain. Both accept writes, and the data diverges.
  • The promoted follower may be behind: any writes acknowledged by the old leader but not yet replicated are simply lost if you discard them, or must be re-applied somehow.

Tools like Patroni (PostgreSQL) and MHA (MySQL) automate failover while managing these hazards.

Read Scaling and Replication Lag

Because followers serve reads, adding replicas lets you scale read throughput horizontally. This is why read-heavy applications (analytics dashboards, product listings) commonly run queries against replicas.

The catch is replication lag — followers apply changes after the leader, so a read from a follower may return slightly stale data. The time gap is usually milliseconds on a healthy LAN, but can grow under load or network issues.

Note: This creates a weaker consistency guarantee called eventual consistency: followers will eventually catch up, but there is no guarantee about when. Chapter 16 examines consistency models in depth.

A common symptom is "read your own writes" breakage: a user submits a form (write goes to leader), then immediately reloads (read hits a lagging follower) and sees the old data. The fix is to route writes and the reads immediately following them to the leader, or use a replication position token to wait until a follower has caught up to the write.
