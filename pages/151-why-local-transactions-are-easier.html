Part IV. Distributed Systems

Chapter 16. Distributed Transactions and Consensus

Why local transactions are easier

When all your data lives on a single database node, committing a transaction is straightforward: one machine decides, and it is done. The moment data is spread across multiple nodes, that simplicity evaporates. Understanding why single-node transactions are so much easier is the best foundation for understanding the hard problems that distributed transactions must solve.

The Single-Node Advantage

On a local database, a transaction involves exactly one actor making exactly one decision. The engine can use a simple, well-understood model:

  1. Write changes to a write-ahead log (WAL) on disk.
  2. If the process crashes before the commit record, recovery replays the log and rolls back.
  3. If the commit record is written, the transaction is durable — full stop.

There is no need to coordinate with anyone else. The database's lock manager, buffer pool, and transaction log are all in the same process, sharing the same memory. Checking whether a lock is held is a pointer lookup, not a network round-trip.

What Changes Across Nodes

In a distributed system, a transaction might touch rows on Node A (the orders table) and Node B (the inventory table). Now there are two machines, each with their own disk, and each must agree on whether the transaction committed or not.

The core difficulty is that partial failures are possible. Node A could commit successfully while Node B crashes mid-write, leaving the system in an inconsistent state. On a single node, "partial commit" is impossible — the commit record either made it to the WAL or it did not.

Property Single node Distributed
Commit decision One machine Must agree across N machines
Partial failure Impossible (one WAL) Possible — any node can crash independently
Lock check In-memory pointer lookup Requires network message
Rollback on crash WAL replay on restart Need protocol to agree on outcome
Latency Microseconds Milliseconds (network RTT per step)

The Atomicity Problem in Two Sentences

A local transaction gives you atomicity "for free" because the commit log is a single file on a single disk: writing that one record is itself an atomic operation at the OS level. A distributed transaction must simulate that single atomic moment across independent machines that can fail, lose messages, or become temporarily unreachable — and that simulation is what makes protocols like Two-Phase Commit (2PC) necessary.

Note: This is why many high-traffic systems deliberately avoid distributed transactions by redesigning data so that all rows a transaction needs live on the same node (a technique called co-location or shard-local transactions). Avoiding the problem is often cheaper than solving it.
