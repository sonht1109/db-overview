Part II. Querying and Language

Chapter 6. How Query Processing Works

Chapter 6 Summary: What We Learned

Page Topic Takeaway in One Sentence
051 Parsing and validation The parser checks syntax and resolves table/column names against the schema catalog.
052 Logical plans A logical plan is an abstract tree of relational operations (scan, filter, join) that represents what the query means.
053 Physical plans A physical plan adds implementation details: which index to use, which join algorithm, sort vs hash.
054 Operators such as scan, filter, join Each plan node is an operator; the executor calls iterators (open, next, close) to pull rows through the tree.
055 Cost estimation The optimizer uses table statistics (row counts, value distributions) to estimate how expensive each plan will be.
056 Rule-based and cost-based planning Rule-based optimizers apply fixed heuristics; cost-based optimizers enumerate alternatives and pick the cheapest.
057 Why the same query can run differently Statistics change, schemas change, data grows — the optimizer can choose a new plan at any time.
058 Reading an explain plan EXPLAIN shows the plan tree; start from the innermost node, check estimated vs actual row counts, and look for full scans.
059 Query planning mistakes The optimizer is not omniscient — stale statistics, correlated columns, and UDFs can lead to bad plans.
Reveal: How do these ideas connect to the next chapter?
Chapter 6 showed you the query execution pipeline — parsing, planning, optimization, and execution. You learned that the optimizer chooses between scanning and seeking based on what is available. Chapter 7 dives into the single most important thing you can provide to the optimizer: indexes. You will learn B-trees, hash indexes, composite indexes, covering indexes, and the critical truth that indexes speed up reads but slow down writes. This is where performance tuning begins.
