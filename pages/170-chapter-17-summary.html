Part IV. Distributed Systems

Chapter 17. Consistency in Distributed Databases

Chapter 17 Summary: What We Learned

Chapter Contents

Page Topic Takeaway in One Sentence
161 Strong consistency After a write completes, all subsequent reads return the new value — as if there is only one copy of the data.
162 Eventual consistency If no new updates are made, eventually all replicas converge to the same value — but reads may see stale data in the meantime.
163 Read-your-writes A user should always see their own updates immediately — requires routing reads to the same replica that handled the write.
164 Monotonic reads A user should never see data go backwards in time — each read should return data at least as fresh as the previous read.
165 Causal consistency Operations that are causally related must be seen in the same order by everyone; unrelated operations can be seen in any order.
166 CAP as a tradeoff lens During a network partition, you must choose between Consistency (correct answers) and Availability (any answer at all) — not both.
167 Latency vs correctness Stronger consistency guarantees require more coordination, which adds latency — this is a fundamental tradeoff, not a bug.
168 Conflict-free approaches CRDTs, last-write-wins, and application-level merging let systems avoid coordination while still making progress.
169 Picking the right guarantee Not every part of your application needs strong consistency — match the guarantee to the business requirement.
