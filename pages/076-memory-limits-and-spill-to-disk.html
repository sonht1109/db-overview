Chapter 8. Joins, Aggregation, and Sorting

Memory limits and spill to disk

Every operation you have studied so far — hash joins, sort-merge joins, GROUP BY, ORDER BY — needs somewhere to hold intermediate data while it works. That "somewhere" is RAM. When the data fits, things are fast. When it does not, the engine has two choices: fail, or find a way to keep going by using disk as overflow storage. This overflow is called spill to disk, and understanding it explains why some queries are dramatically slower than others despite looking nearly identical.

Why Memory Budgets Exist

A database engine serves many queries at once. If every query could grab as much RAM as it wanted, a single heavy ORDER BY could starve every other session. So engines enforce a per-operation memory budget — a cap on how much RAM one sort, hash table, or aggregate can consume.

Database Setting Default
PostgreSQL work_mem 4 MB
MySQL sort_buffer_size 256 KB
SQL Server Query memory grant Estimated at plan time
SQLite Controlled by host application Varies

The defaults are intentionally conservative. Multiply the per-operation default by the number of concurrent connections and you get the worst-case memory pressure — 100 connections each using 4 MB of work_mem is 400 MB just for sort buffers.

Note: In PostgreSQL, work_mem applies per sort or hash operation, not per query. A single complex query can have multiple such operations running simultaneously, each consuming up to work_mem.

What Happens When You Exceed the Budget

When intermediate data grows past the memory budget, the engine spills the overflow to a temporary file on disk. The exact mechanism depends on the operation:

  • Sort spill: Sorted runs are written to temp files and merged (the external sort algorithm from the previous page).
  • Hash join spill: The build-side hash table is partitioned; partitions that exceed memory are written to disk and processed in batches.
  • Hash aggregate spill: Partial aggregation results are flushed to disk and combined in a second pass.

In all cases the query still returns correct results. The penalty is speed, not correctness.

The Cost Cliff

Spill is not a gentle slowdown — it is often an order-of-magnitude jump in query time. Disk throughput is typically 10–100x lower than memory bandwidth, so operations that cross the spill threshold can jump from milliseconds to seconds.

In-memory hash join:   ~50 ms   (hash table lives entirely in RAM)
Spilling hash join:    ~4 s     (writes and reads temp files on disk)

This cliff is worth watching for in production: a query that ran fine when the table had 1 million rows may become painful when it grows to 10 million and pushes the working set past the memory budget.

Observing Spill in Query Plans

Most databases surface spill information in their execution plans. Look for these signals:

  • PostgreSQL EXPLAIN (ANALYZE, BUFFERS) — shows Batches: N on a hash join (more than 1 means spill) and Sort Method: external merge on sorts.
  • SQLite EXPLAIN QUERY PLAN — shows USE TEMP B-TREE when a sort or grouping operation needs temporary on-disk storage.
  • MySQL EXPLAIN — Using filesort and Using temporary in the Extra column signal on-disk temporary work.

Tuning Strategies

The goal is to keep operations in memory or at least minimize how much spills to disk.

1. Raise the memory budget per session (carefully). For a heavy one-off query, raising work_mem can eliminate spill entirely. Do it per-session to avoid starving other connections:

-- PostgreSQL: raise work_mem for this session only
SET work_mem = '64MB';

SELECT customer_id, SUM(amount) AS total
FROM orders
GROUP BY customer_id
ORDER BY total DESC;

RESET work_mem;

2. Add indexes to eliminate sorts. If an ORDER BY matches an index key, the engine reads rows in order directly from the index — no sort phase, no spill possible.

3. Filter and project early. The fewer rows and columns that reach a sort or join, the smaller the working set. Select only the columns you need and push WHERE conditions as close to the source table as possible.

4. Pre-aggregate or partition. For very large aggregations, splitting the work by a natural partition (e.g., by date range or by shard) keeps each sub-query within the memory budget.

Reveal: Why does doubling work_mem sometimes cut query time in half?
A sort that fits in twice the memory may go from two merge passes to one. Each pass reads and writes the entire dataset to disk, so eliminating one pass cuts I/O roughly in half. The relationship is not always that clean — it depends on how many initial sorted runs were produced — but large memory increases often produce disproportionately large speedups because they eliminate entire passes rather than just shrinking them.

Key Takeaways

  • Engines enforce per-operation memory budgets to share RAM fairly across concurrent queries.
  • When intermediate data exceeds the budget, the engine spills to disk — results remain correct, but I/O cost skyrockets.
  • Spill is a cost cliff, not a gradual slowdown; watch for it when data volumes grow past a threshold.
  • Query plans expose spill through keywords like external merge, Batches > 1, USE TEMP B-TREE, and Using filesort.
  • The main remedies are: raise the memory budget per session, add indexes to avoid sorting, filter early, and select only the columns you need.
