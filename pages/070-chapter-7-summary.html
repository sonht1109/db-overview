Part II. Querying and Language

Chapter 7. Indexes

Chapter 7 Summary: What We Learned

Page Topic Takeaway in One Sentence
061 Why scanning everything is slow Full table scans are O(n) — with millions of rows, that becomes seconds or minutes when applications need milliseconds.
062 What an index really is An index is a separate data structure that maps values to row locations, enabling O(log n) lookups.
063 B-tree indexes B-trees keep data sorted, support equality and range queries, and rebalance automatically — the workhorse of database indexing.
064 Hash indexes Hash indexes provide O(1) equality lookups but cannot do range scans — fast for exact-match queries only.
065 Composite indexes A multi-column index speeds up queries that filter on the leading columns in order — column order matters.
066 Covering indexes When an index contains all columns a query needs, the engine reads only the index (index-only scan), skipping the table entirely.
067 Unique indexes A unique index enforces uniqueness on the indexed columns; faster than a table-level unique constraint and serves double duty.
068 Secondary indexes Secondary indexes point back to the primary key (or row ID); they add write overhead but are essential for fast reads on non-key columns.
069 When indexes hurt Indexes slow down writes (INSERT/UPDATE/DELETE), consume storage, and can confuse the optimizer — measure, don't guess.
Reveal: How do these ideas connect to the next chapter?
Chapter 7 covered indexes for single-table access — when your query filters on one table, an index can turn a scan into a seek. But real queries rarely touch just one table. Chapter 8 tackles the operations that make databases powerful and expensive: joins, aggregation, and sorting. You will learn nested-loop, hash, and merge joins; how GROUP BY works under the hood; and why sorting spills to disk when memory runs out. Indexes help here too, but the optimizer's choices multiply when multiple tables are involved.
