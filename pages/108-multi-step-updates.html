Part III. Storage and Execution

Chapter 11. Transactions

Multi-step updates

Most real database work isn't a single INSERT or a lone SELECT — it's a sequence of related writes that must all succeed together to leave the database in a meaningful state. Transferring money, placing an order, or cancelling a reservation all involve touching multiple rows (sometimes across multiple tables) in a specific order. This is what makes multi-step updates both powerful and dangerous.

Why multiple steps are needed

A single SQL statement can change many rows at once, but it can only change one thing at a time — one table, one kind of change. Real operations cross those boundaries.

Consider an e-commerce order:

  1. Insert a new row into orders.
  2. Insert one row per product into order_items.
  3. Decrement the stock count for each product.
  4. Deduct the total from the customer's credit balance.

Each step reads work done by the step before it. If step 3 fails (e.g., a stock count would go negative and violate a constraint), leaving steps 1 and 2 already committed would corrupt the database — you'd have a "ghost" order with items but no reserved stock.

Transactions wrap the steps

The database's answer is the transaction: a named boundary inside which all steps are executed together. You open a transaction with BEGIN, do your work, and then either:

  • COMMIT — finalize all changes at once, making them visible to others.
  • ROLLBACK — undo every change made inside the transaction, as if it never happened.
BEGIN;

INSERT INTO orders (customer_id, total) VALUES (42, 149.99);

UPDATE inventory
SET stock = stock - 1
WHERE product_id = 7;

-- If anything above failed, we'd ROLLBACK here instead:
COMMIT;

The guarantee is atomicity: either both the INSERT and the UPDATE commit together, or neither does. There is no in-between state that other sessions can observe.

What can go wrong (and how the DB helps)

There are three main reasons a multi-step update might need to abort:

Cause Example Database response
Constraint violation Stock goes below 0 (CHECK constraint) Statement fails; transaction can be rolled back
Application error Code detects bad input after step 1 Application calls ROLLBACK explicitly
System failure Process crashes mid-transaction Engine uses its write-ahead log to undo uncommitted work on restart

In all three cases, the database's logging infrastructure — the write-ahead log (WAL) — records what changed and lets the engine reverse it cleanly. You don't have to write "undo" code yourself.

Note: In SQLite (and most databases), if you don't open an explicit BEGIN, each statement runs in its own auto-committed transaction. That's fine for isolated reads, but it means two consecutive UPDATE statements are not atomic with each other.
