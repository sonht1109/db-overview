Part VI. Engineering Practice

Chapter 26. Choosing a Database

Portability and lock-in

Every database you adopt is a commitment, but not all commitments are equally binding. Some databases are built on open standards and can be swapped with modest effort. Others expose proprietary APIs, query languages, and data formats that make migration expensive enough to be effectively permanent. Understanding the spectrum of lock-in before you adopt a system gives you the ability to make that trade-off consciously rather than discovering it when you want to leave.

What Lock-In Actually Looks Like

Lock-in is not a binary property — it exists on a spectrum, and it operates through several distinct mechanisms:

API and query language lock-in. A system that uses standard SQL (even with extensions) is easier to migrate from than one that uses a proprietary query language. Moving from PostgreSQL to MySQL is painful but tractable because both speak SQL. Moving from DynamoDB's API to a different key-value store requires rewriting every database call.

Data format lock-in. If your data is stored in a proprietary binary format that requires the vendor's tools to read, export becomes an active project rather than a background concern. Databases that support standard export formats (CSV, Parquet, SQL dump, JSON) give you an escape hatch.

Feature lock-in. Using a proprietary feature — DynamoDB Streams, Spanner's TrueTime-based external consistency, Snowflake's TIME_TRAVEL — adds value but creates migration cost. When the time comes to move, any feature without a direct equivalent in the target system requires application redesign.

Operational lock-in. Your team's expertise, your monitoring setup, and your deployment automation are all built around the current system. Even if the data is portable, the operational knowledge is not.

The Cost of Migration

The cost of migrating away from a database has two components:

  1. Data migration cost — exporting data from the source, transforming it to the target's format, validating completeness and correctness. For large datasets this can take weeks or months, with complex cutover logistics to avoid data loss during the transition.

  2. Application migration cost — rewriting every place in the application that talks to the database: connection setup, queries, transactions, error handling. The more proprietary the API, the larger this surface area.

A rough heuristic: migrating between two databases that both speak standard SQL costs roughly one to three months of engineering effort for a medium-sized application. Migrating from a proprietary key-value API to anything else is more like rewriting the data access layer from scratch.

Strategies to Reduce Lock-In

Use standard SQL where possible. Even when your database supports proprietary extensions, prefer standard SQL for core queries. This does not mean avoiding useful extensions — PostgreSQL's JSONB or window functions are worth using — but it means being deliberate about which proprietary features you rely on critically.

Wrap the database behind a thin abstraction layer. A repository or data-access layer that hides the database-specific client behind an interface means that changing the underlying database requires changing one layer, not every call site. This is more relevant for application code than for complex query logic.

// Low lock-in: query language hidden behind interface
interface UserRepository {
  findById(id: string): Promise<User>
  save(user: User): Promise<void>
}

// Higher lock-in: DynamoDB-specific code scattered across services
const result = await dynamoClient.query({
  TableName: 'users',
  KeyConditionExpression: 'pk = :pk',
  ExpressionAttributeValues: { ':pk': { S: userId } }
}).promise()

Plan data exports regularly. Test your ability to export the full dataset in a portable format (SQL dump, Parquet, JSON lines) on a schedule. This serves two purposes: it validates your backup, and it proves that the migration path is open.

Separate the system-of-record from the specialised store. If your canonical data lives in a portable relational database and a specialised store (search, cache, analytics) holds derived copies, migrating the specialised store is much simpler — you can rebuild it from the primary.

When Lock-In Is Acceptable

Lock-in is not always bad. If the specialised capabilities of a system provide significant competitive or operational advantage, and the cost of migration is a known, acceptable risk, locking in may be the right choice. DynamoDB's managed infrastructure, multi-region active-active replication, and predictable pricing are genuinely valuable — many companies accept the API lock-in in exchange for that operational simplicity.

The mistake is not accepting lock-in; the mistake is accepting it unknowingly, without having evaluated the migration cost.

Key Takeaways

  • Lock-in operates through four mechanisms: query language, data format, proprietary features, and operational knowledge.
  • Migration costs have two components: data migration and application rewrite — both scale with lock-in level.
  • Reduce lock-in by preferring standard SQL, wrapping databases in thin abstraction layers, and validating data exports regularly.
  • Lock-in can be an acceptable trade-off if the capabilities justify it — the mistake is accepting it unknowingly.
