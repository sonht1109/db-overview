Part VI. Engineering Practice

Chapter 26. Choosing a Database

Chapter 26 Summary: What We Learned

Page Topic Takeaway in One Sentence
251 Start from workload, not fashion Choose a database because it matches your read/write patterns, consistency needs, and scale — not because it is trending on Hacker News.
252 Questions to ask before choosing What are my query patterns? What consistency do I need? What is my data shape? How fast will it grow? Who will operate it?
253 One database or many One database is simpler to operate; multiple databases let you use the best tool for each job. Start with one, add more when justified.
254 Build vs buy Building a custom storage layer is almost never the right call — databases are decades of engineering. Use managed services or open-source.
255 Managed vs self-hosted Managed: less operational burden, higher cost, limited control. Self-hosted: full control, lower cost at scale, significant operational expertise needed.
256 Cost, team, and operations License cost is the smallest factor. Consider: team expertise, operational overhead, migration cost, and long-term maintenance.
257 Portability and lock-in Proprietary features save time now but make migration harder later. Standard SQL and portable data formats reduce lock-in.
258 When to switch systems Switch when the current system cannot meet your scale, latency, or feature needs — and the cost of migration is less than the cost of staying.
259 A practical decision framework 1. Define workload. 2. List constraints. 3. Shortlist databases. 4. Prototype with real data. 5. Test failure modes. 6. Decide.
Reveal: How do these ideas connect to the next chapter?
Chapter 26 taught you to choose a database. But choosing is different from understanding. Chapter 27 opens Part VII — Building Understanding — by doing something radical: building a tiny database from scratch. You will implement storage, an index, a parser, a planner, a WAL, and basic locking. Every concept from the previous 26 chapters — B-trees, write-ahead logs, query optimization, concurrency control — becomes concrete when you build it yourself. This is not about writing production code; it is about never again being mystified by what a database does.
