Chapter 5. A Gentle Introduction to SQL

Joining tables

Real databases spread information across multiple tables. A customers table holds who people are; an orders table records what they bought. To answer "which customers placed orders over $200?" you need rows from both tables at once — that is exactly what a join does. Joins are the mechanism SQL uses to reassemble related data that has been stored separately.

Why Data Lives in Multiple Tables

If you stored a customer's name and address in every order row, you would repeat the same data thousands of times and risk it going out of sync. Splitting the data — customers in one table, orders in another, linked by a shared key — eliminates that redundancy. The tradeoff is that you need to join the tables whenever you want the full picture.

This is the payoff for the normalization you read about in Chapter 2: data is stored once, and queries assemble it on demand.

INNER JOIN: Matching Rows Only

The most common join returns only rows that have a match in both tables. The syntax names the second table and the condition that connects them:

SELECT customers.name, orders.amount, orders.product
FROM customers
INNER JOIN orders ON customers.id = orders.customer_id;

The ON clause is the join condition — it tells the engine which column pairs correspond to each other. Rows with no partner are silently dropped from the result.

Note: JOIN without a qualifier is INNER JOIN in SQLite and standard SQL — both spellings are equivalent.

Column Aliases Keep Queries Readable

When joining multiple tables, columns can share names. Qualify them with table.column, or use table aliases to keep things concise:

SELECT c.name, o.amount, o.product
FROM customers AS c
INNER JOIN orders AS o ON c.id = o.customer_id;

LEFT JOIN: Keep All Rows from the Left Table

Sometimes you want every row from the first (left) table, even if there is no matching row in the second. A LEFT JOIN (also written LEFT OUTER JOIN) fills in NULL for missing right-side columns:

SELECT c.name, o.amount
FROM customers AS c
LEFT JOIN orders AS o ON c.id = o.customer_id;

A customer who has never ordered will still appear — with NULL in the amount column. This is useful for "find all customers who have not placed an order":

SELECT c.name
FROM customers AS c
LEFT JOIN orders AS o ON c.id = o.customer_id
WHERE o.id IS NULL;

Join Types at a Glance

Join type What it returns
INNER JOIN Only rows with a match in both tables
LEFT JOIN All rows from the left table; NULLs for unmatched right rows
RIGHT JOIN All rows from the right table; NULLs for unmatched left rows
FULL OUTER JOIN All rows from both tables; NULLs where no match exists
CROSS JOIN Every combination of rows (Cartesian product) — use with care

SQLite supports INNER, LEFT, and CROSS joins natively. RIGHT and FULL OUTER are available in PostgreSQL, MySQL 8+, and SQL Server, but you can always rewrite them as a LEFT JOIN with the tables swapped.

A Note on Performance

Every join asks the engine to match rows across tables. On small datasets this is instant. On large tables, a missing index on the join column (customer_id in our example) can make a query scan millions of rows unnecessarily. Indexing foreign-key columns is one of the first things to do when a joined query runs slowly — a topic covered in depth in Part IV.
