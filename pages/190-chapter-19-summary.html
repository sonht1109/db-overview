Chapter 19. Document Databases

Chapter 19 Summary: What We Learned

Chapter Contents

Page Topic Takeaway in One Sentence
181 JSON as a data model Documents are self-describing JSON/BSON objects — the structure is part of the data, not declared separately in a schema.
182 Flexible schema Different documents in the same collection can have different fields — great for rapid iteration, dangerous for long-term consistency.
183 Nested data and arrays Embedding related data inside a single document avoids joins — load a user with all their orders in one read.
184 Secondary indexes on documents Document databases support indexes on nested fields and arrays, but index maintenance on deeply nested data is expensive.
185 Querying document fields Query languages (MQL, N1QL) let you filter on arbitrary fields, but multi-collection joins are limited compared to SQL.
186 Document updates and rewrites Partial updates to nested fields are possible but may trigger full document rewrites — large documents amplify this cost.
187 Embedding vs referencing Embed data you always read together; reference data shared across documents — the same normalization tradeoff in document form.
188 Strengths and weaknesses Strengths: developer velocity, flexible schema, natural fit for JSON APIs. Weaknesses: weak joins, eventual consistency, limited aggregation.
189 Good use cases Business intelligence dashboards, periodic reports, machine learning feature engineering, log analysis, financial reconciliation.
