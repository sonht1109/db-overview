Chapter 12. Concurrency Control

Choosing isolation levels

Earlier chapters introduced the four standard isolation levels — Read Uncommitted, Read Committed, Repeatable Read, and Serializable — and showed the anomalies each one prevents. Now the practical question: which level should you actually use, and when? The answer is always a tradeoff between correctness guarantees and throughput.

The Core Tradeoff

Higher isolation means the database does more work to shield your transaction from concurrent activity — more locking, more version tracking, or outright serialization of conflicting work. That extra work costs performance and can reduce how many transactions complete per second.

Isolation Level Prevents Typical Overhead Use it when…
Read Uncommitted nothing lowest almost never — only for raw diagnostics
Read Committed dirty reads low most OLTP reads; fine when re-reads are rare
Repeatable Read dirty + non-repeatable reads moderate reports, audits, multi-read transactions
Serializable all anomalies highest financial transactions, inventory, anywhere correctness beats throughput

Note: Most databases default to Read Committed (PostgreSQL, SQL Server, Oracle). MySQL/InnoDB defaults to Repeatable Read. SQLite uses Serializable by default because it serializes writers entirely.

Matching the Level to the Workload

Simple read-heavy apps (dashboards, feeds)

A dashboard that reads a few rows to render a page almost never needs Serializable. If the page shows slightly stale data because another transaction committed between two queries, users won't notice. Read Committed is the right choice here — it prevents corrupt (uncommitted) data from leaking into your view while keeping concurrency high.

Multi-step reads in one transaction (reports, summaries)

Suppose you generate an invoice that reads orders, order_items, and products in sequence. If any of those tables change between reads, your totals can disagree with each other. That is a non-repeatable read, and it matters here. Use Repeatable Read (or Snapshot Isolation, which most databases implement under that name) so every read within the transaction sees the same version of the data.

Writes that depend on what you just read (check-then-act)

The pattern "read a value, decide based on it, write a new value" is where most bugs hide. Example: deducting from an account balance, booking the last seat on a flight, or assigning a unique order number. If two transactions both read the same value and then both write, you get a lost update or phantom insert.

Serializable is the only level that blocks all of these. Under Serializable, the database either detects the conflict and aborts one transaction, or prevents the conflict from happening in the first place.

-- Flight booking: check-then-act pattern
BEGIN;
  SELECT seats_remaining FROM flights WHERE id = 42;
  -- Without Serializable, two sessions can both see seats_remaining = 1
  -- and both proceed to book, overselling the flight.
  UPDATE flights SET seats_remaining = seats_remaining - 1 WHERE id = 42;
COMMIT;

Under Serializable (or with explicit SELECT ... FOR UPDATE at lower levels), the second transaction will block or fail rather than oversell.

Practical Notes

You can mix levels within an application. A background analytics query can run at Read Committed while a payment handler runs at Serializable. Most database drivers let you set the level per connection or per transaction with SET TRANSACTION ISOLATION LEVEL ....

Serializable is not always slow. Databases like PostgreSQL implement Serializable Snapshot Isolation (SSI), which detects conflicts optimistically rather than holding long locks. Throughput under SSI is often close to Snapshot Isolation unless conflict rates are high.

SELECT ... FOR UPDATE is a practical middle ground. At Read Committed, you can lock specific rows you intend to modify, preventing concurrent updates without paying the full cost of Serializable for the whole transaction.
