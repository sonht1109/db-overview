Part VII. Building Understanding

Chapter 27. A Tiny Database from Scratch

Add a simple index

The full table scan works, but it reads every row on disk for every query. Once your table grows past a few thousand rows, that becomes intolerable. The classic solution is an index: a separate data structure that maps a column's values to the row numbers where they live, letting the engine skip straight to the relevant rows without touching the rest of the file.

What an Index Is

An index is a sorted or hashed auxiliary structure maintained alongside the main table. Our toy engine uses the simplest possible design: a Python dict (hash map) that maps a column value to a list of row numbers.

# The index lives entirely in memory for now.
# Key   = column value (e.g. a salary integer or a name string)
# Value = list of row numbers that have that value
Index = dict[int | str, list[int]]

When we insert a row we update the index in the same call. When we query, we look up the value in the index and read only those row numbers — skipping every other row on disk.

def insert_with_index(
    table: TableFile,
    idx: Index,
    indexed_col: str,
    row: dict,
) -> int:
    row_num = table.write_row(row)            # append to data file
    key = row[indexed_col]
    idx.setdefault(key, []).append(row_num)   # update index
    return row_num

def lookup(
    table: TableFile,
    idx: Index,
    value,
) -> list[dict]:
    row_nums = idx.get(value, [])
    return [table.read_row(n) for n in row_nums]  # only these rows

Build vs. Maintain

There are two ways an index comes into existence:

  1. Build-time: scan the whole table once and populate the index. Expensive upfront, O(n).
  2. Write-time maintenance: update the index on every insert (and later update/delete). Adds latency to writes; keeps the index current.
def build_index(table: TableFile, col: str) -> Index:
    """One-time full scan to create an index on `col`."""
    idx: Index = {}
    for i in range(table.num_rows()):
        row = table.read_row(i)
        key = row[col]
        idx.setdefault(key, []).append(i)
    return idx

Real databases use the build approach for CREATE INDEX on an existing table, then switch to write-time maintenance for all subsequent DML.

Equality Lookups vs. Range Queries

A dict-based hash index handles equality queries (name = 'Bob') in O(1). It cannot handle range queries (salary BETWEEN 60000 AND 90000) efficiently — you would have to iterate every key in the dict and filter.

For ranges, a sorted structure like a B-tree or sorted array is needed:

import bisect

# Sorted index: list of (value, row_num) pairs, kept in sorted order
SortedIndex = list[tuple[int, int]]

def range_lookup(sidx: SortedIndex, lo, hi) -> list[int]:
    """Return row numbers where value is in [lo, hi]."""
    left  = bisect.bisect_left(sidx,  (lo, -1))
    right = bisect.bisect_right(sidx, (hi, float("inf")))
    return [row_num for _, row_num in sidx[left:right]]

This is essentially what a B-tree leaf-level scan does: find the lower bound, then walk forward until the upper bound.

Index type Best for Not good for
Hash (dict) Exact equality Range queries, ordering
Sorted list / B-tree Range, ORDER BY High write throughput
Bitmap Low-cardinality cols High-cardinality cols

The Write Cost of Indexes

Every index you add makes reads faster and writes slower. Our insert_with_index does two operations: one file append (data) and one dict update (index). Real databases may maintain a dozen indexes per table, turning a single INSERT into a dozen B-tree insertions.

Note: This is why bulk-loading a table is often faster with indexes disabled, then rebuilt afterwards — the build is more cache-friendly than thousands of individual B-tree insertions scattered across pages.

Key Takeaways

  • An index trades write overhead for read speed: every insert must update the index, but queries skip most rows.
  • A hash index (dict) gives O(1) equality lookups; it cannot serve range queries.
  • A sorted/B-tree index supports both equality and range queries at O(log n) cost.
  • Indexes must be kept consistent with the table — a bug that updates the table but not the index produces stale or phantom reads, one of the hardest bugs to diagnose in a home-grown engine.
