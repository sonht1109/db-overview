Part IV. Distributed Systems

Chapter 17. Consistency in Distributed Databases

Causal consistency

Imagine you post a message in a group chat, then immediately reply to it. A friend who joins a moment later sees your reply — but not the original message. From their view, your reply is a non-sequitur. This is exactly the kind of anomaly causal consistency is designed to prevent.

Causal consistency is a consistency model that sits between two extremes: it is weaker than linearizability (no global wall-clock ordering required), but stronger than eventual consistency (causally related operations must appear in order). The key insight is that not all operations are related — causal consistency only constrains operations that are causally linked.

What "Causally Related" Means

Two operations are causally related if one could have influenced the other:

  • Write → Read: If process A writes a value and process B reads it, B's subsequent writes depend on that value.
  • Read → Write: If B reads a value and then writes based on what it saw, that write is causally downstream.
  • Transitivity: If A causes B, and B causes C, then A causes C — even if A and C happen on different nodes.

Operations with no path between them are concurrent and causally independent. A causally consistent system may deliver concurrent operations in any order; it only enforces ordering within causal chains.

Note: Databases track causality using vector clocks or version vectors — per-node counters that let every node determine whether one event "happened before" another without relying on synchronized wall clocks.

A Concrete Example

Consider three users — Alice, Bob, and Carol — sharing a distributed document store with two replicas.

Under causal consistency, Carol is guaranteed to see "Hello!" before "Hi Alice!" — because Bob's reply causally depends on Alice's message. If Carol happened to be on a replica that received Bob's write first, the replica must delay delivering it until Alice's message has also arrived.

Without causal consistency (e.g., under plain eventual consistency), Carol could see "Hi Alice!" arrive first — a confusing and broken experience.

Causal Consistency vs. Other Models

Model Global order? Causal order? Concurrent ops order? Availability
Linearizability Yes (wall clock) Yes Yes Lower (quorum needed)
Sequential consistency Yes (logical) Yes Yes Lower
Causal consistency No Yes No (any order) Higher
Eventual consistency No No No (any order) Highest

Causal consistency is the strongest model that remains available under network partitions, according to the PACELC framework. This makes it attractive for geo-distributed systems where you cannot afford the latency of a global quorum.

How Databases Implement It

Real systems track causal dependencies using version vectors (also called dependency vectors). Each client carries a vector summarizing the set of writes it has already observed. When the client issues a new operation, it sends this vector along. The server only executes the operation after it has applied all causally prior writes.

Client vector at Carol's read:
  { replica_A: 5, replica_B: 3 }

This means Carol has seen all writes up to sequence 5 on replica A
and up to sequence 3 on replica B. The server will not show her
a write with replica_A: 6 until she has also seen replica_A: 5.

Notable systems that offer causal consistency:

  • MongoDB (since v3.6) — causally consistent sessions via afterClusterTime
  • DynamoDB — optional strongly consistent reads, but causal sessions via client-side tokens
  • AntidoteDB — built from the ground up around causal+ consistency
  • CockroachDB — uses hybrid logical clocks (HLC) to approximate causal ordering

The Trade-off Worth Knowing

Causal consistency is not free. Version vectors grow with the number of nodes and clients; tracking fine-grained causal histories at scale is expensive. Many systems therefore implement causal+ (causal consistency plus convergent conflict resolution) rather than perfect causal consistency, accepting a small window where concurrent writes must be merged by the application.

The practical question to ask is: "Does the correctness of this operation depend on another operation the user just performed?" If yes, causal consistency — or stronger — is warranted. If operations are genuinely independent (e.g., two users updating unrelated records), eventual consistency may be enough and will give you better write throughput.

Reveal: Would causal consistency prevent a "lost update" where two users increment the same counter concurrently?

No. Causal consistency only orders operations that are causally related — one must have observed the other. If two users each read the counter value (say, 10) and both write 11 without seeing each other's write, those operations are concurrent and causally independent. Causal consistency allows both writes and the system converges to 11, losing one increment. Preventing this requires a stronger model (e.g., serializable transactions) or a CRDT like a grow-only counter that merges concurrent increments correctly.
