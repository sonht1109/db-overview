Chapter 15. Replication

Replication logs

Every replication system needs a way to describe — precisely and durably — what changed on the leader so followers can apply the same changes in the same order. That description is the replication log: an ordered, append-only record of mutations that drives the entire follower pipeline. You have already seen the term appear in earlier sections; this page digs into what replication logs actually look like and why their design matters.

What Goes Into a Replication Log

A replication log entry must capture enough information for a follower to reproduce the exact effect of a write. Different databases take different approaches, and the trade-offs are significant.

Statement-based replication

The simplest idea: log the original SQL statement.

-- What the leader logs
UPDATE orders SET status = 'shipped' WHERE id = 7;

A follower re-executes the statement verbatim. This is compact and human-readable, but breaks on non-deterministic functions like NOW(), RAND(), or UUID() — each node evaluates them independently and gets different results.

Write-ahead log (WAL) shipping

Most production databases already maintain a write-ahead log for crash recovery: every page modification is written to the WAL before the page itself is updated. Replication can simply ship those same WAL segments to followers.

  • PostgreSQL streaming replication works this way: followers connect to the leader and receive a continuous stream of WAL records.
  • The WAL is a physical log — it records which bytes changed on which disk page. Followers must run the identical database version (and often the same minor version) because WAL format is tightly coupled to the storage engine internals.

Row-based (logical) replication

A row-based log records the before and after values of each row that was touched — independent of the SQL that caused the change.

-- Logical event for: UPDATE orders SET status='shipped' WHERE id=7;
{
  "table":  "orders",
  "op":     "UPDATE",
  "before": { "id": 7, "status": "pending" },
  "after":  { "id": 7, "status": "shipped" }
}

This approach is deterministic (no function re-evaluation), version-agnostic, and can replicate to different database engines — which is why MySQL's binlog in row mode and PostgreSQL's logical replication (via the pgoutput plugin) are popular for cross-version upgrades and heterogeneous pipelines (e.g., syncing to a data warehouse).

The cost is volume: a single UPDATE touching a million rows produces a million log events.

Trigger-based replication

Older or exotic setups use application-level triggers that fire on every INSERT/UPDATE/DELETE and write a record to a change table. A separate process reads that table and forwards changes. Flexible but slow, and adds overhead on every write.

Sequence Numbers and Log Position

Every entry in a replication log gets a monotonically increasing position identifier. In PostgreSQL this is the Log Sequence Number (LSN); in MySQL it is a binary log (binlog) position or GTID (Global Transaction ID).

These positions are critical for three reasons:

Use Why position matters
Follower catch-up A follower tells the leader "I'm at LSN 10480 — send me everything after that."
Exactly-once delivery After a follower restart, it resumes from its last confirmed position, never re-applying events it already applied.
Failover safety During promotion, the new leader's position tells the system which writes were durably replicated and which were not.

Note: GTIDs (MySQL 5.6+) encode a unique server UUID with each transaction, making it safe to re-point a replica at a different leader without worrying about log position gaps — the replica simply skips transactions it has already applied regardless of where they originated.

Change Data Capture

Replication logs don't have to stay inside the database cluster. Change Data Capture (CDC) taps the log and publishes changes to external consumers — message queues, search indexes, caches, or data warehouses — in real time.

leader WAL / binlog
       │
       ▼
  CDC connector           (e.g. Debezium, Maxwell, pglogical)
       │
   ┌───┴────────────────────────┐
   ▼                            ▼
Kafka topic              Elasticsearch index
(event stream)           (search replica)

Tools like Debezium (open source) sit on top of the database's logical replication protocol and emit row-level events to Kafka. Downstream services subscribe to the topic and maintain their own derived state. Because the log is ordered and durable, any service can rebuild its state from scratch by replaying from the beginning.

Note: CDC is one of the most powerful patterns in distributed data architectures. It decouples producers from consumers without requiring the application to explicitly write to multiple stores.
