Part V. Database Families

Chapter 22. Analytical Databases and Warehouses

Batch loading

Row-by-row INSERT statements are the natural way to write data in application code — one record arrives, one record goes in. For analytical databases that need to absorb millions or billions of rows on a schedule, that approach collapses under its own weight. Batch loading — ingesting large volumes of data in a single coordinated operation — is the dominant pattern for feeding data warehouses, and understanding why it is so much faster reveals a lot about how databases work internally.

Why Not Just INSERT?

A standard INSERT statement triggers a chain of work for every single row:

  1. Parse and plan the SQL statement
  2. Acquire row-level locks
  3. Write a log record (WAL entry) for the new row
  4. Locate a page with free space in the heap
  5. Write the row into that page
  6. Update all affected indexes
  7. Commit (fsync the log)

Repeat this ten million times and the overhead dwarfs the actual data transfer. Parsing alone can consume more CPU than the data write.

Bulk load bypasses most of this. A COPY command or bulk loader reads a prepared file, sorts it to match index order, writes data pages directly, and builds indexes in one pass at the end — skipping per-row logging and lock acquisition entirely.

Method Rows/second (typical) Notes
INSERT (single row) ~1 000 Full transaction overhead per row
INSERT (multi-row VALUES) ~50 000 Amortizes parse cost
COPY / bulk load ~500 000–5 000 000 Bypasses row-by-row logging
Parallel bulk load 5 000 000+ Multiple files loaded concurrently

Numbers vary wildly by hardware and table complexity, but the order-of-magnitude difference is consistent.

ETL vs ELT

Two dominant patterns exist for getting data from source systems into a warehouse:

ETL — Extract, Transform, Load

Data is extracted from source systems, transformed in a separate processing layer (cleaning, joining, reshaping), and then loaded into the warehouse in its final form. The transformation happens outside the warehouse.

ELT — Extract, Load, Transform

Raw data is loaded into the warehouse first (into staging tables), and the transformation runs inside the warehouse using SQL. Modern columnar warehouses are so fast at SQL that this is often cheaper than building a separate processing cluster.

Trend: ELT has grown in popularity with cloud warehouses (BigQuery, Snowflake, Redshift) because warehouse compute is elastic and cheap, while maintaining a separate Spark or Flink cluster adds operational complexity.

The ETL Pipeline

COPY and Bulk Load APIs

Most databases ship a dedicated bulk-load mechanism:

Database Command / API
PostgreSQL COPY table FROM '/path/file.csv' CSV
MySQL LOAD DATA INFILE '/path/file.csv'
BigQuery bq load --source_format=CSV ...
Snowflake COPY INTO table FROM @stage/file.csv
Redshift COPY table FROM 's3://bucket/prefix' IAM_ROLE '...'

These commands share a common design: they read a prepared file (CSV, Parquet, ORC, JSON), validate or cast data types, and write directly to data pages — often in parallel across multiple CPU cores or nodes.

Staging Tables

A common pattern is the staging table — a temporary table that mirrors the target schema (or the raw source format) and receives the bulk load before transformation:

-- Step 1: bulk load raw data into staging
COPY raw_sales_staging
FROM 's3://pipeline/2024-06-25/sales.csv'
IAM_ROLE 'arn:aws:iam::123456789:role/redshift-s3-role'
FORMAT CSV IGNOREHEADER 1;

-- Step 2: transform and upsert into production table
INSERT INTO fact_sales (date_key, product_key, store_key, sale_amount, quantity)
SELECT
  d.date_key,
  p.product_key,
  s.store_key,
  stg.amount,
  stg.qty
FROM raw_sales_staging stg
JOIN dim_date    d ON stg.sale_date = d.full_date
JOIN dim_product p ON stg.sku       = p.sku
JOIN dim_store   s ON stg.store_id  = s.store_id;

-- Step 3: truncate staging for next run
TRUNCATE raw_sales_staging;

Staging tables also serve as a safety net: if the transformation fails, the raw data is still there and can be reprocessed.

Why Bulk Load Is Faster: The Mechanics

1. No Per-Row Write-Ahead Logging

Standard INSERTs write a WAL record for every row before touching the data page. Bulk loaders bypass this — they write the data pages first, then optionally write a single summary log entry for the entire operation. On crash recovery, the entire load is either rolled back or redone as one unit.

2. Sorted File Ingestion

When data arrives pre-sorted by the clustering key (for columnar stores) or the B-tree key (for indexes), the loader can write pages sequentially without random seeks. This is why preparing sorted Parquet files before loading into Redshift or BigQuery dramatically improves performance.

3. Deferred Index Building

Indexes are disabled (or accumulate changes in a buffer) during the load, then built in one pass at the end. Building a B-tree index from scratch on sorted data is an O(n) scan; updating it row by row during insert is O(n log n) with heavy random I/O.

4. Compression-Aware Writes

Columnar warehouses encode each column using the best compression algorithm (RLE, dictionary, delta) for that column's value distribution. Bulk loaders analyze the full column batch and choose the optimal encoding; per-row inserts cannot do this because they see one value at a time.

Micro-batching and Streaming Ingestion

Pure batch loading means data is available only after each batch completes — potentially hours of lag. Several hybrid approaches reduce latency:

  • Micro-batching: Run the pipeline every 5–15 minutes instead of nightly. Still uses bulk load mechanics, just more frequently.
  • Streaming insert APIs: Snowflake, BigQuery, and Redshift all offer streaming insert APIs that accept individual rows or small payloads with sub-second latency, but at lower throughput and higher cost than bulk load.
  • Change Data Capture (CDC): Tail the source database's transaction log (binlog, WAL) and stream only changed rows into a message queue (Kafka), then batch-load from the queue into the warehouse.

Most production pipelines combine approaches: CDC for low-latency data feeds, nightly bulk loads for historical backfill, and micro-batching for middle-tier latency requirements.

Data Freshness Trade-off

Batch loading is fundamentally a freshness vs. throughput trade-off:

Strategy Freshness Throughput Complexity
Nightly batch Hours old Very high Low
Hourly micro-batch ~1 hour old High Medium
Streaming insert Seconds old Medium High
Real-time CDC Near real-time Medium Very high

For most business intelligence use cases — dashboards refreshed daily, weekly reports — an hourly or nightly batch is entirely adequate and far simpler to operate.

Key Takeaways

  • Batch loading is orders of magnitude faster than row-by-row INSERT because it bypasses per-row logging, defers index building, and writes sorted pages directly.
  • ETL transforms data outside the warehouse; ELT loads raw data first and transforms inside the warehouse using SQL — ELT is increasingly preferred with cloud warehouses.
  • The COPY command (or equivalent bulk load API) is the right tool for loading large datasets; it is not just syntactic sugar over INSERT.
  • Staging tables receive the raw bulk load, then a SQL transformation pipeline moves clean data into the final schema.
  • Data freshness is always a trade-off: nightly batches are simple and fast to operate; real-time streaming is complex and expensive.
  • Micro-batching and CDC offer middle-ground strategies when hourly-or-better latency matters.
