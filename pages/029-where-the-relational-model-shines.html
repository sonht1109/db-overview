Chapter 3. The Relational Model

Where the relational model shines

The relational model is not the only way to organize data — earlier chapters introduced documents, graphs, and key-value stores. So why does the relational model remain the dominant choice decades after its invention? Because a handful of its properties work together to solve problems that appear in almost every real application.

Flexible Querying Without Knowing Queries in Advance

When Edgar Codd proposed the relational model in 1970, one of his central goals was data independence: the ability to ask new questions of existing data without redesigning how it is stored. A spreadsheet or a hand-coded file format forces you to decide up-front what you want to retrieve. A relational database does not.

Because data is stored in normalized tables with declared relationships between them, SQL lets you combine, filter, and aggregate that data in ways that were never anticipated when the schema was designed. A table of orders created for billing can later be queried for customer retention analysis, geographic heat maps, or supply-chain forecasting — without touching the storage layout.

Note: This is sometimes called the "closed-world" benefit: every fact is stored once, and any question answerable from those facts can be expressed as a query.

Referential Integrity Keeps Data Consistent

Relational databases enforce relationships between tables at the database level, not just in application code. A foreign key constraint guarantees that a value referencing another table's primary key must actually exist there.

Consider an orders table that references a customers table. Without a constraint, nothing stops application code from inserting an order for customer ID 9999 if that customer was never created — or from deleting a customer while their orders remain. Foreign keys prevent both:

Transactions Protect Multi-Step Operations

Real work rarely involves a single row. Transferring money between bank accounts, placing an order that decrements inventory, registering a user while creating their profile — all of these require multiple writes that must either all succeed or all be rolled back together. Relational databases have supported ACID transactions since their earliest implementations:

Property What it guarantees
Atomicity All writes in a transaction succeed, or none of them do
Consistency The database moves from one valid state to another — constraints are never violated mid-transaction
Isolation Concurrent transactions do not see each other's partial work
Durability Once committed, writes survive crashes and power loss

This is not glamorous, but it is the reason you can trust that your bank balance is correct after a network hiccup.

Declarative SQL Hides Physical Complexity

In most programming, you specify how to do something step by step. SQL lets you specify what you want, and the query optimizer decides how to retrieve it efficiently. Whether a table has 100 rows or 100 million, the same SQL statement expresses the same intent — the database engine chooses index access paths, join strategies, and parallelism on your behalf.

This separation means:

  • Application code does not need to know about indexes or storage layouts.
  • Performance can often be improved just by adding an index or updating statistics, with zero code changes.
  • A query that runs slowly today may run faster next month after an engine upgrade, again with no code changes.
Reveal: When might the relational model be the wrong choice?
The relational model is a poor fit when your data is highly variable in shape (e.g., each record has different attributes), when you need to traverse deeply nested relationships efficiently (graph databases handle this better), or when you need to scale writes horizontally across many machines with very low latency (some key-value and document stores make this easier). The relational model also assumes your schema is reasonably stable — frequent, sweeping structural changes are expensive. Knowing where it shines makes it easier to recognize when something else is a better tool.
