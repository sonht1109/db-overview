Chapter 11. Transactions

Consistency in practice

The C in ACID stands for Consistency, but it is the most misunderstood of the four properties. Unlike Atomicity, Isolation, and Durability — which the database engine enforces mechanically — Consistency is partly your responsibility. The engine provides tools; you decide what "correct" means for your data.

What Consistency Actually Means

Consistency says: a transaction takes the database from one valid state to another valid state. "Valid" is defined by the constraints, rules, and invariants you have declared.

Two sources contribute to those rules:

Source Examples Who enforces it
Database-level constraints NOT NULL, UNIQUE, FOREIGN KEY, CHECK The engine — automatically
Application-level invariants "total debits must equal total credits", "a user's age must be positive" Your code — inside the transaction

The engine can only enforce rules it knows about. Any invariant you leave implicit lives entirely in application code. If two code paths update the same data, both must respect the invariant — or you end up in an inconsistent state even though every individual query "succeeded."

Note: Consistency in ACID is intentionally vague compared to the other three letters. It is a design goal, not a specific mechanism. The mechanisms that support it are constraints, transactions, and careful schema design.

Constraints as a Safety Net

Database constraints are the most reliable consistency tool because the engine checks them atomically — either the entire transaction satisfies every constraint, or it is rolled back.

Common constraint types and what they guard:

  • NOT NULL — a required field is never missing.
  • UNIQUE — no duplicate values in a column (or group of columns).
  • PRIMARY KEY — implies NOT NULL + UNIQUE; identifies each row.
  • FOREIGN KEY — a value in one table must exist in another; prevents orphaned rows.
  • CHECK — arbitrary condition on one or more columns (quantity > 0, end_date >= start_date).

Application-Level Invariants

Some rules cannot be expressed as a single-column constraint. Consider a double-entry bookkeeping system where every transfer must debit one account and credit another by the same amount. The database cannot enforce this with a CHECK constraint on a single row — it spans multiple rows.

The correct approach is to perform both writes inside a single transaction. If either write fails, the whole transaction rolls back, and the invariant holds.

What the Engine Cannot Do For You

Even with perfect schema design, some consistency problems fall outside what the database can catch automatically:

  • Stale reads in application logic — your code reads a value, makes a decision, then writes. If another transaction changed the value between the read and the write, your decision was based on outdated data. The fix is locking (SELECT ... FOR UPDATE) or optimistic concurrency checks — covered in the next page on isolation.
  • Business rules encoded only in code — if "a premium customer may have at most 10 open orders" lives only in an API handler, a direct SQL connection bypasses it entirely.
  • Data imported from outside — bulk loads often bypass triggers and may skip constraint checks if the import tool disables them for performance.

The practical lesson: treat database constraints as a last line of defense, not the only line. Validate at the application layer first, then let the database confirm. If they disagree, the database wins — a constraint violation is a clear signal that your application logic has a gap.

Note: Some engines (PostgreSQL, Oracle, SQL Server) support deferrable constraints — foreign key and uniqueness checks can be postponed until COMMIT rather than checked after each statement. This is useful when inserting rows that temporarily violate a constraint mid-transaction (such as inserting a parent and child row in reverse order).
