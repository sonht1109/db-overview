Part IV. Distributed Systems

Chapter 14. From One Machine to Many

Shared-nothing systems

The previous page introduced replication and sharding as two techniques for spreading data across machines. Shared-nothing architecture is the design philosophy that unifies them — and it's the reason modern distributed databases can scale to thousands of nodes without a single point of contention.

What "Shared Nothing" Means

In a shared-nothing system, every node is completely independent. Each node has:

  • Its own CPU and memory — no shared RAM bus.
  • Its own local disk — no shared storage array or SAN.
  • A network connection to other nodes — the only communication channel.

No node can directly read another node's memory or disk. To get data from a peer, a node must send a network message and wait for a reply.

This stands in contrast to two older alternatives:

Architecture What is shared Bottleneck
Shared memory RAM across CPUs Memory bus; OS lock contention
Shared disk Storage (e.g., SAN/NAS) I/O bandwidth; distributed locking
Shared nothing Nothing (network only) Network latency; cross-node coordination

The key insight: networks scale horizontally and cheaply. Adding a node adds independent capacity rather than competing for a fixed shared resource.

Why It Scales — and Where It Hurts

Shared-nothing's big win is linear scalability for workloads that map cleanly onto shards. If each node handles 10 000 writes/second and you need 50 000, add four more nodes. No shared bottleneck limits you.

But the architecture moves complexity into the application or middleware layer:

Cross-node queries are expensive. A JOIN between two tables that live on different nodes requires one node to pull data across the network. Some systems push part of the computation down to the remote node (a "push-down"); others ship entire result sets and join in memory. Either way, it costs network round trips.

Distributed transactions are hard. ACID guarantees across two nodes require a coordination protocol — usually two-phase commit (2PC). If any node fails mid-transaction, the others may block waiting for a decision. Many shared-nothing systems weaken isolation guarantees to avoid this cost, offering eventual consistency instead.

Rebalancing is painful. When you add a node, some data must migrate from existing nodes to the new one. During migration the system must continue serving queries, which means careful bookkeeping about which version of a row is authoritative.

Note: Systems like Google Spanner and CockroachDB have engineered around some of these pain points using Paxos-based consensus and globally synchronized clocks — but they don't eliminate the fundamental costs of coordination; they just manage them more gracefully.

Shared Nothing in Practice

Most databases you'll encounter in production are at least partially shared-nothing:

  • PostgreSQL with Citus / Patroni — Postgres extended with a sharding coordinator layer.
  • Cassandra / DynamoDB — pure peer-to-peer shared-nothing; no single leader.
  • Snowflake / Redshift — shared-nothing compute nodes but with shared cloud object storage (a hybrid, sometimes called "shared-disk for storage, shared-nothing for compute").
  • Vitess — shared-nothing sharding middleware on top of MySQL, used at YouTube.

The label "shared-nothing" sometimes gets blurred when cloud object storage (S3, GCS) plays the role of a durable shared layer beneath independent compute nodes. That hybrid trades some of the coordination simplicity for operational convenience — you can resize compute without migrating data.

Reveal: Why does a query on `status` hit all shards even when there are only a few distinct statuses?
The shard key is user_id, not status. The router has no way to know which shards contain rows with status = 'shipped' without asking every one of them. This is the scatter-gather problem: any filter on a non-shard-key column requires a broadcast to all nodes, and the coordinator must merge the partial results. It's one reason choosing a shard key that matches your hottest query patterns is so important.
