Chapter 2. How Data Is Organized

Chapter 2 Summary: What We Learned

Page Topic Takeaway in One Sentence
011 Logical structure vs physical structure Logical structure is what you see (tables); physical structure is how data is stored (pages, indexes) — they are decoupled.
012 Tables and relations A table is a relation — a set of tuples with named attributes. Thinking in sets, not loops, is the key shift.
013 Keys and identifiers Keys (primary, unique, foreign) identify rows and link tables together — they are the backbone of relational integrity.
014 Constraints and rules Constraints (NOT NULL, CHECK, UNIQUE, FOREIGN KEY) move data validation into the database where it cannot be bypassed.
015 NULL and missing values NULL is not zero or empty string — it means 'unknown' and requires special handling with three-valued logic (IS NULL, COALESCE).
016 Data types and domains Choosing the right type (INT, TEXT, TIMESTAMP, etc.) affects storage size, query performance, and data correctness.
017 Normalized and denormalized data Normalization minimizes redundancy; denormalization trades some redundancy for faster reads. Both have their place.
018 Documents, graphs, and key-value shapes Not all data fits tables: document (JSON), graph (nodes/edges), and key-value models each solve different problems.
019 Choosing the right model Start with relational as the default; reach for document/graph/key-value only when a specific access pattern demands it.
Reveal: How do these ideas connect to the next chapter?
Chapter 2 showed you how data is structured in tables with keys, constraints, and types. You also glimpsed that not all data fits neatly into rows and columns — document, graph, and key-value models each have their place. Chapter 3 goes deeper into the relational model specifically: the mathematical foundation of relations, tuples, and attributes that makes SQL so powerful. Understanding what a relation truly is — a set, not a spreadsheet — will forever change how you think about database design.
