Write-ahead logging basics

Before a database can promise that a committed transaction survives a crash, it needs to record what it intended to do before touching the data files on disk. That mechanism is write-ahead logging (WAL) — the foundation of durability in PostgreSQL, SQLite, MySQL/InnoDB, Oracle, SQL Server, and virtually every other production engine.

The Core Idea: Write the Log First

The "write-ahead" rule is simple:

Before any change is written to the main data file, a record of that change must be written to the log.

The log (also called the redo log, transaction log, or WAL file) is an append-only sequence of log records. Each record captures which page was affected, the modification itself (logical or physical), and a log sequence number (LSN) that orders records globally. PostgreSQL WAL records typically contain the changes made rather than full before-images; full-page images are a special case written only for the first modification to a page after a checkpoint.

Note — PostgreSQL WAL Levels: PostgreSQL offers three WAL levels: minimal (no replication support, skips enough info to save WAL volume during bulk loads), replica (default; enough for streaming replication and archive recovery), and logical (adds logical decoding info for change-data-capture). Use minimal only when you do not need PITR or replication.

WAL file (append-only, on disk)
───────────────────────────────────────────────────────
LSN 100 | txn 42 | BEGIN
LSN 101 | txn 42 | UPDATE accounts SET balance=900 WHERE id=1  (old=1000)
LSN 102 | txn 42 | UPDATE accounts SET balance=1100 WHERE id=2 (old=1000)
LSN 103 | txn 42 | COMMIT
───────────────────────────────────────────────────────
         ↑ flushed to disk before the COMMIT returns to the client

When you issue COMMIT, the engine flushes log records through the COMMIT entry to durable storage (fsync). Only after that flush succeeds does it tell your application "committed." The data pages in the heap file may still be dirty in memory — that is fine, because the log has everything needed to replay the change.

Why This Survives a Crash

Suppose the server loses power one second after your COMMIT returns. The data page was still in the buffer pool and never reached disk. On restart, the database runs crash recovery:

  1. Scan the log from the last checkpoint forward.
  2. Redo every change that has a COMMIT record in the log — this replays the change into the data file.
  3. Undo any changes whose transaction never reached a COMMIT (they were in-flight at crash time).

After recovery, the data files are exactly as if the crash never happened. Committed data is not lost; uncommitted data is rolled back cleanly.

Phase What it does Reason
Redo Re-applies committed log records to data pages Data page may not have been flushed
Undo Reverses log records for uncommitted transactions Partial writes must not be visible

Checkpoints: Keeping Recovery Fast

If recovery always replayed the log from the very beginning, startup after a crash could take hours on a busy system. Checkpoints bound that cost. At a checkpoint the engine:

  1. Flushes all dirty data pages to disk.
  2. Writes a checkpoint record into the log noting the LSN at which this happened.

On recovery, the engine only needs to start from the most recent checkpoint LSN — everything before it is already safe on disk. Most engines take checkpoints automatically every few seconds or every N megabytes of log written.

...older log records (already on disk in data files)...
           ↑
     [CHECKPOINT at LSN 500]
           ↓
LSN 501 → LSN 502 → ... → LSN 610 [COMMIT txn 77]
                                   ↑
                           only this window needs redo

Note: A very long-running transaction delays how far back the log can be truncated, because the undo information for that transaction must be retained. This is why engines warn about or limit idle-in-transaction sessions.

Seeing WAL Behavior in SQLite

SQLite has two storage modes: the classic rollback journal and WAL mode. In WAL mode, writes go to a separate .wal file instead of modifying the database in place. Readers see a consistent snapshot of the main file while a writer appends to the WAL concurrently — better read/write parallelism with no extra complexity for application code.

The Trade-Off: Sequential Writes Win

One reason WAL is so effective is that appending to a log is sequential I/O, which is far cheaper than random writes to scattered data pages. A single sequential fsync of a small log buffer can make dozens of random page updates durable in one shot. This is the fundamental performance bargain that makes WAL both safe and fast.
