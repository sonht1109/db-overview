Chapter Contents

Page Topic Takeaway in One Sentence
111 The lost update problem Two transactions read the same value, both modify it, and only one write survives — the other update is silently lost.
112 Dirty reads and non-repeatable reads A dirty read sees uncommitted data; a non-repeatable read sees different values for the same row within one transaction.
113 Phantom reads A phantom read occurs when a transaction's range query returns different rows because another transaction inserted matching rows.
114 Locks and lock modes Shared locks (S) allow concurrent reads; exclusive locks (X) block everything — lock granularity (row, page, table) is a tradeoff.
115 Two-phase locking Phase 1: acquire all locks. Phase 2: release all locks at commit. Prevents serialization anomalies but can deadlock.
116 Deadlocks and detection A deadlock is a cycle of transactions waiting on each other; the database detects and aborts one to break the cycle.
117 Multiversion concurrency control MVCC keeps multiple versions of rows; readers see a consistent snapshot, writers create new versions — readers never block writers.
118 Snapshot isolation Each transaction sees a consistent snapshot of the database as of its start time — good enough for most applications.
119 Choosing isolation levels SERIALIZABLE is safest but slowest; READ COMMITTED is a practical default; SNAPSHOT ISOLATION balances both.
