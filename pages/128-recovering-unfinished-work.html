Chapter 13. Recovery

Recovering unfinished work

A crash, a power cut, or an application bug can all leave a transaction stranded mid-flight. The database made some changes — maybe it debited an account, deleted a row, or wrote a dozen index entries — but never reached COMMIT. Those partial changes have to disappear as if they never happened. That clean erasure is what recovering unfinished work means, and it is one of the strongest guarantees a database makes.

Why Unfinished Work Is Dangerous

Earlier chapters established that a buffer pool can flush dirty pages to disk at any moment — for performance reasons, not because a transaction committed. If the engine crashes while transaction T9 is midway through updating three tables, some of T9's changes may already be sitting in the data files on disk, while others are still only in RAM (and lost forever).

At restart, the database cannot simply ignore those on-disk partial changes. From the heap file's point of view, the data is inconsistent: some columns reflect T9's new values, others still have the old ones. Any query run against that inconsistent state will return wrong answers.

Note: This is distinct from the redo problem (committed work that did not reach disk). Unfinished work is the opposite half: uncommitted work that did reach disk and must be reversed.

Walking the Log Backward

The write-ahead log (WAL) is the key. Every change a transaction makes is recorded in the log before it touches a data page — and that log record contains both the before-image (old value) and the after-image (new value). To undo a change, the engine just applies the before-image back to the page.

Undo happens in reverse LSN order — the engine walks the transaction's log records from newest to oldest, restoring each page to the state it had before that transaction touched it:

Each undo step is itself written to the WAL as a Compensation Log Record (CLR). The CLR says: "the undo of LSN X has been applied." If the engine crashes again during the undo phase, it does not re-undo steps that CLRs already mark as done. This makes undo survivable under multiple crashes.

PostgreSQL: Recovery Without an Undo Pass

The undo mechanism described above — walking the WAL backward and writing before-images — is the ARIES protocol used by MySQL InnoDB, SQL Server, and others. PostgreSQL does not work this way.

PostgreSQL recovery has only an analysis pass and a redo pass — there is no undo phase. Here is why:

  • PostgreSQL's WAL stores only redo information (after-images). There is no before-image in the WAL record.
  • Uncommitted work is handled by MVCC visibility, not by physical undo. When a transaction modifies a row, the old version stays in the heap page. The new version carries a transaction ID (xmin). If that transaction never commits and the database crashes, recovery marks it as aborted — the row version it created simply becomes invisible to all future transactions.
  • No physical rollback occurs. The "undo" is purely logical: the visibility rules hide the uncommitted version, and the old version (which was never overwritten) remains visible.
  • VACUUM later cleans up the dead row versions.

Note: This is why PostgreSQL recovery is generally faster than ARIES-style recovery — there is no undo pass scanning the log backward. The trade-off is that PostgreSQL accumulates dead tuples, requiring periodic VACUUM to reclaim space and prevent table bloat. Monitoring pg_stat_user_tables.n_dead_tup helps track this overhead.

Crash recovery is not the only way unfinished work gets cleaned up. A running session can deliberately back out to a named savepoint — a marker set earlier in the transaction. Everything after the savepoint is rolled back; work before it stays intact.

BEGIN;

UPDATE accounts SET balance = balance - 200 WHERE id = 42;  -- step 1
SAVEPOINT after_debit;

INSERT INTO transfers (from_id, to_id, amount) VALUES (42, 99, 200);  -- step 2

-- Something went wrong — roll back only to the savepoint:
ROLLBACK TO SAVEPOINT after_debit;

-- Step 1 is still in effect. We can try step 2 differently:
INSERT INTO transfers (from_id, to_id, amount, note) VALUES (42, 99, 200, 'retry');

COMMIT;

The engine uses the same log-walking mechanism internally: it undoes all log records for this transaction that follow the savepoint's LSN, writing CLRs along the way. From the engine's perspective, a savepoint rollback is just a partial undo.

How Long Does Undo Take?

The undo cost is proportional to how much work the transaction did before it was interrupted, not to how large the database is. A transaction that updated 1 million rows takes a long time to undo; one that updated 3 rows is nearly instant.

Transaction size Typical undo time Notes
A few rows Milliseconds Barely noticeable at restart
Thousands of rows Seconds to tens of seconds May surprise users after a crash
Bulk loads / long batches Minutes This is why large ETL jobs are risky without checkpoints

Note: Some databases let you monitor undo progress. In PostgreSQL, pg_stat_activity shows transactions in state idle in transaction (aborted) and the pg_locks view can show blocking. Long undo operations are a real operational concern, not just theory.

Key Takeaways

  • Uncommitted changes that reached disk are the target of the undo phase. They are not automatically safe just because the data file was written.
  • The WAL's before-images make undo possible: every log record stores the value to restore.
  • Undo runs newest-to-oldest within each transaction, so that dependent changes are reversed in the right sequence.
  • CLRs protect undo from being interrupted: if the engine crashes mid-undo, it knows exactly where to resume.
  • Savepoints use the same undo machinery, allowing partial rollbacks inside a still-running transaction.
