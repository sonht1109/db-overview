Part V. Database Families

Chapter 19. Document Databases

Good use cases

Document databases are not universally better or worse than relational databases — they are optimized for a specific set of access patterns. This page walks through the use cases where document databases genuinely shine, explains why they fit, and calls out the warning signs that suggest a relational database would serve better.

The Core Criterion: Document-Centric Access

A document database is a strong fit whenever your application:

  1. Almost always loads an entity in its entirety (not individual fields from many rows).
  2. Stores entities that naturally have variable structure (not all instances look alike).
  3. Rarely needs to query across entities via arbitrary cross-entity joins.

When all three hold, the document model eliminates joins, removes the mismatch between application objects and database rows, and enables schema evolution without migrations.

Use Case 1: Content Management Systems

A CMS stores articles, blog posts, landing pages, FAQs, and product descriptions. Each content type has a different set of fields:

// Article: has author, tags, body, reading_time
{ "_id": "article-991", "type": "article", "title": "...", "author": {...}, "tags": ["tech"], "body": "...", "reading_time": 5 }

// FAQ: has question, answer, category
{ "_id": "faq-042", "type": "faq", "question": "...", "answer": "...", "category": "billing" }

// Landing page: has hero, sections array, cta
{ "_id": "page-home", "type": "page", "hero": {...}, "sections": [...], "cta": {...} }

Forcing these into a relational schema requires either a wide table with many NULLs, or an entity-attribute-value anti-pattern. A document collection holds all types naturally.

Use Case 2: User Profiles and Settings

User profiles are read in full on almost every authenticated request and grow organically over time:

{
  "_id": "usr-8821",
  "email": "priya@example.com",
  "plan": "pro",
  "preferences": {
    "theme": "dark",
    "notifications": { "email": true, "sms": false },
    "locale": "en-IN"
  },
  "integrations": [
    { "provider": "google", "connected_at": "2024-01-10" }
  ],
  "feature_flags": { "beta_dashboard": true }
}

Every new feature adds a new sub-field under preferences or feature_flags. With a document store this requires no migration. With a relational table, every new preference column requires an ALTER TABLE or a separate preference table with a join.

Use Case 3: Product Catalogs with Variable Attributes

E-commerce product catalogs are a classic document database use case precisely because products are heterogeneous:

Product Type Unique Attributes
Clothing size, color, material, gender
Electronics voltage, wattage, dimensions, warranty_years
Digital download file_format, download_size_mb, platform
Perishable food ingredients, allergens, expiry_days, refrigeration

In a relational database you must choose: one giant table with hundreds of mostly-NULL columns, or a separate table per product type (hard to query together), or EAV (complex to query, no type safety). A document collection stores each product's actual attributes naturally.

Use Case 4: Event Logs and Activity Feeds

High-volume append-only data where each event is a self-contained record:

{ "_id": "evt-1001", "type": "user.login", "user_id": "usr-441", "ip": "203.0.113.5", "at": "2025-06-20T08:14:00Z" }
{ "_id": "evt-1002", "type": "order.placed", "user_id": "usr-441", "order_id": "ORD-9901", "total": 250, "at": "2025-06-20T08:15:30Z" }
{ "_id": "evt-1003", "type": "email.sent", "template": "order_confirm", "recipient": "aroha@x.com", "at": "2025-06-20T08:15:31Z" }

Events have different shapes. They are written once and rarely updated. They are read by _id or by user_id + time range. These are excellent conditions for a document store.

Use Case 5: Real-Time Applications and Session Data

Mobile apps and games often need to load a complete user or game state in one round trip:

{
  "_id": "session-9abc",
  "player_id": "usr-882",
  "level": 14,
  "position": { "x": 120, "y": 340 },
  "inventory": ["sword-bronze", "potion-health", "key-iron"],
  "stats": { "hp": 87, "mp": 50, "xp": 14200 },
  "quest_flags": { "castle_unlocked": true, "boss_defeated": false }
}

The entire game state is one document. Saving progress is one write. Loading the game is one read. This is exactly the document model's sweet spot.

What Signals a Poor Fit

You are fighting the document model when:

  • You find yourself writing many $lookup stages (joining collections). You want a relational database.
  • You need multi-document transactions frequently. You are modeling relational data as documents.
  • Your documents reference dozens of other collections. The data is naturally relational.
  • Your analytics team cannot query the data without exporting it first. The document model is hostile to ad-hoc aggregation.
  • You need schema enforcement for compliance or data quality. A relational database with CHECK constraints and NOT NULL is more appropriate.

The heuristic: If your entity relationships look like a graph (many cross-entity links, arbitrary depths), or if your analytics team writes complex GROUP BY queries, the relational or graph model is likely a better fit. Document databases are for document-shaped data with document-shaped access patterns.

Key Takeaways

  • Document databases are the right tool when: your entities are hierarchical, read as a whole, variable in structure, and not frequently joined with other entity types.
  • The best use cases are: CMS content, user profiles, product catalogs with heterogeneous attributes, event logs, and game/session state.
  • Warning signs that you need a different tool: many cross-collection joins, frequent multi-document transactions, complex aggregation pipelines, or regulatory requirements for schema enforcement.
  • "Use a document database because it's flexible" is not a sufficient reason — the flexibility is a trade-off, not a free lunch. Match the model to the access pattern.
