Chapter 20. Column-Family Databases

Chapter 20 Summary: What We Learned

Chapter Contents

Page Topic Takeaway in One Sentence
191 Wide rows and sparse data Column-family stores organize data into rows with many sparse columns — each row can have millions of columns but only populated ones are stored.
192 Partitioning and clustering keys The partition key distributes data across nodes; the clustering key sorts rows within a partition — both determine query patterns.
193 Write-optimized design Column-family databases use LSM trees for write-optimized storage — sequential writes, background compaction for reads.
194 LSM trees Write to an in-memory memtable, flush to sorted SSTables on disk, compact SSTables in the background to reduce read amplification.
195 Compaction Merging and sorting multiple SSTables into fewer files — critical for read performance but consumes I/O and CPU.
196 Time-series and event workloads Column-family databases excel at high-ingest, append-heavy workloads: metrics, logs, IoT sensor data, user events.
197 Modeling for query patterns Design tables around the queries you need, not around normalizing data — denormalization is expected and necessary.
198 What is hard in these systems No joins, limited ad-hoc queries, complex compaction tuning, and data modeling requires deep understanding of access patterns.
199 Good use cases Business intelligence dashboards, periodic reports, machine learning feature engineering, log analysis, financial reconciliation.
