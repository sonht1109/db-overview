Buffer pools

Every database page you read about in the previous sections — heap pages, B-tree nodes, WAL records — lives on disk. But disk is slow: even a fast NVMe drive is thousands of times slower than RAM. The buffer pool is the database engine's main memory cache. It sits between your queries and the disk, and almost every read or write passes through it.

What a Buffer Pool Is

The buffer pool is a fixed-size region of RAM that the database manages itself, divided into frames. Each frame holds exactly one page (typically 8 KB or 16 KB, matching the on-disk page size). When the engine needs a page, it checks the buffer pool first:

  1. Cache hit — the page is already in a frame. Return it immediately.
  2. Cache miss — the page is not in memory. Find a free frame (or evict an existing one), read the page from disk into that frame, then return it.

A small hash table maps (file_id, page_number) → frame index, so lookups are O(1).

Disk                        Buffer Pool (RAM)
┌──────────┐               ┌────────┬────────┬────────┬────────┐
│ page 1   │◄──── miss ────│ frame0 │ frame1 │ frame2 │ frame3 │
│ page 2   │               │ page 7 │ page 2 │ page 9 │  free  │
│  ...     │               └────────┴────────┴────────┴────────┘
│ page N   │                    ▲         ▲
└──────────┘               query A   query B

Because most real workloads access a small "hot" set of pages repeatedly, a modest buffer pool can absorb the vast majority of disk I/O. PostgreSQL defaults to 128 MB (shared_buffers); production systems routinely allocate tens of gigabytes.

Dirty Pages and the Write Path

When a query modifies a page (an INSERT, UPDATE, or DELETE), the engine changes the in-memory copy and marks that frame dirty. The dirty page is not written to disk immediately — this is intentional. Batching writes is far more efficient than flushing every change one at a time.

Dirty pages are eventually written to disk by a background process (PostgreSQL calls it the bgwriter; InnoDB has its own page cleaner). The write-ahead log (WAL) you read about earlier ensures that even if the server crashes before a dirty page is flushed, the change can be replayed from the log. That is the contract: the log is durable, the buffer pool is not.

Note: A page must never be evicted from the buffer pool while a transaction that modified it has not yet written its log record to disk. This ordering rule — "log before evict" — is a cornerstone of crash recovery.

Eviction and Replacement Policy

When the buffer pool is full and a new page must be loaded, the engine must choose a victim frame to evict. The classic policy is LRU (Least Recently Used): evict the page that was accessed least recently.

LRU works well for random access patterns, but it can backfire during a sequential scan. If a query scans a huge table, every page gets loaded exactly once, flooding the pool and evicting hot pages that other queries need. Most engines handle this with clock sweep (PostgreSQL) or a midpoint insertion variant of LRU (InnoDB), which gives new pages a lower priority until they are accessed a second time.

Policy Good for Weakness
LRU Mixed OLTP workloads Sequential scans flush hot data
Clock sweep General use, low overhead Approximate, not strictly LRU
LRU-K (k=2) Mixed + analytical More bookkeeping
2Q / midpoint Scan-heavy workloads Tuning required

Seeing Buffer Pool Pressure in Practice

Buffer pool behavior is mostly invisible — until the pool is too small. When hit rates drop, queries that used to be fast suddenly wait on disk I/O. Most databases expose cache statistics you can query directly.
