Chapter 24. Designing for Reliability

Point-in-time recovery

A developer runs DELETE FROM orders; without a WHERE clause at 14:32 on a Tuesday. With a weekly backup, you lose up to six days of data. With point-in-time recovery (PITR), you restore to 14:31:59 and lose nothing but 60 seconds. PITR is the capability that transforms a coarse backup strategy into a precise surgical tool — and understanding how it works is essential for any database reliability practice.

How PITR Works

All major relational databases maintain a write-ahead log (WAL) — a sequential record of every change made to the database, written before the change hits the data files. Crash recovery already relies on this log. PITR extends that mechanism for restore purposes:

  1. Take a base backup (a consistent snapshot of the data files at a known LSN/position)
  2. Archive WAL segments continuously to durable storage as they are filled
  3. At restore time, copy the base backup to a fresh server, then replay WAL forward from the backup's end position to any target point (timestamp, transaction ID, or named restore point)

The database reaches exactly the state it was in at the chosen moment — every committed transaction before that point is present, every transaction after is absent.

Configuring PITR in PostgreSQL

Step 1 — Enable WAL archiving

# postgresql.conf
wal_level = replica          # or logical
archive_mode = on
archive_command = 'aws s3 cp %p s3://my-wal-archive/%f'

%p is the WAL segment path; %f is the filename. The archive command must return exit code 0 only when the file is safely stored.

Step 2 — Take a base backup

pg_basebackup -h localhost -U replicator -Ft -z -Xs -P \
  -D /mnt/backups/base_$(date +%Y%m%d)

-Xs streams WAL during the backup so no segments are missed. -Ft -z produces a compressed tar.

Step 3 — Restore to a point in time

# 1. Place base backup in data directory
tar -xzf base_20240624.tar.gz -C /var/lib/postgresql/data

# 2. Create recovery configuration (Postgres ≥ 12)
cat >> /var/lib/postgresql/data/postgresql.conf <<EOF
restore_command = 'aws s3 cp s3://my-wal-archive/%f %p'
recovery_target_time = '2024-06-24 14:31:59'
recovery_target_action = 'promote'
EOF

touch /var/lib/postgresql/data/recovery.signal

# 3. Start — engine replays WAL, stops at target, promotes to read-write
pg_ctl start -D /var/lib/postgresql/data

Named Restore Points

Instead of a timestamp, you can use a named restore point created with SELECT pg_create_restore_point('before-big-migration');. This is more reliable than matching a clock time and is common practice before large schema changes.

RPO and RTO Implications

Metric Meaning How PITR affects it
RPO (Recovery Point Objective) Maximum data loss acceptable WAL archiving gives RPO down to the last archived segment (typically seconds)
RTO (Recovery Time Objective) Maximum time to restore Depends on base backup size + WAL replay volume; can range from minutes to hours

Practical note: WAL replay speed is roughly 1–3 GB/minute on modern hardware. A 500 GB base backup plus 60 GB of WAL (6 hours of heavy write load) may take 45–90 minutes to restore. Plan for this in your SLA.

Common Pitfalls

  • Gaps in WAL archive — if one segment fails to archive, replay stops before the target time. Monitor archive success with alerts on pg_stat_archiver.failed_count.
  • Clock skew — recovery target times are wall clock times on the standby server. Ensure NTP is synchronised.
  • WAL level too low — wal_level = minimal does not produce enough WAL for PITR. Set to replica or logical.
  • Forgotten recovery.signal — without this file (Postgres ≥ 12), the engine starts normally, ignoring restore_command and recovery_target_time.

Key Takeaways

  • PITR replays the WAL forward from a base backup to any chosen second — it is your best tool against accidental data loss
  • Configure WAL archiving before you need it; you cannot archive retroactively
  • Named restore points are more reliable than clock timestamps before risky operations
  • WAL replay speed sets your RTO floor; size base backups and archive storage accordingly
  • Monitor the archive process actively — a silent failure means the next restore will fall short
