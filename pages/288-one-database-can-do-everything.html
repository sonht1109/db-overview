Part VII. Building Understanding

Chapter 29. Common Myths About Databases

One database can do everything

Every database vendor's marketing implies their product can handle any workload. PostgreSQL can do JSON documents, time-series, full-text search, and geospatial. MongoDB can handle analytics. DynamoDB claims any scale. In practice, no single database excels at every data access pattern simultaneously — and trying to force one to do everything often means it does everything badly. Understanding which problems genuinely require specialized databases is as important as understanding how each database works.

The Myth: One Ring to Rule Them All

The appeal of a single database is real: one operational system to monitor, one backup strategy, one team to learn one query language, one billing line. These operational benefits are genuine and should not be dismissed. For many applications — perhaps most — a single well-chosen database is genuinely the right answer.

The myth is the extension of this to "any database can handle any workload at any scale." Different databases optimize for different access patterns, and those optimizations involve fundamental architectural choices that cannot be papered over with plugins.

Access Pattern Taxonomy

Different data access patterns have genuinely different optimal storage structures:

Access Pattern Optimized Storage Examples
Random key lookups Hash table / B-tree Session storage → Redis, DynamoDB
Complex queries / joins Row-store B-tree Transactional data → PostgreSQL, MySQL
Time-series append + range TSM / LSM with time partitioning Metrics → InfluxDB, TimescaleDB
Full-text search Inverted index Search → Elasticsearch, Meilisearch
Graph traversal Adjacency list + index Social graphs → Neo4j, Amazon Neptune
Column aggregations Column-store Analytics → ClickHouse, BigQuery
Geospatial queries R-tree / GiST Location data → PostGIS, MongoDB

A B-tree row store (PostgreSQL) can simulate all of these. But "can" and "optimally handles" are different claims. Running a time-series workload on a row-store database with timestamp ranges requires careful partitioning and index management. Running it on TimescaleDB (which is built on PostgreSQL, with time-series-specific compression and partitioning) gives you 10–100× better performance and storage efficiency out of the box.

The Hidden Costs of Polyglot Persistence

If each database is optimized for its workload, why not use five databases for everything? Because polyglot persistence has real operational costs:

  • Operational complexity — five databases means five monitoring setups, five backup strategies, five upgrade cycles, five sets of expertise to maintain.
  • Data synchronization — when PostgreSQL is the source of truth and Elasticsearch holds a search index, any write to PostgreSQL must eventually sync to Elasticsearch. This sync can fail, lag, or drift. Handling sync failures is non-trivial application logic.
  • Cross-store queries — a query that joins transactional data with analytics data from two different stores must be assembled in the application layer or an ETL pipeline. There is no cross-store SQL.
  • Team knowledge — each specialist database requires someone who understands it deeply. For small teams, this is prohibitive.

The right answer is not "use every database" or "use one database." It is use the minimum number of stores that genuinely serve your access patterns, and add specialization only when the benefit outweighs the operational overhead.

When to Add a Specialist Database

The decision to add a new database to the stack should be triggered by a measured pain point, not anticipation:

Trigger Consider
Full-text search quality is poor with LIKE '%term%' Elasticsearch or Meilisearch
Time-series queries are slow and storage is ballooning TimescaleDB or InfluxDB
Cache miss rate is causing latency spikes Redis or Memcached
Graph traversal queries take seconds Neo4j or Amazon Neptune
Analytics queries block production DB ClickHouse or read replica

Before adding a specialist store, check whether an extension to your existing database solves the problem:

  • PostgreSQL + pg_trgm extension for fuzzy text search
  • PostgreSQL + TimescaleDB extension for time-series
  • PostgreSQL + PostGIS extension for geospatial

Extensions often give you 80% of the specialist performance with none of the operational complexity of a separate system.

The Pragmatic Starting Point

For most new systems, a pragmatic starting point is:

  1. Start with PostgreSQL — it covers transactional workloads, has excellent JSON support for semi-structured data, PostGIS for geospatial, and full-text search sufficient for many use cases.
  2. Add Redis when session storage or caching latency becomes a measured bottleneck.
  3. Add a specialist store only when you have measured that PostgreSQL with the appropriate extension cannot meet the requirement.

This strategy keeps the stack simple at the start while leaving a clear migration path as needs evolve.

Key Takeaways

  • No single database is optimal for all access patterns — the architectural choices that make one pattern fast make others slower.
  • Polyglot persistence (using multiple specialized databases) is a real solution but carries real operational costs: synchronization complexity, multiple expertise requirements, and cross-store query limitations.
  • Prefer extensions (PostGIS, TimescaleDB, pg_trgm) over separate systems when they can solve the problem at 80% of the performance.
  • Start with one database and add specialists only in response to measured bottlenecks, not anticipated ones.
  • The operational cost of a new database is permanent; the performance benefit must clearly exceed it.
