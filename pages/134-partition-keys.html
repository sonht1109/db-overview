Chapter 14. From One Machine to Many

Partition keys

When you split a large table across multiple nodes (sharding), every row has to land somewhere. The partition key — sometimes called the shard key — is the column (or columns) whose value decides which node a row lives on. Choosing it well is one of the most consequential design decisions in a distributed system, because you can't easily change it later without moving every row.

How a Partition Key Works

The database (or a routing layer in front of it) takes the partition key value, runs it through a partitioning function, and maps the result to a shard. Three strategies are common:

Strategy How rows are assigned Good for Watch out for
Range Consecutive key values go to the same shard (e.g. user_id 1–1 M on shard A) Range scans, time-series Hot spots when new data concentrates on one shard
Hash hash(key) mod N → shard index Even spread of random writes Range queries must scatter to all shards
List Explicit mapping of values to shards (e.g. country → shard) Geographic locality Manual maintenance; skewed if one bucket is huge

Hot Spots: The Main Enemy

A hot spot happens when one shard receives a disproportionate share of traffic. This defeats the whole point of sharding — you've still got one machine doing all the work.

Common causes:

  • Monotonically increasing keys (like auto-increment IDs or timestamps) with range partitioning: all new inserts land on the last shard.
  • Skewed data with list partitioning: 80% of your orders might come from one region.
  • Famous entities: a celebrity's account being accessed millions of times per second hits only one shard.

Hash partitioning largely solves the monotonic-key problem because the hash scrambles the distribution. But it creates a new tradeoff: you lose the ability to do efficient range scans.

Note: Some systems (like Apache Cassandra) add a random suffix to hot partition keys — for example, a celebrity user ID becomes user:42:0, user:42:1, … user:42:9 — then aggregate across them at read time. This is called write sharding or salting.

Queries and the Partition Key

The partition key shapes every query in your system:

  • Key-aligned query (filter on the partition key): the router sends the query to exactly one shard. Fast and cheap.
  • Scatter-gather query (filter on a non-partition column): the router broadcasts to all shards, waits for each to reply, then merges the results. Cost scales with the number of shards.

This is why picking the partition key to match your most frequent query pattern matters so much. If 90% of your queries look up orders by user_id, partitioning on user_id makes those fast. If you instead partition on order_id and query by user_id, every lookup becomes a scatter-gather.

Partition Keys in Practice

Real distributed databases make the partition key explicit in their data model:

  • Apache Cassandra: you declare a PRIMARY KEY ((partition_key), clustering_cols). The double-parenthesis part is the partition key; the clustering columns sort data within the partition.
  • DynamoDB: every table has a mandatory partition key (and an optional sort key). Queries without the partition key require a full-table scan via a Global Secondary Index.
  • CockroachDB / Spanner: use range-based partitioning under the hood, but expose it through SQL PARTITION BY clauses on indexes.

The recurring theme: you choose the partition key at schema-design time, and the system optimizes the data placement around it. Change it later and you're re-writing the whole table — which in a distributed system means shuffling gigabytes or terabytes across the network. Get it right up front.

Reveal: Why can't you just index every column and avoid the scatter-gather problem?
You can add secondary indexes, but in a distributed system a secondary index on a non-partition column is itself partitioned — and usually differently from the base table. A query on that column either hits a global secondary index (a separate distributed index that must be kept in sync with every write, adding latency and consistency complexity) or it still scatters to all shards. Either way there's a cost. Primary key lookups remain the only truly cheap, single-shard operation.
