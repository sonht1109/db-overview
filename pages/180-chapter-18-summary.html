Part V. Database Families

Chapter 18. Key-Value Databases

Chapter 18 Summary: What We Learned

Chapter Contents

Page Topic Takeaway in One Sentence
171 The key-value model A simple dictionary: put(key, value), get(key), delete(key). The simplest database model with the fewest guarantees.
172 Fast lookups and simple APIs With no query language or schema, key-value stores achieve sub-millisecond latency — ideal for caching and session storage.
173 Range scans and ordered keys Ordered key-value stores (like RocksDB) support range queries on sorted keys, useful for time-series and prefix scans.
174 Common storage engines LSM trees (write-optimized) and B-trees (read-optimized) are the two dominant key-value storage backends.
175 Caching vs persistence Some key-value stores prioritize speed and cache data (Redis); others prioritize durability and persist to disk (RocksDB).
176 When key-value is enough When access is always by primary key, no joins are needed, and data is simple — session tokens, config, counters.
177 Limits of the model No joins, no aggregations, no declarative queries. Complex queries require application code. Embedding logic in keys is a warning sign.
178 Real-world examples Redis for caching/rate-limiting, DynamoDB for simple workloads, etcd for configuration — each optimized for a specific pattern.
179 Design patterns Use TTL for auto-expiry, hash slots for partitioning, sorted sets for leaderboards — key-value systems reward clever key design.
