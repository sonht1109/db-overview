Part III. Storage and Execution

Chapter 13. Recovery

Steal and no-steal

When a transaction modifies data, those changes live first in the buffer pool — the region of RAM where the database caches disk pages. The question of when those dirty (modified) pages are allowed to reach disk — and which transactions' changes can go first — turns out to be one of the most consequential design decisions in a storage engine. Two paired policies answer it: steal / no-steal and force / no-force. This page covers the steal side; the next covers force.

The Problem: Dirty Pages and Crashes

Imagine transaction T1 has updated a page in the buffer pool but hasn't committed yet. Meanwhile the buffer pool is running low on space, so the page manager wants to evict that page to make room for something else. If it writes the dirty page to disk, it has stolen that slot on behalf of T1 before T1 is done. What happens if T1 then aborts — or the system crashes mid-write?

That is the steal problem. The policy you choose shapes what work recovery must do.

Steal vs. No-Steal

Policy Definition Implication for recovery
No-steal A dirty page belonging to an uncommitted transaction is never written to disk. Undo is not needed — uncommitted data never reaches disk.
Steal The buffer manager may evict and write a dirty page to disk even before its transaction commits. Recovery must be able to undo changes of transactions that were in-flight when the crash happened.

No-steal: simple but expensive

No-steal is conceptually clean: if a page is dirty and its transaction hasn't committed, it stays in RAM. There is nothing to undo on disk after a crash because uncommitted data never got there. The downside is memory pressure — you can never reclaim buffer frames owned by long-running transactions, and if one transaction touches enough pages, the buffer pool fills up and work grinds to a halt.

Steal: realistic but requires undo

Almost every production database uses steal. The buffer manager is free to evict any page at any time. This keeps memory usage bounded regardless of transaction size. The cost: if the system crashes, some pages on disk may contain changes from transactions that never committed. The recovery subsystem must undo those changes by consulting the write-ahead log (WAL).

Note: "Steal" is the buffer manager stealing a frame from an uncommitted transaction. The frame goes to disk, and the transaction loses its claim on that memory slot.

How Steal Interacts with the Write-Ahead Log

Steal only works safely alongside WAL. Before a dirty page is written to disk, the log record describing that change must already be on disk (the WAL rule: log before data). That way, if a crash occurs right after the stolen page lands on disk, the recovery process can find the log entry and roll back the change.

Choosing a Policy in Practice

Real systems overwhelmingly choose steal + no-force (no-force is covered next). This combination gives the buffer manager maximum freedom: evict any page whenever needed, and never force a commit to flush all its pages to disk synchronously. The trade-off is that recovery must handle both undo (for stolen, uncommitted pages) and redo (for committed pages that hadn't reached disk yet). Systems like InnoDB, PostgreSQL, and SQLite all implement undo logs or MVCC rollback segments precisely to support steal.

No-steal survives in niche settings — some in-memory databases or logging systems where all data fits comfortably in memory — but it is rarely viable for general-purpose workloads.

Reveal: Why can't no-steal simply pin all dirty pages in RAM?
It can — and that is exactly what no-steal does. But "pin in RAM" means that memory is reserved until the transaction commits. If a transaction touches thousands of pages (say, a bulk update), all of those frames are locked in the buffer pool. When the pool is exhausted, every other query that needs a new page is blocked. Steal releases that pressure by allowing eviction, at the cost of needing undo support in recovery.

Quick Summary

No-steal Steal
Uncommitted dirty pages on disk? Never Yes (when evicted)
Undo needed at recovery? No Yes
Memory pressure High for large transactions Bounded — frames can be recycled
Used in practice? Rarely Almost universally

The steal/no-steal choice is invisible to application developers writing SQL, but it quietly determines how large a transaction you can run and how fast the database recovers from a crash.
