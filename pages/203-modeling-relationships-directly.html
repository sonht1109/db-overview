Part V. Database Families

Chapter 21. Graph Databases

Modeling relationships directly

Relational databases model relationships as a derived concept — you reconstruct them at query time by joining tables. Graph databases flip that design on its head: relationships are first-class citizens stored alongside nodes, ready to traverse at near-zero cost regardless of dataset size. This page shows exactly what that means in practice and why it matters for highly connected data.

The Relational Way: Relationships as Tables

In a relational database, relationships live inside data — encoded as foreign keys or resolved through join tables. Consider a simple social graph where people can follow each other.

-- Relational schema for a social graph
CREATE TABLE users (
  id   INTEGER PRIMARY KEY,
  name TEXT NOT NULL
);

CREATE TABLE follows (
  follower_id INTEGER REFERENCES users(id),
  followee_id INTEGER REFERENCES users(id),
  PRIMARY KEY (follower_id, followee_id)
);

This is clean and normalized. A single hop ("who does Alice follow?") is one join. But add another hop and you join the same tables again:

-- Two hops: "friends of friends" of Alice (id = 1)
SELECT DISTINCT u3.name
FROM users u1
JOIN follows f1 ON u1.id = f1.follower_id
JOIN users u2 ON f1.followee_id = u2.id
JOIN follows f2 ON u2.id = f2.follower_id
JOIN users u3 ON f2.followee_id = u3.id
WHERE u1.id = 1;

Three hops adds another pair of joins. The query grows linearly with depth, and each new join forces the database to re-scan the follows table — which can have hundreds of millions of rows in a real social network.

Key problem: In a relational database, the cost of a relationship traversal scales with the size of the join table, not the degree of the start node. If follows has 500 million rows, every hop through it is expensive.

The Graph Way: Relationships as Stored Edges

A property graph database like Neo4j stores data as nodes and edges. An edge is not a lookup — it is a physical pointer from one node to another. Traversal means following a pointer, not scanning a table.

Each node stores a list of its own edges. To find Alice's followees, the database reads Alice's node and follows her outgoing edge pointers — it never touches any data belonging to other users. Depth-2 traversal doubles the pointer-following work, not the scan width.

Node: Alice  →  edge ptr → Bob's node  → edge ptr → Carol's node
              → edge ptr → Dave's node → edge ptr → Eve's node

No index needed. No join table. The relationship itself is the path.

Property Graphs: Nodes and Edges with Attributes

Most graph databases implement the property graph model. Both nodes and edges can carry arbitrary key-value properties — there is no need for a separate attribute table.

Element What it stores Example
Node Entity + label(s) + properties (:Person {name:"Alice", age:31})
Edge Directed relationship + type + properties [:FOLLOWS {since:"2022-01"}]

Properties on edges are a critical advantage over join tables. In a relational schema, adding a followed_since date to a many-to-many relationship requires altering the join table or adding a new one. In a property graph you simply add the property to the edge.

// Property graph notation (Cypher-like)
(Alice:Person {age:31}) -[:FOLLOWS {since:"2022-01"}]-> (Bob:Person {age:28})

Index-Free Adjacency

The technical term for what makes graph traversal fast is index-free adjacency: each node directly references its neighbors. When you load Alice's node, you get Alice's edge list; no secondary lookup is needed.

Compare this with relational adjacency, where finding neighbors always requires querying an index on follower_id in the follows table. That index lookup is O(log n) at best, repeated for every hop, across a table that grows with every new relationship in the entire system.

In a property graph:

  • Hop cost = number of edges on the current node (local)
  • In relational = log(total rows in join table) per hop (global)

For sparse graphs with millions of users and billions of edges, this difference is orders of magnitude in traversal speed.

Note: Index-free adjacency does not help with global aggregations like "count all edges." For those, a relational database may still be faster. Graph databases shine on local traversal from a known start node.

When to Choose a Graph Database

Graphs are the right tool when your most important queries traverse relationships of unknown or variable depth:

  • Social networks — friend recommendations, mutual connections, influence propagation
  • Fraud detection — ring detection, money-laundering chains, account linkage graphs
  • Knowledge graphs — entity relationships, ontology traversal, recommendation engines
  • Network topology — routing, dependency resolution, impact analysis

If your queries are mostly lookups by a single attribute and joins are shallow (1–2 hops, bounded), a relational database handles the job more simply. The graph model earns its keep when relationship traversal is the central operation.

Key Takeaways

  • Relational databases encode relationships as data (foreign keys, join tables); graph databases encode them as structure (physical edge pointers).
  • Index-free adjacency means hop cost is proportional to the local degree of a node, not the global size of any table.
  • Edges in a property graph are first-class: they carry a type and arbitrary properties, avoiding the need for auxiliary tables.
  • The "friend-of-friend" query illustrates the core asymmetry: adding a hop in SQL requires a new join; in a graph query language it is a single parameter change.
  • Graph databases are not a replacement for relational systems — they are purpose-built for workloads where traversing deeply connected data is the primary access pattern.
