Part VII. Building Understanding

Chapter 27. A Tiny Database from Scratch

Chapter 27 Summary: What We Learned

Page Topic Takeaway in One Sentence
261 A toy table format Start with the simplest possible storage: a fixed-length record format in a flat file. Understanding this makes real systems less magical.
262 Insert and scan Writing records at the end (append) and scanning sequentially — this is O(n) but teaches the fundamental scan operator.
263 Add a simple index Build an in-memory hash map from key to file offset — now lookups are O(1) instead of O(n). The core index idea.
264 Add a parser Write a tiny SQL parser that recognizes SELECT, FROM, WHERE — transform text into a query AST. Parsing is simpler than it looks.
265 Add a planner Transform the parsed query into a plan: choose between index scan and full table scan based on whether a WHERE clause matches the index.
266 Add transactions Implement a simple WAL: log changes before applying them, so you can replay after a crash. This makes writes durable.
267 Add recovery On startup, replay the WAL to restore the database to its pre-crash state — the simplest possible crash recovery.
268 Add concurrency control Use a readers-writer lock: multiple reads concurrent, writes exclusive. This introduces the concepts of isolation and blocking.
269 What this teaches you Building a toy database demystifies indexes, planners, WAL, and locking — real databases are the same ideas, production-hardened.
Reveal: How do these ideas connect to the next chapter?
Chapter 27 demystified databases by building a tiny one. Now Chapter 28 puts everything into practice through case studies — real-world scenarios that force you to apply what you have learned. You will design schemas for a to-do app, an e-commerce platform, a chat system, an analytics pipeline, and a multi-tenant SaaS. Each case study reinforces that data modeling is about tradeoffs, and that the right answer depends on your specific query patterns and consistency needs.
