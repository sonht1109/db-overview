Part IV. Distributed Systems

Chapter 14. From One Machine to Many

Rebalancing data

When you spread data across multiple nodes, the distribution is rarely perfect forever. Nodes are added to handle growth, removed when they fail or are retired, and some partitions naturally attract more traffic than others. Rebalancing is the process of moving data between nodes to restore an even distribution — keeping every machine carrying a fair share of the load without taking the cluster offline.

Why Data Drifts Out of Balance

In Chapter 13 you saw how partitioning splits a dataset into shards assigned to specific nodes. The assignment that was sensible last month may be a problem today for several reasons:

Trigger What happens
Node added The new node sits idle while existing nodes stay overloaded.
Node removed / failure Its partitions must move somewhere or they become unreachable.
Hot partition One key range attracts disproportionate traffic (e.g., a viral user).
Data growth A partition that started small may now dwarf the others.

The goal of rebalancing is always the same: spread both data and request load roughly evenly, move as little data across the network as possible, and keep the cluster serving reads and writes throughout the process.

Strategies for Rebalancing

Fixed number of partitions

Create far more partitions than nodes from the very beginning — say, 1000 partitions for a 10-node cluster. Each node owns ~100 partitions. When you add an eleventh node, simply move a few whole partitions to it; no data inside a partition is reorganized.

Trade-off: you must choose the total number of partitions upfront. Too few and you cannot distribute finely; too many and the overhead of tracking them grows.

Dynamic partitioning

Used by databases like HBase and RethinkDB. Partitions split automatically when they exceed a size threshold (say, 10 GB) and merge when they shrink. The cluster tracks partition boundaries in real time, so you never have to pre-commit to a partition count.

Consistent hashing (virtual nodes)

Systems like Cassandra and DynamoDB assign each node multiple positions ("virtual nodes" or vnodes) on a hash ring. When a node is added, it takes over a few vnodes from its neighbors. When a node leaves, its vnodes are handed to the remaining nodes. The key insight: only the data belonging to those specific vnodes moves — everything else stays put.

Note: Consistent hashing dramatically reduces the fraction of keys that need to move when the cluster changes size. With a naive modulo hash (key % N), changing N from 10 to 11 remaps almost every key. With consistent hashing, only ~1/N of the keys move.

Automatic vs. Manual Rebalancing

Most production databases offer a choice:

  • Automatic rebalancing — the cluster decides when and how to move data (e.g., Cassandra's nodetool repair, MongoDB's balancer). Convenient, but a rebalance can spike network and disk I/O exactly when the cluster is already under load.
  • Manual rebalancing — an operator triggers moves explicitly. This gives you control over timing (schedule it during off-peak hours) at the cost of operational complexity.

Many systems land in the middle: the cluster detects imbalance and proposes a rebalancing plan, but a human must approve or schedule it.

What Moves During a Rebalance

During a rebalance the cluster streams partition data from one node to another. While the transfer is in progress:

  1. The source node continues serving reads and writes for the partition.
  2. New writes are also forwarded to the destination node so it stays current.
  3. Once the transfer is complete, the routing table is updated and the destination node becomes the primary owner.

This "copy then switch" approach keeps the partition online throughout the move — clients never see a gap in availability.

Reveal: Why not just stop writes during a rebalance?

Stopping writes would guarantee the source and destination end up identical, which sounds clean — but it means downtime. Real clusters can't afford to go offline every time a node is added or removed. Instead, databases track which writes arrived after the copy started (via a replication log or version counter) and apply those delta writes to the destination before doing the final cutover. The window where both copies must agree is shrunk to milliseconds.

Key Takeaways

Strategy Partition count Best for
Fixed partitions Chosen upfront, never changes Predictable workloads, simple ops
Dynamic partitioning Grows/shrinks with data Datasets with unpredictable size
Consistent hashing / vnodes Scales with node count Large clusters, frequent membership changes

Rebalancing is not a one-time event — it is an ongoing background process in any long-lived distributed database. Understanding it helps you predict when your cluster will be most vulnerable to latency spikes and gives you the tools to schedule that risk intelligently.
