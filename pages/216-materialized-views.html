Materialized views

Queries that aggregate millions of rows into a dashboard metric are expensive — but they tend to run repeatedly against largely the same data. A materialized view turns this pattern inside-out: instead of recomputing the aggregation on every query, the result is precomputed and stored on disk, then served like a regular table. The computation moves from query time to refresh time. That trade-off — faster reads, potentially stale results — defines when materialized views shine and when they create problems.

Regular Views vs Materialized Views

A regular (logical) view is nothing more than a named query stored in the catalog. Every time you query the view, the database substitutes the view definition and runs the underlying query from scratch.

-- Regular view: recomputed every time it is queried
CREATE VIEW daily_revenue AS
SELECT sale_date, SUM(sale_amount) AS revenue
FROM fact_sales
GROUP BY sale_date;

-- This SELECT triggers the full aggregation every time:
SELECT * FROM daily_revenue WHERE sale_date = '2024-06-25';

A materialized view stores the result of that query as actual rows on disk. The query above runs once during refresh, and subsequent reads hit the precomputed table.

-- Materialized view: result stored on disk
CREATE MATERIALIZED VIEW daily_revenue_mv AS
SELECT sale_date, SUM(sale_amount) AS revenue
FROM fact_sales
GROUP BY sale_date;

-- Refresh (recompute and store):
REFRESH MATERIALIZED VIEW daily_revenue_mv;

-- This SELECT now reads from the stored rows — very fast:
SELECT * FROM daily_revenue_mv WHERE sale_date = '2024-06-25';

Query Path Comparison

When to Use Materialized Views

Materialized views are most valuable when:

  • The underlying query is expensive (aggregating millions of rows, joining large tables)
  • The query is run frequently (dashboard loaded by dozens of users per minute)
  • The data does not need to be perfectly current (stale by minutes or hours is acceptable)
  • The result set is significantly smaller than the base data (a daily summary is tiny vs. billions of transaction rows)

They are less useful when:

  • Data changes constantly and staleness of even a few seconds is unacceptable
  • The query is run only occasionally (the refresh cost outweighs the savings)
  • The result set is nearly as large as the base data (storage cost with no real benefit)

Refresh Strategies

How and when a materialized view is updated defines its behavior:

Strategy How it works Freshness Cost
Manual refresh Operator runs REFRESH explicitly Depends on schedule Low (full recompute on demand)
Scheduled refresh Cron job or warehouse scheduler triggers refresh Predictable lag Medium
On-commit refresh Refreshed automatically when base table commits Near real-time High — refresh on every write
Incremental refresh Only rows changed since last refresh are reprocessed Near real-time Low per refresh, complex to implement

Incremental Refresh

Incremental (also called differential) refresh is the most sophisticated and most valuable strategy. Instead of recomputing the entire result, the system tracks which base rows changed and merges only those deltas into the materialized view.

new_mv_result = old_mv_result
              + aggregations of newly inserted rows
              - aggregations of deleted rows
              + updated aggregations for modified rows

This works cleanly for additive aggregates (SUM, COUNT). It is harder for non-additive aggregates (MEDIAN, PERCENTILE) because you cannot easily combine partial results. Support for incremental refresh varies by database:

  • PostgreSQL: No incremental refresh built-in; full refresh only with REFRESH MATERIALIZED VIEW
  • Oracle: Supports incremental refresh via materialized view logs
  • Snowflake: Dynamic tables with automatic incremental refresh
  • BigQuery: Materialized views with incremental updates for supported query shapes

Query Rewriting

Advanced optimizers do not require you to explicitly query the materialized view — they automatically rewrite queries against the base table to use the materialized view instead, if the view's precomputed result can satisfy the query.

-- User writes this query against the base table:
SELECT sale_date, SUM(sale_amount) AS revenue
FROM fact_sales
GROUP BY sale_date;

-- Optimizer rewrites it to:
SELECT sale_date, revenue
FROM daily_revenue_mv;
-- (because the materialized view covers exactly this aggregation)

This is called query rewriting or transparent rewriting. It means application code does not need to know that materialized views exist — the optimizer routes the query transparently. BigQuery, Oracle, and Snowflake support this; PostgreSQL does not (you must query the view by name).

Materialized Views in Practice

Dashboard Acceleration

A BI dashboard that shows "revenue by region by week" aggregates facts constantly. Instead of scanning the fact table on every page load:

CREATE MATERIALIZED VIEW weekly_revenue_by_region AS
SELECT
  d.year,
  d.week_number,
  s.region,
  SUM(f.sale_amount) AS revenue,
  COUNT(*)           AS transactions
FROM fact_sales f
JOIN dim_date  d ON f.date_key  = d.date_key
JOIN dim_store s ON f.store_key = s.store_key
GROUP BY d.year, d.week_number, s.region;

Refresh nightly. The dashboard reads millisecond-level queries instead of multi-second scans.

Pre-aggregation Layers

Data warehouses often layer multiple materialized views:

fact_sales (base, billions of rows)
  └── mv_daily_product_sales   (daily grain, millions of rows)
        └── mv_weekly_category (weekly grain, thousands of rows)
              └── mv_monthly_summary (monthly grain, hundreds of rows)

Upper layers build on lower layers, each refresh being faster because the input is already smaller.

Staleness and Correctness

The fundamental risk of materialized views is stale data. A dashboard showing last night's numbers as if they were live can mislead decision-makers. Mitigations:

  1. Display the refresh timestamp clearly in every dashboard using a materialized view
  2. Set user expectations — label dashboards "Updated hourly" or "As of 06:00 UTC"
  3. Alert on refresh failures — a stale view that hasn't refreshed in 48 hours is a bug, not a feature
  4. Use on-commit refresh selectively only for small, critical aggregates where staleness is unacceptable

Key Takeaways

  • A regular view is a stored query — recomputed every time it is queried with no performance benefit over writing the query inline.
  • A materialized view stores the precomputed result on disk; reads are fast because they scan a small, pre-aggregated table instead of billions of base rows.
  • Refresh strategies range from manual (lowest cost, maximum staleness) to on-commit (lowest staleness, highest write overhead) — incremental refresh balances both.
  • Query rewriting allows optimizers to transparently route queries to materialized views even when the user queries the base table.
  • Materialized views excel for expensive aggregations queried frequently — dashboard metrics, fixed KPI summaries, pre-aggregation layers.
  • The central trade-off is speed vs. freshness: always surface the refresh timestamp to users so they understand when the data was last updated.
  • Layered materialization (daily → weekly → monthly) amortizes refresh cost across multiple granularities.
