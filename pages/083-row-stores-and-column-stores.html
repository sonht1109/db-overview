Chapter 9. Pages, Records, and Files

Row stores and column stores

How a database lays out your data on disk turns out to matter enormously — not just for performance, but for the kinds of questions your database can answer efficiently. Two fundamentally different strategies have emerged: row stores (also called row-oriented or N-ary storage, NSM) and column stores (column-oriented or DSM — decomposition storage model). Most databases you encounter pick one, and understanding why helps you choose the right tool for the job.

How Row Stores Work

A row store keeps all the fields of a single row together on the same page. If you have a table of orders, the page on disk looks roughly like:

Page 42:
[ order_id=1001, customer='Alice', amount=99.00, status='shipped' ]
[ order_id=1002, customer='Bob',   amount=14.50, status='pending' ]
[ order_id=1003, customer='Carol', amount=230.75, status='shipped' ]
...

Reading a single row is fast: one disk read fetches everything about that row in one shot. This is great for OLTP (Online Transaction Processing) workloads — inserting a new order, updating a status, or fetching one customer's record. PostgreSQL, MySQL, and SQLite all use row storage.

How Column Stores Work

A column store flips the layout: all values for a single column are stored together, separate from other columns.

order_id column:  [ 1001, 1002, 1003, ... ]
customer column:  [ 'Alice', 'Bob', 'Carol', ... ]
amount column:    [ 99.00, 14.50, 230.75, ... ]
status column:    [ 'shipped', 'pending', 'shipped', ... ]

If you only need the amount and status columns to compute total revenue by status, the database reads only those two columns — it never touches order_id or customer. For wide tables with dozens of columns, this can cut I/O by an order of magnitude. Column stores also compress extremely well: similar values (like repeated status strings) pack tightly together. Databases like DuckDB, ClickHouse, Amazon Redshift, and Apache Parquet files use columnar layout.

Trade-offs at a Glance

Concern Row Store Column Store
Single-row read/write Fast (one I/O) Slow (touches many column files)
Aggregate over one column Slow (reads all columns) Fast (reads one column)
Compression Moderate High (column values are similar)
Typical workload OLTP (many small transactions) OLAP (few, large analytical queries)
Examples PostgreSQL, MySQL, SQLite DuckDB, ClickHouse, Redshift

Note: Many modern systems blur the line. PostgreSQL has columnar extensions (like pg_mooncake). Some "hybrid" engines (like SingleStore or SQL Server with columnstore indexes) maintain both layouts and pick at query time.

Seeing the Difference in Query Patterns

The SQL you write looks the same regardless of storage layout — but the engine's internal behavior differs drastically. Consider these two queries against a million-row sales table:

-- Query A: fetch one full order (row store wins)
SELECT * FROM sales WHERE order_id = 999;

-- Query B: aggregate one column across all rows (column store wins)
SELECT status, SUM(amount) FROM sales GROUP BY status;

Query A touches exactly one row — a row store finds it with one index lookup and one page read. A column store must stitch the row back together from multiple column segments.

Query B scans every amount and status value but never touches customer, address, or any other column. A column store reads only two narrow columns and can vectorize the arithmetic. A row store drags every column off disk even though most of it is irrelevant.
