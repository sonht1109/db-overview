Part V. Database Families

Chapter 19. Document Databases

Strengths and weaknesses

Every database family makes trade-offs. Document databases exchange relational power for document-centric simplicity — and that exchange is genuinely good in some situations and genuinely harmful in others. This page maps the strengths and weaknesses clearly so you can make an honest assessment for your use case.

Core Strengths

1. Natural Fit for Hierarchical, Self-Contained Entities

When a real-world entity is naturally a tree — an order with line items, a blog post with comments, a user profile with preferences — a document is the most direct representation. There is no impedance mismatch between the object in your code and the record in the database.

// A user profile maps directly to a programming language object
{
  "_id": "usr-441",
  "name": "Aroha Ngata",
  "address": { "city": "Auckland", "country": "NZ" },
  "preferences": { "theme": "dark", "language": "en" },
  "roles": ["editor", "reviewer"]
}

Compare this to a relational representation that might span users, user_addresses, user_preferences, and user_roles tables — four joins just to reconstruct the object.

2. Flexible Schema for Evolving Data

Adding a new field requires no ALTER TABLE, no downtime, and no migration script for existing documents. For a product catalog with 20 product types that have different attributes (a t-shirt has size and color; a digital game has platform and download_size), the flexible schema is a natural fit.

3. Horizontal Scalability

Document databases were largely designed for sharding from the start. Because a document is the natural unit of access and data, it is also the natural unit of distribution. Sharding on the document _id (or a shard key) partitions data across nodes without the cross-partition join problem that plagues relational sharding.

4. High Write Throughput for Independent Documents

Because each document is self-contained, concurrent writes to different documents never conflict. There are no foreign-key checks, no cascading updates, no join-induced lock contention. Systems that write many independent records — event logs, IoT sensor readings, clickstream data — can often achieve high throughput with document databases.

5. Developer Velocity

For many teams, the reduced ceremony of working without migrations is a genuine productivity win during early development. Frameworks like Mongoose (Node.js + MongoDB) make the start-to-first-query time very short.

Core Weaknesses

1. No Efficient Multi-Document Transactions (Historically)

For most of their history, document databases offered atomicity only at the single-document level. MongoDB added multi-document ACID transactions in version 4.0 (2018), but they come with significant performance overhead and are generally discouraged as a primary pattern.

The root issue: the document model assumes each document is an independent unit. When your business logic requires atomically updating two documents together (e.g., debit one account, credit another), you need either transactions (slow) or a redesign that embeds both accounts in one document (not always possible).

2. No Joins → Denormalization → Data Inconsistency Risk

Document databases do not natively join collections. Denormalization (embedding data that is also stored elsewhere) is the common workaround. But every embedded copy is a potential source of inconsistency. If a customer's email changes and it was embedded in 10,000 order documents, you must update all 10,000 — or live with stale copies.

The data integrity guarantee: Relational databases enforce referential integrity at the database level (FOREIGN KEY). Document databases enforce nothing — if you delete a customer and forget to update the orders that reference it, you have orphaned references with no database error.

3. Inefficient for Cross-Document Queries

"Find all orders placed by customers in Auckland" — in a relational database this is a two-table join with indexes. In a document database where customer data is referenced (not embedded), it requires loading all customer documents first or using $lookup (an application-level join without relational-engine optimizations). Cross-collection queries are expensive.

4. Schema Drift Over Time

Schema flexibility is a feature in the short term and a maintenance burden in the long term. Without discipline, collections accumulate inconsistent document shapes that break queries, confuse new engineers, and make refactoring painful. The relational constraint system that felt bureaucratic turns out to have been doing important work.

5. Aggregations Require a Pipeline

Complex aggregations (GROUP BY, windowed functions, multi-step transformations) require learning an aggregation pipeline API that is significantly more verbose than SQL:

// MongoDB aggregation equivalent of:
// SELECT customer_id, SUM(total) FROM orders GROUP BY customer_id HAVING SUM(total) > 500
db.orders.aggregate([
  { $group: { _id: "$customer_id", total: { $sum: "$total" } } },
  { $match: { total: { $gt: 500 } } }
])

SQL's expressive GROUP BY + HAVING is more readable for most developers. Analytics workloads generally don't fit the document model well.

Key Takeaways

  • Document databases excel when your data is naturally hierarchical, your access patterns are document-centric (load the whole thing at once), and your schema is evolving rapidly.
  • They struggle when you need cross-entity queries, multi-entity transactions, or strong referential integrity guarantees.
  • The lack of joins is not a bug — it is a deliberate trade-off that enables horizontal scalability and document-centric performance. But it becomes a problem when your data model is actually relational.
  • Schema flexibility is most valuable early in a project. Late in a project, the absence of enforcement is a maintenance liability.
  • Multi-document transactions exist in modern document databases but are an escape hatch, not a design target.
