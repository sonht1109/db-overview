Part III. Storage and Execution

Chapter 9. Pages, Records, and Files

Free space management

When a database deletes a row or updates a variable-length value to a shorter one, it does not immediately hand that space back to the operating system. Instead, the engine tracks the now-empty area internally so it can reuse it for future inserts. That bookkeeping is called free space management, and getting it right is the difference between a healthy database that stays compact and a bloated one that wastes disk and slows down every scan.

Why Gaps Appear

Pages (the fixed-size chunks your storage layer reads and writes) fill up row by row. Delete a row and you leave a hole: the bytes are free, but the page still occupies the same spot on disk. The same thing happens when you UPDATE a TEXT column from a long value to a short one — the leftover bytes sit unused inside the page.

Before delete:
┌──────────────────────────────────────────┐
│ [row 1][row 2][row 3][row 4]  free area  │
└──────────────────────────────────────────┘

After deleting row 2:
┌──────────────────────────────────────────┐
│ [row 1][     ][row 3][row 4]  free area  │
└──────────────────────────────────────────┘

If the engine ignores these holes, it will keep appending new rows to the "free area" at the end. Eventually the page is full of both live rows and holes — the engine allocates a fresh page even though plenty of bytes are wasted inside existing ones. Over time, a heavily-updated table bloats to several times its logical size.

How Engines Track Free Space

Free Space Maps

Most production engines maintain a free space map (FSM) — a compact index that records, for each page, roughly how many bytes are available. PostgreSQL, for example, keeps one FSM file per table. When an insert arrives, the engine consults the FSM to find a page with enough room, writes the row there, and updates the FSM entry. SQLite uses a similar structure called the freelist: a linked chain of pages that have been freed entirely (by dropping a table or deleting all rows from a page). Partial free space within a page is tracked in each page's header.

Approach Granularity Used by
Free space map (per-page byte count) Fraction of a page PostgreSQL
Freelist (whole free pages) Whole page SQLite, InnoDB
Page-level free-space percentage Rounded bucket SQL Server (PFS pages)

Fragmentation and Compaction

Even with a free space map, internal fragmentation builds up. Many small holes scattered across many pages means no single hole is big enough for a large row, even though the total free bytes would be sufficient. Engines handle this in two ways:

  • In-page compaction (slot shifting): When a page is written, the engine slides live rows together and merges the holes at the end. This is cheap and happens automatically.
  • Table-level vacuuming / reorganization: A background process rewrites pages, discarding deleted rows entirely and reclaiming space. PostgreSQL calls this VACUUM; SQL Server calls it a rebuild or reorganize of the heap/index; SQLite exposes it as VACUUM.

Note: SQLite's VACUUM rewrites the entire database file into a new, compact file and then replaces the old one. It reclaims space from dropped tables and deleted rows but requires enough disk space to hold two copies of the database at once during the operation.

The Cost of Ignoring Free Space

Neglecting free space management has cascading effects:

  1. Bloated files — the database file grows without bound even as logical data stays flat.
  2. Slower full scans — a sequential scan reads every page, including ones that are mostly empty.
  3. Index bloat — B-tree index pages suffer the same fragmentation; a bloated index means more I/O per lookup.
  4. Stale statistics — many planners estimate row counts from page counts. A bloated table makes the optimizer think there is more data than there is, leading to bad plans.

Scheduling regular VACUUM (or equivalent) runs — ideally during low-traffic windows — is one of the simplest operational habits that keeps a database healthy over months and years.
