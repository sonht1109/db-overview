Part VI. Engineering Practice

Chapter 25. Security and Access Control

Encryption in transit and at rest

Data has two vulnerable moments: when it travels across a network and when it sits on a storage device. Encryption in transit protects data as it moves between clients and the database server; encryption at rest protects data that lives on disk. Both layers are essential, and neither substitutes for the other — a wire-tapper can't read an encrypted connection, and a stolen hard drive is useless if the data is encrypted at rest.

Encryption in Transit: TLS

Transport Layer Security (TLS) is the standard for encrypting database connections. Without it, anyone on the same network segment can intercept SQL queries, result sets, and authentication credentials in plain text.

How TLS Works for Databases

  1. The client connects to the database server.
  2. The server presents an X.509 certificate signed by a trusted CA.
  3. They negotiate a cipher suite and establish a session key.
  4. All subsequent traffic — queries, results, error messages — is encrypted.

Most modern databases default to requiring TLS or at least offering it as an option. In PostgreSQL, the pg_hba.conf hostssl record forces TLS:

# pg_hba.conf — require TLS for all remote connections
hostssl  all  all  0.0.0.0/0  scram-sha-256

Mutual TLS (mTLS)

Standard TLS authenticates only the server. Mutual TLS also authenticates the client via a client certificate, so both sides prove their identity. This is common in database-as-a-service environments and microservice meshes.

# PostgreSQL: require client certificate
hostssl  all  app_user  0.0.0.0/0  cert

Verifying Your Connection

After enabling TLS, verify that connections actually use it:

-- PostgreSQL: check connection encryption
SELECT ssl, version, cipher, bits FROM pg_stat_ssl
WHERE pid = pg_backend_pid();

Never disable certificate verification. Drivers often have a sslmode=require (encrypt but don't verify the certificate) option. Use sslmode=verify-full to also validate the server's hostname — otherwise you're vulnerable to man-in-the-middle attacks.

PostgreSQL server-side config: TLS is configured via ssl_cert_file (server certificate), ssl_key_file (private key), and ssl_ca_file (CA certificate for verifying client certs) in postgresql.conf. On the client side, sslmode=verify-full requires the server certificate's Common Name (CN) to match the hostname used in the connection string; a mismatch will cause the connection to fail.

Encryption at Rest

Encryption at rest protects data stored on disk — data files, WAL/redo logs, temporary files, and backups. Without it, a stolen disk or misconfigured cloud storage bucket exposes all your data.

Transparent Data Encryption (TDE)

Most enterprise and cloud databases implement Transparent Data Encryption: the database engine encrypts data pages before writing them to disk and decrypts them on read. The application sees no difference; the encryption is handled below the SQL layer.

Database TDE support
PostgreSQL No native TDE; use pgcrypto for column-level encryption or file-system encryption (LUKS, ZFS native encryption) for at-rest protection
MySQL / MariaDB InnoDB tablespace encryption (enabled per-table or globally)
SQL Server Full TDE since 2008
Oracle Advanced Security TDE
AWS RDS / Aurora AES-256 at rest via AWS KMS, enabled at instance creation 
-- MySQL: enable encryption for a table
ALTER TABLE sensitive_data ENCRYPTION = 'Y';

-- MySQL: check encryption status
SELECT table_name, create_options FROM information_schema.tables
WHERE table_schema = 'mydb' AND create_options LIKE '%ENCRYPTION=Y%';

Encryption Granularity

At-rest encryption can operate at different levels:

  1. Full-disk encryption (FDE) — the OS or hardware encrypts the entire volume. Protects against physical theft; the database process itself runs with unencrypted data in memory.
  2. File-system encryption — the file system layer (e.g., Linux LUKS, ZFS encryption) encrypts before writing to disk.
  3. Database-level TDE — the database engine encrypts pages. More portable; key management is explicit.
  4. Column-level / application-level encryption — the application encrypts specific fields before storing them. Strongest isolation; the database itself never sees plaintext. Used for PII, PAN data, credentials.

Key Management

Encryption is only as strong as its key management. Keys must be:

  • Stored separately from the encrypted data (never on the same disk)
  • Rotatable without downtime where possible
  • Access-controlled so only authorized services can decrypt

Cloud key management services (AWS KMS, Google Cloud KMS, HashiCorp Vault) provide hardware security module (HSM)-backed key storage with audit logs for every key access.

# AWS: create a KMS key for database encryption
aws kms create-key --description "RDS encryption key" \
  --key-usage ENCRYPT_DECRYPT

Backup encryption: Encrypting the live database is not enough if backups are stored in plaintext S3. Ensure backup encryption is enabled and that backup keys are independently managed.

Column-Level Encryption

For especially sensitive fields — social security numbers, payment card data, health records — column-level encryption adds a layer that even DBAs cannot bypass:

-- PostgreSQL pgcrypto: encrypt before insert
INSERT INTO patients (id, name, ssn_encrypted)
VALUES (1, 'Alice', pgp_sym_encrypt('123-45-6789', 'secret_passphrase'));

-- Decrypt only where authorized
SELECT id, name, pgp_sym_decrypt(ssn_encrypted, 'secret_passphrase') AS ssn
FROM patients WHERE id = 1;

The downside: encrypted columns cannot be efficiently indexed or used in WHERE clauses (you can't range-scan ciphertext). This is an intentional trade-off — if you need the data, you decrypt it explicitly.

Key Takeaways

  • TLS in transit is non-negotiable. Use verify-full mode — not just require — so clients validate the server's certificate.
  • TDE at rest protects against physical theft and storage misconfigurations; most cloud databases enable it by default.
  • Keep encryption keys separate from encrypted data and rotate them on a schedule. Use a managed KMS rather than rolling your own.
  • For the most sensitive data, use column-level encryption: the database stores only ciphertext, and only code with the decryption key can access the plaintext.
  • Ensure backups are also encrypted — a plaintext backup negates all on-disk encryption.
