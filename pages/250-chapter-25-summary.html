Part VI. Engineering Practice

Chapter 25. Security and Access Control

Chapter 25 Summary: What We Learned

Page Topic Takeaway in One Sentence
241 Authentication and authorization Authentication verifies identity (who you are); authorization controls access (what you can do). Both are essential.
242 Roles and permissions Group permissions into roles (read-only, read-write, admin); grant roles to users — simpler to audit and manage.
243 Encryption in transit and at rest TLS/SSL for data in transit; transparent data encryption (TDE) or filesystem encryption for data at rest.
244 Auditing and logging Log who accessed what, when, and what changed — essential for compliance (GDPR, SOC 2) and forensic investigation.
245 Secrets management Never hardcode credentials; use a secrets manager (Vault, AWS Secrets Manager) and rotate credentials regularly.
246 Multi-tenant isolation Separate databases, separate schemas, or row-level security — each level trades isolation strength for operational simplicity.
247 SQL injection and safe queries Use parameterized queries or an ORM. Never concatenate user input into SQL strings. The number one database security vulnerability.
248 Privacy and data retention Minimize data collection; set retention policies; implement soft-delete and anonymization for GDPR/CCPA compliance.
249 Secure database habits Apply least-privilege access, update regularly, restrict network access, and review access logs. Security is a process, not a checkbox.
