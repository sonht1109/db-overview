Part VI. Engineering Practice

Chapter 25. Security and Access Control

Privacy and data retention

Databases routinely accumulate more personal data than they need to, for longer than they are permitted to keep it. Privacy is the practice of collecting and storing only the data that is necessary, protecting it appropriately, and giving individuals meaningful control over it. Data retention is the practice of deleting or anonymizing data when it is no longer needed or permitted. Both are increasingly non-optional: GDPR, CCPA, HIPAA, and similar laws impose strict obligations — and significant penalties — on organizations that handle personal data carelessly.

Privacy Principles in Database Design

Data Minimization

Collect only the personal data that is strictly necessary for the stated purpose. Every field you store is a field that can be breached, subpoenaed, or disclosed by accident. Before adding a column:

  • What business purpose does this serve?
  • Is this the minimum data that satisfies that purpose?
  • How long is this data needed?

A date of birth might be needed to verify an age threshold. The full date is more than necessary — just a boolean is_over_18 or the year alone may suffice.

Purpose Limitation

Data collected for one purpose should not be repurposed for another without explicit consent. A common database smell: an events table collected for usage analytics that slowly becomes the source of truth for user profile data, fraud detection, and marketing — all without the user's knowledge.

Data Accuracy

Personal data that is incorrect can cause real harm (denial of services, financial errors, legal problems). Provide mechanisms for users to correct their data, and don't over-denormalize personal fields — a single authoritative location is easier to correct than copies scattered across ten tables.

Implementing Right to Erasure ("Right to be Forgotten")

GDPR Article 17 grants users the right to have their personal data deleted. In a relational database, this is harder than it sounds:

  • The user's email address may be in users, email_campaigns, audit_logs, orders, support_tickets, and a dozen other tables.
  • Foreign key constraints may prevent deletion of parent records while children exist.
  • Backup tapes and data warehouse snapshots may contain the data even after it is deleted from the live database.

A practical approach uses pseudonymization rather than deletion for records that are referenced by other data:

-- Replace PII with anonymized values; preserve referential integrity
UPDATE users
SET
  email         = 'deleted-' || id || '@anonymized.invalid',
  full_name     = 'Deleted User',
  phone         = NULL,
  date_of_birth = NULL,
  deleted_at    = strftime('%Y-%m-%dT%H:%M:%S', 'now')
WHERE id = :user_id;

The user's orders and financial records (needed for accounting and legal compliance) remain intact and linked by user_id, but the personal identifiers are gone.

Tracking Erasure Requests

Maintain a table that records each erasure request and its status — regulators can ask you to prove that you honored the request:

CREATE TABLE erasure_requests (
  id           SERIAL PRIMARY KEY,
  user_id      INTEGER NOT NULL,
  requested_at TEXT NOT NULL,
  completed_at TEXT,
  status       TEXT NOT NULL DEFAULT 'pending'
);

Data Retention Policies

A retention policy defines: for each category of data, how long it is kept, and what happens to it afterward. Common retention periods:

Data category Typical retention Regulation
User PII (names, emails) Duration of account + 30 days post-deletion GDPR
Financial transaction records 7 years Tax law (varies by country)
Audit logs 1–7 years SOC 2, PCI-DSS
Server / application logs 90 days Common practice; some regs require longer
Health records 7–10 years (adults) HIPAA
Marketing consent records 3 years after consent GDPR recital 171

Enforcing Retention with Scheduled Deletions

-- Soft-deleted accounts older than 30 days: anonymize PII
UPDATE users
SET email = 'anon-' || id || '@deleted.invalid',
    full_name = 'Deleted',
    phone = NULL
WHERE deleted_at IS NOT NULL
  AND deleted_at < strftime('%Y-%m-%dT%H:%M:%S', 'now', '-30 days');

-- Purge log rows older than retention period
DELETE FROM app_logs
WHERE created_at < strftime('%Y-%m-%dT%H:%M:%S', 'now', '-90 days');

Run these as scheduled jobs (cron, pg_cron, cloud scheduler). Make them idempotent so a failed run can be re-executed without harm.

Data Masking for Non-Production Environments

A significant percentage of data breaches involve development, staging, or analytics environments — which often receive production data copies without the same controls. Data masking replaces PII with realistic but fake values before populating non-production databases:

Field Original Masked
email alice@example.com anon_4921@test.invalid
phone +1-555-234-5678 +1-555-000-0001
SSN 123-45-6789 XXX-XX-6789
credit card 4111-1111-1111-1111 4000-XXXX-XXXX-0000

Tools like faker libraries, Delphix, or AWS DMS data transformation rules automate masking during database refresh.

Key Takeaways

  • Minimize collection: every personal data field is a liability — collect only what you need, store it only as long as you need it.
  • Implement pseudonymization on deletion: nullify or replace PII when an account is deleted, but keep the record shell for referential integrity and legal records.
  • Define and enforce retention policies per data category: automate the cleanup with scheduled jobs so it actually runs.
  • Mask PII before copying to non-production environments — staging and dev databases are a common breach vector.
  • Maintain an erasure request log to demonstrate compliance; regulators expect proof that you honored deletion requests.
