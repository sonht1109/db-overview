Part II. Querying and Language

Chapter 6. How Query Processing Works

Parsing and validation

Every time you send a SQL query to a database engine, it goes through several stages before a single row of data is touched. The very first two stages — parsing and validation — turn your raw text into something the engine can reason about and check for correctness. Understanding them helps you read error messages, write better SQL, and build an accurate mental model of how the engine works.

Stage 1: Parsing — From Text to Tree

A SQL query arrives as a plain string of characters. The parser's job is to verify that the string follows SQL's grammar rules and, if it does, convert it into a structured representation called an abstract syntax tree (AST).

The AST is an in-memory tree where each node represents a syntactic element:

SELECT name, salary FROM employees WHERE salary > 50000

Produces a tree roughly like:

SelectStatement
├── Columns: [name, salary]
├── From: employees
└── Where: BinaryOp(>)
         ├── Left: salary
         └── Right: 50000

The parser does not know whether the table employees exists, or whether salary is a real column — that comes next. Its only concern is grammar. If you write SELECT FROM WHERE name, it fails here because the structure violates SQL syntax rules, regardless of what tables you have.

Note: Databases typically implement their parser as a hand-written or generated tokenizer + recursive descent parser. The tokenizer breaks the string into tokens (SELECT, name, ,, FROM, …), then the parser builds the tree according to the language grammar.

What Causes a Parse Error?

Parse errors come from broken grammar:

Broken query Why the parser rejects it
SELEC name FROM t Unrecognized keyword
SELECT name FROM Missing table reference after FROM
SELECT name, FROM t Trailing comma before a keyword
SELECT (name FROM t Unclosed parenthesis

The error message usually names the unexpected token and its position, which is why database errors often say something like syntax error near "FROM".

Stage 2: Validation — Does This Query Make Sense?

Once the AST exists, the engine passes it to the semantic analyzer (often called the validator or binder). This stage checks that the meaning of the query is valid against the actual schema stored in the catalog (the database's internal directory of tables, columns, types, and constraints).

Three things are checked:

  1. Object existence — Does the table employees exist in this database?
  2. Column resolution — Does salary exist as a column in employees? If a query references multiple tables, which table does an ambiguous column name belong to?
  3. Type compatibility — Is comparing salary > 50000 valid given salary's declared data type? Would salary > 'hello' cause a type mismatch?

If any check fails, the query is rejected before any data is read — a useful guarantee that saves wasted I/O.

Note: The catalog is also called the system catalog or data dictionary. In SQLite you can inspect it directly via sqlite_schema. In PostgreSQL it lives in pg_catalog. It is just a set of tables that the engine itself queries internally during validation.

Seeing Validation Errors in Practice

A parse error and a validation error feel similar from the outside, but they come from different stages:

-- Parse error: broken syntax (no table given)
SELECT name FROM;

-- Validation error: table does not exist (syntax is fine)
SELECT name FROM ghost_table;

-- Validation error: column does not exist
SELECT ghost_column FROM employees;

From Validated AST to Execution

After parsing and validation succeed, the engine has a clean, fully-resolved AST — every name is confirmed to refer to a real object, and types are consistent. This validated AST is what gets handed off to the query optimizer, which we will cover on the next page.
