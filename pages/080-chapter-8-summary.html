Part II. Querying and Language

Chapter 8. Joins, Aggregation, and Sorting

Chapter 8 Summary: What We Learned

Chapter Contents

Page Topic Takeaway in One Sentence
071 Nested loop joins Simple and always available: for each row in the outer table, scan the inner table for matches. Excellent with a small outer + indexed inner.
072 Hash joins Build a hash table from one side, probe with the other. Best for large, unsorted inputs when equality join is needed.
073 Merge joins Requires both inputs sorted on the join key; walks both sorted streams in parallel. Excels for large, pre-sorted data.
074 Grouping and aggregation The engine can hash-group or sort-group; hash grouping is faster when groups fit in memory.
075 External sort When data exceeds memory, the engine uses multi-pass merge sort with temporary files — this is why sorting can be slow.
076 Memory limits and spill to disk Every operator has a memory budget; when exceeded, data spills to disk, causing a dramatic performance drop.
077 Join order matters Joining A then B vs B then A can differ by orders of magnitude — the optimizer's job is to find the best order.
078 Cardinality and row counts Accurate cardinality estimates are critical: wrong estimates cause the optimizer to choose the wrong join order or algorithm.
079 Practical query tuning Start with EXPLAIN, look for scans where you expected index seeks, check actual vs estimated rows, and add indexes selectively.
