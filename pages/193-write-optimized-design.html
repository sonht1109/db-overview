Part V. Database Families

Chapter 20. Column-Family Databases

Write-optimized design

Column-family databases like Apache Cassandra are not just "NoSQL with a different API" — they are engineered from the ground up around one constraint: writes must be as fast as possible, even at petabyte scale. This shapes every design decision, from the on-disk format to the way updates and deletes work. This page explains the write-optimized architecture: why in-place updates are slow, what the append-only model looks like, and how data flows from a client write all the way to durable storage.

Why In-Place Updates Are Slow

In a traditional B-tree-based database (PostgreSQL, MySQL, SQLite), an update works roughly like this:

  1. Find the page on disk that contains the row.
  2. Lock the page.
  3. Modify the byte range in-place.
  4. Write the change to the WAL.
  5. Mark the page dirty in the buffer pool.

Step 1 is the bottleneck at scale. On a spinning disk (HDD), finding a page requires a seek — physically moving the read head to the right track. A typical HDD seek takes 5–10 milliseconds. At 200 updates/second per spinning disk, you're already near the physical limit. Even with SSDs (where there is no moving head), random writes cause write amplification: the SSD's flash translation layer must erase and rewrite entire flash blocks (128KB–1MB) to change a few bytes, wearing out the device faster and reducing throughput.

The fundamental problem: in-place updates require random I/O, and random I/O is expensive whether you're on HDD or SSD.

The Append-Only Alternative

Column-family systems sidestep random I/O by making writes always sequential and append-only:

  • A new value never overwrites the old one.
  • Instead, the new value is appended as a newer version with a higher timestamp.
  • Old versions are cleaned up later, in the background, during compaction.

This means writes are fast because sequential I/O to an append-only log saturates disk bandwidth, rather than crawling through random seeks. A single SSD can sustain hundreds of thousands of sequential writes per second — orders of magnitude more than random writes.

Key insight: "Update" in Cassandra is not a modification — it is an insert of a new version. The system resolves which version is current at read time (or during compaction) using timestamps.

The Write Path: Commit Log + Memtable

Every write in Cassandra follows a two-step path before it is acknowledged to the client:

Client
  │
  ├──► Commit Log (WAL) ── sequential append to disk ── durable immediately
  │
  └──► Memtable ── in-memory sorted data structure ── fast, not yet on disk

  1. Commit log (Write-Ahead Log): The write is appended to an on-disk sequential log file. This is what makes the write durable — if the node crashes before the memtable is flushed, the commit log can replay all writes on restart.

  2. Memtable: The write is also inserted into an in-memory sorted structure (a memtable — think of it as an in-memory SSTable). Reads can be served from the memtable before data ever hits a permanent SSTable file.

Once the write is safely in both the commit log and the memtable, Cassandra acknowledges success to the client. The entire operation involves zero random disk I/O — the commit log is sequential, and the memtable is in RAM.

The Memtable Flush Cycle

The memtable is bounded in size (configurable, typically 64MB–256MB per table). When it fills:

  1. The memtable is frozen — no more writes go to it.
  2. A new, empty memtable takes over immediately. Writes continue without interruption.
  3. The frozen memtable is flushed to disk as an immutable SSTable (Sorted String Table) file.
  4. Once the SSTable is fully written, the corresponding portion of the commit log can be discarded (since the data is now safely on disk).

This cycle repeats continuously. The result: a steady stream of sequential disk writes, and an accumulation of SSTable files.

What "No In-Place Update" Means for Reads and Deletes

The append-only model has some non-obvious consequences:

Updates are new versions. If you write (user_id=42, name='Alice') and then write (user_id=42, name='Alice Smith'), both versions exist somewhere — possibly in different SSTables or the memtable. A read for user_id=42 must check the memtable and potentially multiple SSTables, then take the most recent version by timestamp. This is why reads in Cassandra can be slower than writes — they may require merging multiple sources.

Deletes are "tombstones." You cannot erase a byte in an immutable SSTable. Instead, a delete writes a special marker called a tombstone — a record saying "this key was deleted at time T". On a read, if a tombstone is found that is newer than any actual value, the key is treated as deleted. Tombstones are physically removed only when compaction runs.

Compaction resolves versions. Compaction is the background process that reads multiple SSTables, merges them, keeps only the most recent version of each cell, discards tombstoned rows, and writes a new, clean SSTable. After compaction, reads are faster because fewer files need to be consulted.

Comparing B-Tree vs LSM-Tree Write Paths

Characteristic B-Tree (PostgreSQL, MySQL) LSM-Tree (Cassandra, HBase)
Write mechanism In-place update in page Append to commit log + memtable
Disk I/O type Random (seek to page) Sequential (append-only)
Write durability WAL + in-place Commit log
Update cost Seek + lock + modify Always O(1) append
Read cost Single B-tree traversal Merge memtable + N SSTables
Delete Mark as deleted (vacuum later) Write tombstone
Background work Vacuum / autovacuum Compaction
Write amplification Moderate (WAL + heap) High (compaction rewrites data)
Read amplification Low (one path through tree) Higher (multiple SSTables)

The fundamental trade-off: LSM-trees optimize write throughput by making reads slightly more expensive. B-trees optimize read performance by making writes involve random I/O. Choose based on your workload's write-to-read ratio.

Why This Matters for Time-Series and Event Workloads

Column-family databases excel at workloads where:

  • Data is append-only by nature (events, logs, sensor readings, audit trails).
  • Rows are never updated — only new readings are inserted.
  • Write rate vastly exceeds read rate.
  • Reads are always by partition key (e.g., "give me all events for device X in the last hour").

For these patterns, the write-optimized design is not a compromise — it is the ideal architecture. A single Cassandra node can sustain tens of thousands of writes per second. A cluster of dozens of nodes can sustain millions.

Key Takeaways

  • In-place updates are slow because they require random I/O — seeking to the right page on disk before modifying it.
  • Cassandra's write path is always sequential: commit log (WAL) + in-memory memtable. No random seeks on write.
  • Every write is an append, never an overwrite. Updates create new versions with higher timestamps.
  • Deletes write tombstones — logical markers, not physical erasure.
  • The memtable flushes to an immutable SSTable when full; background compaction merges SSTables and removes stale versions.
  • This design makes writes extremely fast at the cost of slightly more complex reads.
  • Best fit: high-volume append workloads — events, IoT sensors, time-series, logs — where write throughput is the primary constraint.
