Part VII. Building Understanding

Chapter 29. Common Myths About Databases

Chapter 29 Summary: What We Learned

Page Topic Takeaway in One Sentence
281 SQL is slow SQL is not slow — bad queries, missing indexes, and poor schema design are slow. Well-tuned SQL on a good schema is extremely fast.
282 NoSQL means no schema NoSQL databases have a schema — it is implicit in the application code and harder to enforce, not absent.
283 Indexes always help Indexes help reads but hurt writes. Adding an index to every column guarantees slow writes and bloated storage.
284 ACID means no scale Modern relational databases (CockroachDB, Spanner, Aurora) scale horizontally while providing ACID — the myth is outdated.
285 Distributed means faster Distribution adds network hops and coordination overhead. A distributed system can be slower than a well-tuned single-node database for moderate workloads.
286 Flexible schema means easy design Flexible schema means you defer design decisions to application code, where they become harder to track, enforce, and refactor.
287 Denormalization always improves performance Denormalization improves read performance at the cost of write complexity and data consistency risk. Use it surgically.
288 One database can do everything Every database optimizes for a specific tradeoff. Knowing what it does poorly is as important as knowing what it does well.
289 What to believe instead Understand the tradeoffs yourself. Every database choice is a set of tradeoffs — the right answer depends on your specific workload.
Reveal: How do these ideas connect to the next chapter?
Chapter 29 — and this book — ends here. But your database journey is just beginning. You now have a mental framework that spans the full stack: from how data is organized in tables, through how queries are optimized and executed, down to the physical storage layer, across distributed systems, and through the operational realities of performance, reliability, and security. The next chapter is not in this book — it is in the database you design, the query you optimize, and the system you keep running. Take everything you have learned and apply it.
