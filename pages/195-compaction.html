Chapter 20. Column-Family Databases

Compaction

Without compaction, an LSM-tree-based database would eventually collapse under its own weight. SSTables would pile up indefinitely on disk, reads would slow to a crawl as they checked more and more files, and space would fill up with stale values and phantom deletes. Compaction is the background process that keeps all of this in check: it merges SSTables, eliminates superseded versions, and physically removes deleted records.

Why Compaction Exists

Consider what happens after a few hours of writes in a write-heavy system:

  • Dozens or hundreds of L0 SSTables accumulate, all with overlapping key ranges.
  • Every UPDATE creates a new entry — the old value still occupies disk space.
  • Every DELETE writes a tombstone marker — the underlying data still exists.
  • Bloom filters help, but they cannot eliminate the overhead of checking 50 files where 5 would do.

Compaction is the answer: it reads a set of SSTables, merges their contents using a merge-sort algorithm, writes a new set of compacted SSTables, and discards the originals.

Before compaction (L0):
  SST-1: [a:v1, c:v3, e:TOMBSTONE]
  SST-2: [a:v2, b:v1, c:v4]
  SST-3: [b:TOMBSTONE, d:v1]

After compaction (L1):
  SST-A: [a:v2, b:TOMBSTONE, c:v4, d:v1]
  (e was tombstoned and had no older version → removed entirely)
  (b tombstone stays until lower levels are checked)

Note: Tombstones are not immediately removed by every compaction pass. A tombstone can only be dropped when compaction is certain no lower level still holds the old value. Until then, the tombstone must be preserved — otherwise deleted data would "resurface."

Size-Tiered Compaction (STCS)

Size-tiered compaction is the simpler of the two dominant strategies and is the default in Apache Cassandra for write-heavy tables.

How It Works

  1. SSTables are grouped into size tiers (e.g., files of similar size).
  2. When enough files accumulate in a tier (typically 4), they are merged into a single larger file.
  3. That larger file joins the next tier, and the process repeats.
Tier File size (approx.) Trigger
Tier 1 ~50 MB 4 files → merge into 1
Tier 2 ~200 MB 4 files → merge into 1
Tier 3 ~800 MB 4 files → merge into 1
Tier 4 ~3.2 GB 4 files → merge into 1

Trade-offs

Strengths:

  • Minimal write amplification — each byte is rewritten fewer times.
  • Excellent write throughput; SSTables are merged infrequently.

Weaknesses:

  • Files within a tier can overlap in key range, so reads may still check multiple files.
  • Temporarily requires up to 2× disk space (old files + new merged file coexist).
  • Tombstones may persist for a long time if the triggering merge is delayed.

Leveled Compaction (LCS)

Leveled compaction takes the opposite philosophy: prioritize read performance by ensuring that within each level (L1 and below), SSTables have non-overlapping key ranges. This is the strategy used in LevelDB, RocksDB, and optionally in Cassandra.

How It Works

  1. L0 SSTables are merged with L1 SSTables. The result is sorted and split into fixed-size files (e.g., 10 MB each) with non-overlapping ranges.
  2. When L1 grows beyond its size limit, files are promoted into L2 — merged with existing L2 files, non-overlapping maintained.
  3. Each level is roughly 10× larger than the one above.
L0: [a-z, a-z, a-z]  ← overlapping (just flushed)
L1: [a-d][e-h][i-m][n-z]  ← non-overlapping, 10 MB each
L2: [a-b][c-d][e-f]...[y-z]  ← non-overlapping, 10 MB each, ~10× more files

Trade-offs

Strengths:

  • At most one file per level contains the target key → much faster reads.
  • Tombstones are removed promptly because all data for a key range passes through compaction regularly.
  • Space amplification is lower (~10% overhead vs 50–100% for STCS).

Weaknesses:

  • Higher write amplification — the same data is rewritten many times as it migrates through levels.
  • Higher I/O pressure on write-heavy workloads.

Tombstones: Deletes as Writes

In an LSM-tree, you cannot delete a record by erasing it — the SSTable is immutable. Instead, a delete writes a tombstone: a special marker that says "this key was deleted at sequence N."

-- Application issues: DELETE FROM sensor_data WHERE id = 42;
-- LSM-tree writes:    [key=42, value=TOMBSTONE, seq=9901]
-- Old data in SST:   [key=42, value=71.3, seq=8801]  ← still on disk
-- After compaction:  key=42 disappears entirely

Tombstones carry a risk: if compaction is delayed and tombstones accumulate, tombstone-heavy reads become very slow — the database must wade through many tombstones to find the live data. In Cassandra this is known as "tombstone overload" and can trigger TombstoneOverwhelmingException.

Operational rule: Monitor tombstone counts in Cassandra. If SELECT queries produce warnings about tombstones, your data model likely relies on too many deletes. Consider using TTL instead — TTL expiry is handled more efficiently than explicit tombstones.

The Compaction I/O Tax

Compaction is not free. While it runs, it:

  1. Reads all source SSTables from disk.
  2. Performs an in-memory merge sort.
  3. Writes the result back to disk.
  4. Deletes the source SSTables.

This means compaction competes with your application's I/O. Under heavy write load, compaction may fall behind — L0 accumulates files faster than compaction can drain it. This is called compaction debt and results in degraded read performance.

Tuning levers:

  • compaction_throughput_mb_per_sec (Cassandra) — throttle compaction to limit I/O impact.
  • Number of compaction threads — more parallelism helps; too many starve foreground I/O.
  • min_sstable_size and sstable_size_in_mb — control trigger thresholds.

STCS vs. LCS: Choosing a Strategy

Criterion Size-Tiered (STCS) Leveled (LCS)
Best for Write-heavy, time-series, append-only Read-heavy, low-latency lookups
Read performance Worse (overlapping files) Better (one file per level)
Write amplification Lower Higher
Space overhead Up to 2× during compaction ~10% steady-state
Tombstone cleanup Slower Faster
Default in Cassandra (most tables) RocksDB, LevelDB

Key Takeaways

  • Compaction is what keeps LSM-trees healthy: without it, reads degrade, disk fills up with stale data, and tombstones accumulate.
  • Size-tiered compaction (STCS) optimizes for write throughput at the cost of read performance — ideal for append-heavy workloads.
  • Leveled compaction (LCS) optimizes for read performance and space efficiency at the cost of write amplification — ideal for read-heavy workloads.
  • Tombstones are how deletes work in LSM-trees: the old data persists until compaction physically removes it. Excessive tombstones are a serious operational hazard.
  • Compaction consumes real I/O; running behind on compaction is called compaction debt and can severely impact read latency.
  • Choosing and tuning the right compaction strategy is one of the most impactful operational decisions when running Cassandra, RocksDB, or any LSM-backed database.
