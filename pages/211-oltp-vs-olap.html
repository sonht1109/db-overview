Chapter 22. Analytical Databases and Warehouses

OLTP vs OLAP

Every production system eventually faces the same tension: the database that keeps your application running second-by-second is rarely the right tool for the queries that tell you why it ran the way it did. OLTP (Online Transaction Processing) and OLAP (Online Analytical Processing) are the two poles of this spectrum — and understanding what separates them is the foundation for every decision in database architecture, from index design to the choice of a data warehouse.

What OLTP Means in Practice

OLTP systems are the heartbeat of operational software. Every time a user places an order, transfers money, or updates a profile, an OLTP database records that event — atomically, durably, and fast. The defining characteristic is many short transactions hitting individual rows, often thousands per second.

A typical OLTP transaction looks like this:

BEGIN;
  -- Point lookup on a single order
  SELECT id, status, amount
  FROM orders
  WHERE id = 10042;

  -- Update exactly one row
  UPDATE orders
  SET status = 'shipped'
  WHERE id = 10042;
COMMIT;

Notice what this query does not do: it does not scan millions of rows, compute averages, or group results. It touches one row with a known primary key, and it does so in milliseconds.

OLTP Characteristics

  • Query pattern: Point lookups and short-range scans via indexed columns.
  • Data volume touched per query: Typically one to a few hundred rows.
  • Latency target: Single-digit milliseconds; sub-millisecond for cached paths.
  • Concurrency: Hundreds to thousands of simultaneous transactions.
  • Write intensity: High — every user action is a write.
  • Schema style: Third Normal Form (3NF) to minimize redundancy and prevent update anomalies.
  • Primary optimization target: Write throughput and low-latency point reads.

The 3NF schema is not an accident. When you process thousands of inserts per second, you cannot afford to write the same customer name in fifty order rows every time it changes. Normalizing into separate customers, products, and orders tables means each piece of data lives in exactly one place.

What OLAP Means in Practice

OLAP systems are built for understanding data, not recording it. A business analyst asking "What was our revenue by region last quarter, broken down by product category?" is issuing an OLAP query. It touches millions of rows, computes aggregations, and might run for minutes.

-- OLAP query: aggregate over millions of rows
SELECT
  DATE_TRUNC('month', created_at) AS month,
  product_id,
  SUM(amount)                     AS total_revenue,
  COUNT(*)                        AS order_count
FROM orders
WHERE created_at >= '2024-01-01'
  AND created_at <  '2025-01-01'
GROUP BY 1, 2
ORDER BY 1, total_revenue DESC;

This query might scan every order from the past year. On a row-oriented OLTP database with 50 million rows, this can take tens of seconds even with optimal indexes — because indexes are designed for point lookups, not full-column aggregations.

OLAP Characteristics

  • Query pattern: Full or partial table scans with GROUP BY, aggregations, and multi-table joins.
  • Data volume touched per query: Millions to billions of rows.
  • Latency target: Seconds to minutes are acceptable; humans tolerate longer waits for insight.
  • Concurrency: Low — a handful of analysts, not thousands of application users.
  • Write intensity: Low — data is typically loaded in bulk (ETL/ELT pipelines), not updated row by row.
  • Schema style: Star schema or snowflake schema — denormalized for fast reads.
  • Primary optimization target: Scan throughput and aggregation speed.

The star schema denormalizes deliberately. A fact_orders table surrounded by dim_customers, dim_products, and dim_dates dimension tables means an analyst can answer almost any business question with a single join — at the cost of some data redundancy.

The Fundamental Tension

Why can't one database do both well? The answer lies in the physical storage layout.

Row-oriented storage (used by OLTP engines) keeps all columns of a row together on disk:

[id=1, customer_id=42, product_id=7, amount=99.00, status='paid', created_at='2024-03-01']
[id=2, customer_id=17, product_id=3, amount=12.50, status='shipped', created_at='2024-03-01']
...

When you fetch order #10042 by primary key, you read exactly one disk page and get every column you need. Perfect for OLTP.

But when you want SUM(amount) across 50 million rows, you must read every single row — pulling id, customer_id, product_id, status, and created_at into memory even though you only care about amount. You pay the I/O cost of all columns to aggregate one.

Column-oriented storage (used by OLAP engines) keeps each column contiguous:

id column:          [1, 2, 3, 4, 5, ...]
customer_id column: [42, 17, 88, 42, 7, ...]
amount column:      [99.00, 12.50, 45.00, 230.00, 8.99, ...]

Now SUM(amount) reads only the amount column — a fraction of the data. It also compresses beautifully (sorted integers compress 10–50×). But updating a single row means touching a location in every column file, which is expensive and complex under high concurrency.

The core tradeoff: Row storage writes one row atomically and reads it back in one I/O. Column storage reads one attribute across all rows in minimal I/O but pays dearly for single-row writes. You cannot optimize both simultaneously.

This is not merely a tuning dial. It is a fundamental architectural divide — which is why the industry long ago split into separate systems for each workload.

Schema Design Reflects the Workload

The schema style is not decoration — it is a direct consequence of the access pattern.

OLTP: Third Normal Form

In an OLTP system, each fact is stored once. The orders table references customers and products by foreign key rather than embedding their attributes:

-- OLTP schema (3NF)
CREATE TABLE customers (
  id         SERIAL PRIMARY KEY,
  name       TEXT NOT NULL,
  email      TEXT UNIQUE NOT NULL,
  country    TEXT NOT NULL
);

CREATE TABLE products (
  id         SERIAL PRIMARY KEY,
  name       TEXT NOT NULL,
  category   TEXT NOT NULL,
  unit_price NUMERIC(10,2) NOT NULL
);

CREATE TABLE orders (
  id          SERIAL PRIMARY KEY,
  customer_id INTEGER REFERENCES customers(id),
  product_id  INTEGER REFERENCES products(id),
  amount      NUMERIC(10,2) NOT NULL,
  status      TEXT NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT now()
);

Updating a customer's email touches exactly one row. There are no anomalies.

OLAP: Star Schema

In an OLAP warehouse, you pre-join the data into a fact table surrounded by dimension tables. Redundancy is intentional:

-- OLAP star schema
CREATE TABLE fact_orders (
  order_id     BIGINT,
  customer_key BIGINT,   -- FK to dim_customers
  product_key  BIGINT,   -- FK to dim_products
  date_key     INTEGER,  -- FK to dim_date
  amount       NUMERIC(10,2),
  quantity     INTEGER
);

CREATE TABLE dim_customers (
  customer_key BIGINT PRIMARY KEY,
  name         TEXT,
  country      TEXT,
  region       TEXT,
  segment      TEXT      -- e.g. 'Enterprise', 'SMB'
);

CREATE TABLE dim_products (
  product_key  BIGINT PRIMARY KEY,
  product_name TEXT,
  category     TEXT,
  subcategory  TEXT,
  brand        TEXT
);

An analyst can now answer "revenue by country and product category" with a single join from fact_orders — no multi-hop join chain through normalized tables.

Why Mixing Workloads Breaks Databases

Running OLAP queries against an OLTP database causes two classes of pain:

1. Lock Contention and Latency Spikes

A long-running analytic query holds shared locks (or MVCC snapshots) on large swaths of data. Even with MVCC, the query inflates the undo/version chain, increasing memory pressure and checkpoint lag. Concurrent OLTP transactions slow down because the buffer pool fills with cold analytic pages, evicting hot operational data.

2. Index Misuse

OLTP indexes are B-trees designed for point lookups. An OLAP query doing a full-table aggregate cannot use a B-tree index efficiently — it triggers an index scan or falls back to a sequential scan anyway. Now you have the worst of both: slow analytic queries and a B-tree maintenance overhead that taxes every OLTP write.

Industry response: The dominant solution is to separate the systems. An OLTP database captures writes in real time; an ELT pipeline (Fivetran, dbt, Airbyte) replicates data into an OLAP warehouse (Snowflake, BigQuery, Redshift, DuckDB) on a schedule. Analysts query the warehouse; the application never touches it.

Some databases offer a middle ground — HTAP (Hybrid Transactional/Analytical Processing). TiDB and SingleStore maintain both a row store and a columnar replica of the same data, routing queries to the appropriate engine. This adds significant complexity but can eliminate the ETL pipeline for use cases where near-real-time analytics matter.

Key Takeaways

  • OLTP handles the operational heartbeat of your application: many short, concurrent transactions that read or write individual rows with strict latency requirements. Normalized (3NF) schemas and B-tree indexes are the right tools.

  • OLAP handles analytical insight: few, long-running queries that scan and aggregate millions of rows. Denormalized star schemas, columnar storage, and vectorized execution engines are the right tools.

  • The fundamental tension is physical: row-oriented storage optimizes single-row access; column-oriented storage optimizes per-attribute scans. No single layout excels at both simultaneously.

  • Lock contention and buffer pool thrashing make it dangerous to run heavy analytic queries against a live OLTP database. Even with MVCC, long transactions degrade operational performance.

  • The standard architecture is separation: OLTP database → ETL/ELT pipeline → OLAP warehouse. HTAP systems (TiDB, SingleStore) blur this boundary at the cost of operational complexity.

  • When you encounter a performance problem in a database, asking "is this OLTP or OLAP traffic?" is almost always the right first question — because the answer determines every subsequent design choice.
