Part II. Querying and Language

Chapter 8. Joins, Aggregation, and Sorting

Join order matters

When you write a multi-table query, the database engine has to decide how to combine those tables internally — which table to read first, which to probe into next, and what algorithm to use at each step. This sequence of decisions is called the join order, and it can be the difference between a query that finishes in milliseconds and one that crawls for minutes on the exact same data.

Why the sequence matters

Every join produces an intermediate result set — a temporary "virtual table" in memory (or on disk if it's large). If you join a huge table early and a small table late, you carry that heavy intermediate result through every subsequent step. Flip the order and the intermediate results stay small the whole way through.

Consider three tables:

Table Rows
orders 10,000,000
customers 500,000
promo_codes 50

Worst order: orders JOIN customers → 10 M intermediate rows → JOIN promo_codes
Better order: promo_codes JOIN orders → ~50 matching rows → JOIN customers

The data hasn't changed. The query hasn't changed. Only the join order changed — and the engine touches millions fewer rows.

What the optimizer does

Modern database engines include a query optimizer that analyzes your SQL before executing it. It considers statistics (row counts, value distributions, index availability) and picks the join order it estimates will be cheapest. You don't have to write tables in a magic order; the optimizer will often rearrange them.

Note: The optimizer works from statistics, not live data. If your statistics are stale — because a table grew 10× since the last ANALYZE — the optimizer can choose a bad plan. Keeping statistics fresh is a routine DBA task.

The optimizer's plan is visible via EXPLAIN (or EXPLAIN ANALYZE in PostgreSQL). This is the single most useful debugging tool for slow queries.

When the optimizer gets it wrong

The optimizer can be fooled by:

  • Stale statistics — table grew but ANALYZE hasn't run.
  • Data skew — a column value appears in 90 % of rows; average-based estimates are off.
  • Complex predicates — functions on columns (WHERE YEAR(created_at) = 2024) prevent index use and confuse cardinality estimates.
  • Too many tables — join-order search space is exponential; optimizers prune it heuristically, sometimes badly.

In these cases you can use hints (supported by MySQL, Oracle, SQL Server) or rewrite the query to force a better plan.

Key takeaways

  • Join order determines how large intermediate result sets grow; smaller intermediate sets mean less work.
  • The query optimizer rewrites join order automatically using table statistics.
  • EXPLAIN / EXPLAIN QUERY PLAN shows you the plan the engine actually chose — always check it when a query is slow.
  • Keep statistics current (ANALYZE, UPDATE STATISTICS, etc.) so the optimizer can make good decisions.
  • When the optimizer gets it wrong, you can guide it with index hints or by restructuring subqueries, but that should be a last resort after understanding why it chose a bad plan.
