Part VI. Engineering Practice

Chapter 23. Designing for Performance

Picking the right indexes

An index is a trade-off, not a free win. Every index you add speeds up the queries that use it and slows down every write to the table. Picking the right indexes means matching index type and column order to the specific query patterns your application needs — not indexing everything and hoping for the best.

How Indexes Work (Briefly)

A B-tree index stores column values in sorted order with pointers back to heap rows. Lookups are O(log n) — far better than a sequential scan for large tables. But the tree must be updated on every INSERT, UPDATE (if the indexed column changes), and DELETE. A table with ten B-tree indexes pays ten tree-update costs per write.

An index scan is fast when it is selective — when the indexed condition filters most rows out. If a query returns 40 % of the table, the planner will often prefer a sequential scan over an index scan because random I/O to follow index pointers is more expensive than reading pages sequentially.

Selectivity: The Most Overlooked Factor

Selectivity = fraction of rows returned by the predicate. High selectivity = few rows returned = index is valuable.

Column Distinct values Selectivity Index useful?
order_id (PK) 10 million ~0.000001 Yes — point lookup
status (3 values) 3 ~0.33 Usually no — low selectivity
email ~10 million ~0.000001 Yes — login lookup
country ~200 ~0.005 Maybe — depends on distribution

Rule: Do not index low-cardinality columns in isolation. If status has only three values, indexing it alone is rarely worth the write overhead — the planner will do a sequential scan anyway. Combine it in a composite index where the leading column is selective.

Composite Indexes and Column Order

A composite index on (a, b, c) can serve queries that filter on a, on (a, b), or on (a, b, c). It cannot efficiently serve a query that filters only on b or only on c — the left-most prefix rule.

-- Composite index: (status, created_at)
CREATE INDEX idx_orders_status_date ON orders(status, created_at);

-- Uses the index (leading column present)
SELECT * FROM orders WHERE status = 'pending' AND created_at > '2024-01-01';

-- Uses only the first column of the index
SELECT * FROM orders WHERE status = 'shipped';

-- Does NOT efficiently use this index (skips leading column)
SELECT * FROM orders WHERE created_at > '2024-01-01';

Put the equality column first, range column second. This lets the engine seek directly to the matching status value and then scan only the relevant date range within that partition of the index.

Covering Indexes

A covering index includes all columns the query needs — so the engine never touches the heap. This is the fastest possible index lookup because it eliminates the second I/O (the "heap fetch" or "bookmark lookup").

-- If this query runs millions of times per day:
SELECT customer_id, total FROM orders WHERE status = 'pending';

-- A covering index makes it heap-free:
CREATE INDEX idx_orders_pending_cover ON orders(status, customer_id, total);

Partial Indexes

A partial index indexes only rows matching a condition. Smaller index, faster lookups, lower write overhead:

-- Only index unprocessed events — the hot minority
CREATE INDEX idx_events_unprocessed ON events(created_at)
WHERE processed = 0;

Indexes to Avoid

Anti-pattern Problem
Index on every column Writes pay N update costs; planner confusion
Index on a boolean (true/false) Only 2 values — selectivity near 50 %, rarely useful
Duplicate indexes Same columns in same order — pure overhead
Unused indexes Write cost with no read benefit; check pg_stat_user_indexes
Function-wrapped column in WHERE WHERE LOWER(email) = ? can't use an index on email without a function index

Key takeaways: Match index type to query pattern. Put equality predicates before range predicates in composite indexes. Use covering indexes for hot queries. Use partial indexes to exclude cold rows. Audit unused indexes regularly — they are pure write overhead.
