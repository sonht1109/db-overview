Part I. Foundations

Chapter 3. The Relational Model

Chapter 3 Summary: What We Learned

Page Topic Takeaway in One Sentence
021 Relations, tuples, and attributes A relation is a mathematical set — no duplicate rows, no ordering, just facts with named attributes.
022 Primary keys and foreign keys Primary keys uniquely identify rows; foreign keys reference them, enforcing referential integrity across tables.
023 Set-based thinking SQL operates on sets, not individual rows — filter, transform, and combine entire collections in one statement.
024 Projection, selection, and join Projection (SELECT columns), selection (WHERE rows), and join (linking tables) are the three fundamental relational operations.
025 Why relations are powerful The closed-world assumption means every fact is stored once, and any answer derivable from those facts is expressible as a query.
026 Data independence Change storage layout, add indexes, or upgrade hardware — your SQL queries stay the same.
027 Integrity by design Constraints and foreign keys guarantee data stays consistent at the database level, not in bug-prone application code.
028 Common misunderstandings NULL handling, three-valued logic, and the difference between a table and a relation trip up even experienced developers.
029 Where the relational model shines Ad-hoc queries, strong integrity, data independence, and a mature ecosystem make relational the safest default.
Reveal: How do these ideas connect to the next chapter?
Chapter 3 gave you the relational model's rules: sets, keys, foreign keys, constraints, and the closed-world assumption. Those rules are the theory. Chapter 4 is where theory meets practice. You will learn how to translate real-world entities and relationships into a concrete schema — which tables to create, how to name them, when to use a join table, and how to evolve a schema safely as requirements change. The relational model tells you what is possible; data modeling tells you what is wise.
