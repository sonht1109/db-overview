Part IV. Distributed Systems

Chapter 14. From One Machine to Many

Failure becomes normal

On a single machine, a hardware failure is a crisis — the whole system is down. Spread your database across dozens or hundreds of machines and the math changes completely: at that scale, something is almost always broken. A disk dies, a network switch reboots, a power supply flickers, a kernel panic takes out one node at 3 a.m. Distributed databases are built on the recognition that partial failure is the default state, not an edge case to handle someday.

Why Failure Is Inevitable at Scale

Consider a back-of-the-envelope calculation. If a single server has 99.9% uptime (about 8.7 hours of downtime per year), a cluster of 100 such servers will see at least one node unavailable roughly 9.5% of the time — nearly one day in ten. Real hardware failure rates, network partitions, and software bugs make the true number higher.

Cluster size Prob. all nodes healthy Expected outages/year on one node
1 node 99.9% ~8.7 hours
10 nodes 99.0% at least one node down ~87 hours
100 nodes 90.5% constantly managing incidents

Note: These numbers assume fully independent failure probabilities, which is optimistic. Shared power, shared network racks, and correlated software deployments mean real clusters see correlated failures — multiple nodes going down for the same reason at the same time.

The implication is stark: a distributed database that requires all nodes to be healthy before it can serve requests is not a real distributed database. It must make progress even when some nodes are unavailable.

The Failure Taxonomy

Not all failures look the same, and a good distributed system must reason about each kind differently.

Crash-stop (fail-stop): The node halts and sends no further messages. This is the easiest failure to reason about — silence means dead. Most crash recovery protocols assume this model.

Network partition: The node is alive and healthy, but network links are severed. It can still serve local reads and accept writes — it just cannot talk to the rest of the cluster. This is far more common than outright crashes, and far more dangerous, because the isolated node and the main cluster may both believe they are the authoritative copy.

Byzantine failure: The node behaves arbitrarily — sending stale data, returning wrong answers, or replying to some nodes but not others. This is the hardest model to handle and requires protocols (like Practical Byzantine Fault Tolerance) that assume up to ⌊(n−1)/3⌋ traitors. Most databases in trusted data-center environments skip Byzantine tolerance entirely and focus on crash and partition failures.

What "Handling" Failure Means

A distributed database cannot simply prevent failures. Instead it must detect them and respond without stalling the whole system. The two core tools are:

Timeouts and heartbeats. Nodes send periodic "I'm alive" pings. If a heartbeat is missed for longer than a threshold (say, 10 seconds), the coordinator marks the node suspected failed and stops routing requests to it. The tricky part: a slow network looks identical to a dead node from the outside, so the timeout threshold is always a trade-off between false positives (declaring a slow node dead) and slow detection (waiting too long to give up).

Replication as the survival strategy. The reason a single-node outage doesn't bring down the whole cluster is that data is replicated across multiple nodes. If node A holds copies of data that node B and C also hold, then A's failure is survivable — B and C can continue serving reads and accepts writes. How many simultaneous failures a cluster can tolerate depends on its replication factor and the quorum rules it uses (covered in the next pages).

Note: Replication and fault tolerance come with a catch: you now have multiple copies of data, and those copies can disagree. Every distributed database has to decide what to do when two nodes have conflicting versions of the same row — a question that leads directly into consistency models and the CAP theorem, coming up soon.
