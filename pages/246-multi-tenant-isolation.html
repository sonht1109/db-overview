Part VI. Engineering Practice

Chapter 25. Security and Access Control

Multi-tenant isolation

When a single database instance serves multiple customers — or multiple teams, environments, or organizational units — each group's data must be completely invisible to the others. This is multi-tenant isolation: ensuring that tenant A cannot read, modify, or even discover tenant B's data, even if both are authenticated to the same database. Getting this wrong does not just violate a policy; it violates the trust of every customer whose data leaks.

The Three Isolation Models

There is a fundamental trade-off in multi-tenant database design between isolation strength and resource efficiency. Three models sit at different points on that spectrum:

Model Isolation mechanism Shared resources Typical use case
Database per tenant Separate databases Server, OS, connection pool (partially) High-compliance SaaS, enterprise on-premise
Schema per tenant Separate schemas in one database One database instance Mid-market SaaS
Shared schema with tenant column Application + RLS policy Tables, indexes, storage High-scale SaaS with many tenants

Database per Tenant

Each tenant gets their own database. Maximum isolation: a bug or a compromised application cannot cross tenant boundaries at the database level. Drawbacks: resource overhead grows linearly with tenant count; schema migrations must be applied to every database; connection pooling is more complex.

-- PostgreSQL: create a completely separate database per tenant
CREATE DATABASE tenant_acme;
CREATE DATABASE tenant_globex;
-- Each database has its own roles, tables, and data

Schema per Tenant

All tenants share one database, but each gets their own schema. The schema name acts as a namespace:

CREATE SCHEMA tenant_acme;
CREATE SCHEMA tenant_globex;

-- Same table structure, isolated data
CREATE TABLE tenant_acme.orders (...);
CREATE TABLE tenant_globex.orders (...);

-- Set search_path per connection to restrict access
SET search_path = tenant_acme;
SELECT * FROM orders;  -- sees only acme's orders

Schema-per-tenant is popular because migrations can be applied programmatically across all schemas, and isolation is enforced at the schema permission level.

Shared Schema with Row-Level Security

The most resource-efficient model: one set of tables, every row tagged with a tenant_id column, and database-enforced row-level security policies that automatically filter every query.

-- Shared orders table with tenant tag
CREATE TABLE orders (
  id          SERIAL PRIMARY KEY,
  tenant_id   INTEGER NOT NULL,
  amount      NUMERIC,
  status      TEXT
);

-- Enable RLS
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders FORCE ROW LEVEL SECURITY;

-- Policy: only see your own tenant's rows
CREATE POLICY tenant_isolation ON orders
  USING (tenant_id = current_setting('app.tenant_id')::integer);

-- Application sets this at session start:
SELECT set_config('app.tenant_id', '42', false);

After this policy is in place, SELECT * FROM orders always returns only the current tenant's rows — regardless of what the application sends. A developer bug that omits a WHERE tenant_id = ? clause is automatically corrected by the database.

Common Pitfalls

Forgetting FORCE ROW LEVEL SECURITY

By default, table owners bypass RLS. If the application connects as the table owner, policies are ignored. Always use FORCE ROW LEVEL SECURITY on tenant-shared tables, or connect as a non-owner role:

ALTER TABLE orders FORCE ROW LEVEL SECURITY;

Tenant ID Injection

If the application sets app.tenant_id via SQL, a malicious or buggy application could set it to a different tenant's ID. Mitigate by:

  1. Setting the tenant ID immediately after connection, in the connection pool checkout hook
  2. Using a separate role per tenant in the schema-per-tenant model
  3. Treating tenant_id as a server-side session attribute, not a client-controlled variable

Cross-Tenant Joins

In a shared-schema model, a badly written query can still join across tenants if RLS is not applied uniformly. Enable RLS on every table that holds tenant-scoped data, not just the primary table.

Backup Isolation

Backups of a shared database contain all tenants' data. Consider whether your tenants' contracts require the ability to produce per-tenant exports or perform per-tenant restores.

Key Takeaways

  • Choose an isolation model based on tenant count, compliance requirements, and operational capacity: database per tenant for highest isolation, shared schema with RLS for highest density.
  • Row-level security is the most efficient mechanism for shared-schema multi-tenancy, but requires FORCE ROW LEVEL SECURITY and RLS policies on every tenant-scoped table.
  • Set the tenant_id session variable on the server side at connection time — never trust client-supplied values.
  • Apply RLS uniformly: a missed table in a shared schema is a cross-tenant data leak waiting to happen.
  • Plan for per-tenant backup and restore: even if data is physically co-located, tenants expect logical isolation at every level.
