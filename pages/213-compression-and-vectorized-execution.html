Chapter 22. Analytical Databases and Warehouses

Compression and vectorized execution

Columnar storage gives analytical databases their storage layout advantage. But layout alone is only half the story. The other half is what happens after the data is read off disk: compression shrinks the bytes that must move through the memory hierarchy, and vectorized execution changes the fundamental rhythm of how the CPU processes those bytes. Together they are why a modern analytical engine can scan billions of rows per second on commodity hardware.

Why Columnar Data Compresses So Well

Row-oriented data mixes values of different types and meanings on the same page — an order row might hold a timestamp, a customer ID, a status string, a decimal amount, and a country code all interleaved. There is little statistical regularity to exploit.

A column file is the opposite: every byte in the file is the same logical attribute — all country values, or all amount values. This homogeneity creates three compression-friendly properties:

Property What it means Example
Homogeneous types Every value in the column shares the same data type and encoding width A SMALLINT column needs exactly 2 bytes per value — no padding, no type tags
Sorted values If the column is sorted (or clustered), long runs of identical or near-identical values appear After sorting by country, all US rows group together
Low cardinality Many real-world categorical columns have a tiny set of distinct values repeated millions of times A status column with only PENDING, PAID, REFUNDED

These properties make generic compression (gzip, zstd) already more effective on column files than on row files. But analytical engines go further: they apply data-aware compression schemes that understand the column's semantics and can be operated on without full decompression.

Compression Techniques

Run-Length Encoding (RLE)

Run-length encoding replaces a sequence of repeated values with a single (value, count) pair.

Raw column (8 values):    [US, US, US, CA, CA, CA, CA, MX]
RLE encoded (3 pairs):    [(US, 3), (CA, 4), (MX, 1)]

Storage drops from 8 entries to 3. More importantly, an aggregate like COUNT(*) WHERE country = 'CA' can be answered by reading the count field of the matching pair — without decompressing or scanning individual rows at all. This is called operating directly on compressed data.

RLE is most effective when the column is sorted; a sort on country before storing turns scattered occurrences into contiguous runs.

Dictionary Encoding

Dictionary encoding maps each distinct value to a small integer, then stores the integers instead of the original strings.

Dictionary:   {US → 0, CA → 1, MX → 2, BR → 3}
Raw column:   [US, US, CA, MX, US, BR, CA, CA]
Encoded:      [ 0,  0,  1,  2,  0,  3,  1,  1]

A country name like "United States" might take 13 bytes as a string. Its dictionary code takes 2 bits if there are fewer than 4 countries, or 1 byte for up to 256. Equality filters (WHERE country = 'CA') translate into integer comparisons, which are faster than string comparisons and can be vectorized trivially.

Most columnar engines (Parquet, DuckDB, Redshift) apply dictionary encoding by default for string columns with fewer than a few thousand distinct values.

Delta Encoding

Delta encoding stores the difference between consecutive values instead of the values themselves. It shines on monotonically increasing sequences like timestamps, IDs, or prices.

Raw column:   [1000, 1003, 1007, 1008, 1015, 1020]
Deltas:       [1000,    3,    4,    1,    7,    5]

Deltas are much smaller numbers, so they need fewer bits. A timestamp column spanning one day of data has raw values in the billions; the per-row deltas are typically under 1000 ms, fitting in 10 bits instead of 32.

Delta encoding is often combined with bit-packing for maximum effect.

Bit-Packing

Bit-packing stores integers using only as many bits as the data actually requires, instead of the fixed 32 or 64 bits a standard integer type would use.

Max value in column: 200   →   needs only 8 bits (2^8 = 256)
Standard storage:    200 values × 32 bits = 6 400 bits
Bit-packed storage:  200 values ×  8 bits = 1 600 bits   (4× smaller)

After dictionary encoding reduces a string column to small integers, bit-packing compresses those integers further. A column with 100 distinct values only needs 7 bits per entry. Engines like Apache Arrow and DuckDB use bit-packed representations as their default in-memory column format.

What Vectorized Execution Means

Compression controls how much data must be processed. Vectorized execution controls how efficiently the CPU processes it.

To understand vectorized execution, it helps to first see what it replaced.

The Volcano Model (Row-at-a-Time)

The classic database execution model — sometimes called the Volcano model or iterator model — structures query execution as a tree of operators, each implementing a next() method that returns one row at a time to its parent.

ProjectionOperator.next()
  └─ FilterOperator.next()
       └─ ScanOperator.next()   ← reads one row from disk

When the query engine needs results, it calls Projection.next(). Projection calls Filter.next(). Filter calls Scan.next(), which reads one row. That row bubbles up through the chain: Filter checks the predicate, then Projection selects columns, and the result row is returned. Then the whole process repeats for the next row.

This is elegantly composable — you can chain any operators together — but it has a severe performance cost: one function call per row, per operator. For a scan of 100 million rows through 3 operators, that is 300 million virtual function calls. Function call overhead, branch misprediction, and cache thrashing add up quickly.

Note: The Volcano model was designed in the early 1990s when disk I/O dominated query time. Today, analytical workloads keep data in memory, and CPU efficiency is the bottleneck. Row-at-a-time is the wrong granularity.

The Vectorized Model (Batch-at-a-Time)

Vectorized execution changes the unit of work from one row to a fixed-size batch — typically 1,024 or 2,048 values — called a vector or chunk.

ProjectionOperator.next_batch()   ← returns 1024 rows at once
  └─ FilterOperator.next_batch()
       └─ ScanOperator.next_batch()   ← reads 1024 rows from disk

Each operator now does real work — processing 1024 values — for every function call. The function-call overhead becomes negligible compared to the data-processing work. Within each operator, the inner loop over the 1024 values is tight enough for the CPU's branch predictor and instruction pipeline to stay warm.

More importantly, modern CPUs include SIMD instructions (Single Instruction, Multiple Data) that can operate on 4, 8, or 16 values simultaneously in a single CPU instruction. A tight loop over a vector of integers can be compiled to AVX-512 instructions that process 16 int32 values per clock cycle. The Volcano model's row-at-a-time loop cannot take advantage of SIMD — the work per iteration is too irregular.

Attribute Volcano (row-at-a-time) Vectorized (batch-at-a-time)
Function calls One per row per operator One per batch per operator
SIMD utilization Very low — irregular per-row work High — tight inner loops over uniform data
CPU cache behavior Poor — operator state evicted between calls Good — vector fits in L1/L2 cache
Code complexity Simple Moderate — operators must handle batch semantics

Practical Batch Sizes

The batch size (1,024 is common in DuckDB and Velox) is chosen so that the vector fits entirely in the CPU's L1 or L2 cache. An L1 cache is typically 32–64 KB; 1,024 int32 values take 4 KB. Keeping the active data in L1 cache means memory accesses take ~4 cycles instead of ~200 cycles for a main-memory access.

Why Compression and Vectorization Compound

Compression and vectorized execution are not independent optimizations — they multiply each other's benefits.

Consider a columnar table with 1 billion rows of sales data. The country column, dictionary-encoded and bit-packed to 8 bits per entry, takes roughly 1 GB. Uncompressed it would take 8–13 GB (as a variable-length string). A 10× compression ratio means:

  • 10× more data fits in L3 cache — the CPU can process data at cache bandwidth (~500 GB/s) instead of memory bandwidth (~50 GB/s).
  • 10× more values fit in each vector — a 4 KB vector now holds 4,096 compressed values instead of 400 uncompressed strings.
  • SIMD operates on compressed codes — integer comparison on 8-bit dictionary codes is faster than string comparison, and SIMD can compare 32 of them per instruction.

The engine can often skip decompression entirely for filter predicates: a WHERE country = 'CA' translates to WHERE dict_code = 1, and the bit-packed integer comparison is faster than the original string comparison.

Key insight: A query that reads 100 GB of uncompressed data might only read 8 GB after columnar compression. Vectorized execution then processes those 8 GB at full SIMD throughput. The combined effect is not additive — it is multiplicative. This is why DuckDB can aggregate a 1-billion-row TPC-H dataset in seconds on a laptop.

Compression Ratios in Practice

Real-world compression ratios vary by column characteristics:

Column type Typical cardinality Technique Compression ratio
Status / enum (PENDING, PAID) 2–10 Dictionary + bit-pack 10–40×
Country code ~250 Dictionary + 8-bit 4–13×
Timestamp (sorted) High, monotonic Delta + bit-pack 4–8×
Numeric amounts High, varied Bit-pack or generic 1.5–3×
Free-text strings Very high Generic (zstd) 2–4×

The sweet spot for both compression and vectorization is low-cardinality, homogeneous categorical columns — exactly the kind of columns that dominate dimension tables and fact table foreign keys in analytical schemas.

Key Takeaways

  • Columnar data compresses well because storing homogeneous values in sequence creates statistical regularity: repeated values, monotonic sequences, and small cardinalities.
  • RLE replaces runs of identical values with (value, count) pairs and enables aggregate queries to operate directly on compressed data without decompression.
  • Dictionary encoding maps strings to small integers, enabling fast equality filters and dramatically reducing storage for low-cardinality columns.
  • Delta encoding and bit-packing shrink numeric columns by storing only what varies and using only the bits required.
  • The Volcano model (row-at-a-time) incurs millions of function calls and cannot use SIMD; vectorized execution amortizes overhead across fixed-size batches (typically 1,024 values) that fit in CPU cache and enable SIMD parallelism.
  • Compression and vectorization multiply each other's benefits: compressed data fits more values per cache line, and SIMD operates on compact integer codes faster than on raw strings.
  • The combination is why modern analytical engines (DuckDB, Snowflake, BigQuery) can scan billions of rows per second on modest hardware — the CPU spends nearly all its time doing useful arithmetic, not shuffling bytes or dispatching function calls.
