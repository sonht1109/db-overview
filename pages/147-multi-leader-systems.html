Part IV. Distributed Systems

Chapter 15. Replication

Multi-leader systems

Leader-follower replication has one hard constraint: every write must pass through a single node. That leader becomes a bottleneck — and a single point of failure for writes. Multi-leader replication (also called multi-master or active-active replication) lifts this constraint by letting two or more nodes each accept writes, then propagate their changes to every other leader.

When Multi-Leader Makes Sense

Multi-leader is uncommon in single-datacenter setups because the complexity rarely pays off. It shines in two scenarios:

Multi-datacenter deployments — Place one leader in each datacenter. Users in each region write to their local leader (low latency). Leaders sync with each other asynchronously across the WAN. If a datacenter goes offline, the others keep accepting writes.

Offline-capable clients — Every device (phone, laptop) acts as its own "leader" with a local database. Changes sync whenever connectivity is restored. CouchDB and the replication layer behind apps like Google Docs use this model.

The Write Conflict Problem

The central challenge in multi-leader replication is that two leaders can independently accept different writes to the same data at the same time. When those changes propagate, the system must decide which one wins — or how to merge them.

Example: Two users simultaneously edit a wiki page title. Leader A records "Introduction", Leader B records "Overview". After replication, both leaders have two conflicting versions with no obvious winner.

Conflict Avoidance

The simplest strategy is to route all writes for a given record through the same leader — no writes to the same row ever happen on different leaders, so no conflict arises. Many multi-leader deployments rely on this: route each user's writes to their "home" datacenter consistently.

Last-Write-Wins (LWW)

Each write is timestamped. The write with the higher timestamp survives; the other is silently discarded. It is simple to implement and used by Cassandra and optional Dynamo configurations. The downside is real: writes can vanish with no error, and clocks across nodes are rarely perfectly synchronized.

Application-Level Merge

The database stores both conflicting versions and passes them to the application to resolve. CouchDB does this — your app receives both values and must produce one merged result (like a version-control merge). It is the most flexible approach, but it pushes complexity into application code.

CRDT-Based Merging

Conflict-free Replicated Data Types (CRDTs) are data structures mathematically designed so that any two replicas can always be merged without conflicts. A grow-only counter, for example, merges by summing each node's increment. Riak and some collaborative editing systems use CRDTs internally so the application never sees a conflict at all.

The tradeoff landscape:

Strategy Data loss risk Complexity Example systems
Conflict avoidance None (by design) Low Sharded multi-leader setups
Last-write-wins Yes — silently Low Cassandra, Dynamo
App-level merge None High (on app) CouchDB, Riak
CRDTs None High (in DB) Riak, Ditto

Replication Topologies

In a multi-leader system with more than two leaders, you must also decide how writes flow between nodes. Three common topologies:

  • All-to-all — every leader sends its changes directly to every other leader. Most resilient, but requires deduplication to avoid applying the same change twice.
  • Star — one designated node forwards writes from all others to all others. Single forwarding bottleneck, simpler routing.
  • Circular — each node forwards writes to the next in a ring. One broken node can interrupt the ring.

Note: All-to-all topologies can suffer from causality violations: an update can arrive before the insert it depends on, if the insert took a slower route. Systems guard against this with version vectors (logical clocks that track which writes each node has seen).
