Part V. Database Families

Chapter 18. Key-Value Databases

Real-world examples

Theory is clean; production is messier and more interesting. Key-value databases power some of the highest-traffic systems on earth — and understanding which system engineers reach for, and why, makes the abstractions from earlier pages suddenly concrete.

The Big Four in Production

Four names dominate the key-value landscape. Each makes a different trade-off on the same fundamental model.

System Primary Use Where Values Live Notable Feature
Redis Cache, session store, pub/sub RAM (persisted to disk optionally) Rich value types (lists, sorted sets, streams)
Memcached Pure cache layer RAM only — no persistence Simpler than Redis; scales horizontally by adding nodes
Amazon DynamoDB Durable application data at cloud scale SSD (fully managed) Single-digit-millisecond latency; serverless billing
etcd Distributed configuration & service discovery SSD (Raft-replicated) Strong consistency; used as Kubernetes' brain

Each occupies a distinct niche. Redis is the default choice when you need speed plus some structure in the values. Memcached is the choice when you need a pure, stateless cache with nothing extra. DynamoDB is the choice when you need a key-value store that your ops team never has to touch. etcd is the choice when correctness matters more than throughput — it is optimized for configuration data that a cluster stakes its life on.

Redis: The Swiss Army Key-Value Store

Redis is the first tool most teams reach for. At its core it is still a key-value store — every command addresses a key — but it extends the value side with six distinct types:

String  → SET user:4291:name "Alice"          # plain bytes or numbers
List    → RPUSH notifications:4291 "msg1"     # ordered list, push/pop both ends
Hash    → HSET user:4291 email a@example.com  # mini key-value inside a key
Set     → SADD online_users 4291              # unordered, no duplicates
ZSet    → ZADD leaderboard 9800 user:4291     # sorted set with a numeric score
Stream  → XADD events * action purchase       # append-only log with consumer groups

The killer feature is that all operations are atomic. INCR rate:user:4291:hour increments an integer counter in one round-trip with no race condition. ZADD leaderboard keeps a scoreboard sorted at all times without a SELECT … ORDER BY query. These primitives let application code stay simple while the data structure does the heavy lifting.

Note: Redis is single-threaded for command execution. That sounds like a limitation, but it is the reason every command is inherently atomic — there is no need for locks. This design choice lets Redis sustain millions of operations per second on a single node.

A Caching Layer in Practice

The most common Redis deployment sits between your application and your relational database as a read-through cache. The idea: store the result of expensive queries under a predictable key; serve subsequent requests from RAM without touching the database.

The application code for this pattern is typically fewer than ten lines — check the cache, populate it on a miss, serve subsequent requests from RAM. A five-minute TTL on product data means stale data is bounded and the database sees a fraction of the traffic it would otherwise handle.

DynamoDB: Key-Value at Cloud Scale

While Redis is typically self-hosted (or managed via ElastiCache), Amazon DynamoDB is a fully managed key-value and document store where you provision throughput, not servers. Every table has a partition key (the key in "key-value") and an optional sort key. Reading by exact partition key is the primary fast path, exactly as in any key-value store.

Teams building on AWS use DynamoDB for things like:

  • User profiles — userId as partition key; one item per user.
  • Game state — gameId → entire serialized game state blob.
  • Shopping carts — userId as partition key, itemId as sort key.

The trade-off: querying outside the key requires either a full scan (expensive) or a secondary index (you must declare it up front). The key-value access pattern is the happy path; anything else is a signal to reconsider the data model.

etcd: Key-Value for Infrastructure

etcd is a specialized key-value store optimized for one thing: storing small, critical configuration data that a distributed cluster agrees on. It is the storage layer behind Kubernetes — every pod definition, service endpoint, and cluster configuration lives as a key-value pair replicated across an etcd cluster using the Raft consensus protocol.

Unlike Redis, etcd sacrifices raw throughput for strong consistency: every read is guaranteed to see the latest write. That guarantee is what lets a Kubernetes control plane trust that two nodes do not simultaneously believe they are the leader.

Note: etcd is intentionally not a general-purpose cache. Its recommended dataset size is a few gigabytes; for anything larger, teams use a relational database or object store. The key-value model fits here because Kubernetes always looks up configuration by a known path — never runs ad-hoc queries across values.

Reveal: Why can't you just query Redis for all sessions belonging to a user?

Redis stores keys in a flat hash table. To find all keys matching a pattern (like session:user:4291:*) you would have to call SCAN with a match filter — which iterates every key in the store. At millions of keys this is slow and blocks other clients. The correct Redis pattern is to maintain a secondary index yourself: for example, a Set under user:4291:sessions that lists session IDs. You manage it explicitly; the key-value store does not do it for you. This is the fundamental trade-off: you get speed and simplicity, but you lose declarative querying.
