Part VI. Engineering Practice

Chapter 24. Designing for Reliability

Capacity planning

A database that runs out of disk space, connections, or memory does not degrade gracefully — it crashes. Capacity planning is the practice of measuring current resource consumption, modelling future growth, and acting before limits are reached. Done well, it is invisible engineering that prevents emergencies. Done poorly, it results in 2 AM pages and emergency infrastructure tickets.

The Three Resource Families

Database capacity problems cluster around three families of resources:

Family Resources Typical failure mode
Storage Disk space, IOPS, throughput Disk full → database halts all writes
Memory Shared buffer pool, work_mem, OS page cache Buffer pool thrashing → cache hit rate collapses → 10–100× latency spike
Compute CPU cores, connections, thread pool Connection exhaustion → new connections rejected → 503 errors

Planning for all three is required; they interact (insufficient memory forces more IOPS, which saturates CPU for I/O wait).

Measuring Current Usage

Before projecting forward, establish accurate baselines. The key measurements:

Storage

-- PostgreSQL: table and index sizes
SELECT relname AS table_name,
       pg_size_pretty(pg_total_relation_size(oid)) AS total_size,
       pg_size_pretty(pg_relation_size(oid))        AS table_size,
       pg_size_pretty(pg_indexes_size(oid))         AS index_size
FROM pg_class
WHERE relkind = 'r'
ORDER BY pg_total_relation_size(oid) DESC
LIMIT 10;

Memory (Buffer Cache Hit Rate)

-- Cache hit rate: target > 99% for OLTP workloads
SELECT
  sum(heap_blks_hit) AS cache_hits,
  sum(heap_blks_read) AS disk_reads,
  ROUND(100.0 * sum(heap_blks_hit) /
    NULLIF(sum(heap_blks_hit) + sum(heap_blks_read), 0), 2) AS hit_rate_pct
FROM pg_statio_user_tables;

Connections

-- Active connections vs. maximum
SELECT count(*) AS active,
       (SELECT setting::int FROM pg_settings WHERE name = 'max_connections') AS max_conn,
       ROUND(100.0 * count(*) /
         (SELECT setting::int FROM pg_settings WHERE name = 'max_connections'), 1) AS pct_used
FROM pg_stat_activity
WHERE state != 'idle';

Growth Modelling

Once you have baselines, project forward using historical growth rates. The simplest model is linear extrapolation from a time series of measurements:

Current size:         800 GB
Average growth/month: 40 GB (from past 6 months)
Available disk:       2 TB (2048 GB)
Headroom:             2048 - 800 = 1248 GB
Months to 85% full:   (2048 × 0.85 - 800) / 40 = 23.5 months

For write-heavy workloads with exponential growth (e.g., event logs, telemetry), model with compound growth:

size(t) = current × (1 + monthly_growth_rate)^t

Always plan to 80%, not 100%. Running a disk past 80% degrades PostgreSQL performance (autovacuum needs free space to rewrite pages). Alert at 75%, page at 85%.

Connection Capacity

The database's max_connections setting is a hard limit. Hitting it is immediately catastrophic — new connections fail with "too many connections" and the application returns errors to users.

The correct approach is connection pooling (PgBouncer, ProxySQL): applications connect to the pooler, which maintains a smaller number of real database connections. A database can comfortably handle 200–400 real connections; behind a pooler, thousands of application threads share those connections.

Application tier: 50 pods × 10 threads = 500 concurrent connections to pooler
PgBouncer pool:   30 real connections to PostgreSQL
PostgreSQL:       max_connections = 100 (safe headroom)

Table Bloat and Autovacuum

In PostgreSQL, UPDATEs and DELETEs do not immediately reclaim space — they leave dead tuples that autovacuum must clean up. Tables that are vacuumed too infrequently grow bloated: a 10 GB table may physically occupy 40 GB on disk.

-- Identify bloated tables (approximate)
SELECT relname,
       n_dead_tup,
       n_live_tup,
       ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 1) AS dead_pct
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
ORDER BY dead_pct DESC;

Tune autovacuum aggressively for hot tables; consider manual VACUUM ANALYZE after large bulk operations.

Key Takeaways

  • Capacity problems happen in three families: storage, memory, and compute — monitor all three
  • Establish growth baselines from 3–6 months of historical data; linear extrapolation is usually sufficient
  • Alert at 75% disk usage; order storage upgrades with enough lead time to avoid emergency provisioning
  • Use connection pooling (PgBouncer) to keep real database connections well below max_connections
  • Watch table bloat (dead tuple %) on high-churn tables and tune autovacuum accordingly
