Part III. Storage and Execution

Chapter 10. Trees, Logs, and Internal Structures

Space amplification

Every database stores more bytes on disk than the "pure" data actually requires. That gap — the ratio of bytes consumed to bytes of useful data — is called space amplification. It is one of the three fundamental trade-offs in storage system design (alongside read amplification and write amplification), and understanding it will help you reason about why different engines behave so differently in production.

Where the Extra Space Comes From

Space amplification is not a bug; it is an unavoidable consequence of how storage engines work. Several forces push it upward:

Source What it looks like on disk
Dead row versions (MVCC) Old row images kept until vacuum/GC runs
B-tree internal pages Branch nodes that hold keys but not full rows
Fragmentation holes Gaps left after deletes inside heap/B-tree pages
WAL / redo log A separate copy of every change, before it lands in the main file
LSM compaction overhead Multiple sorted runs holding overlapping key ranges
Index structures One or more extra files per indexed column

A freshly vacuumed, freshly rebuilt table has the lowest amplification. After months of steady writes and deletes, that number climbs — sometimes dramatically.

Measuring it

The simplest definition:

space amplification = bytes used on disk
                      ─────────────────────
                      bytes of live data

A ratio of 1.0 is impossible in practice (even a one-row table needs a page header). A ratio of 1.1 is excellent. Ratios above 3–4× are common in write-heavy LSM-based stores during heavy compaction lag, and PostgreSQL tables bloated from MVCC can exceed 2× without regular AUTOVACUUM.

B-Trees vs. LSM-Trees

The two dominant storage engine families handle the amplification trade-off very differently.

B-tree engines (PostgreSQL, MySQL InnoDB, SQLite) update data in place inside fixed-size pages. When a row is deleted or superseded, the old bytes sit as dead space inside the page until a maintenance pass (VACUUM, OPTIMIZE TABLE) reclaims them. Because pages are reused rather than appended, the total file size grows slowly — but internal fragmentation accumulates.

LSM-tree engines (RocksDB, Cassandra, LevelDB) never overwrite data. Every write is appended to a new file (an SSTable or segment). This eliminates in-place fragmentation but creates a different problem: multiple files may contain different versions of the same key. Until compaction merges them, all those versions occupy disk space simultaneously.

Note: RocksDB's space amplification in the worst case (before compaction) can be up to the number of levels × the size of a single level, because each level can hold a full copy of overlapping key ranges. In practice, RocksDB targets roughly 1.1× space amplification after a full compaction pass.

Practical Levers

You can actively manage space amplification:

  • Run VACUUM / AUTOVACUUM (PostgreSQL) — reclaims dead tuples; VACUUM FULL rewrites the file entirely, shrinking it to minimum size.
  • OPTIMIZE TABLE (MySQL) — rebuilds the clustered index and reclaims free pages.
  • SQLite VACUUM — rewrites the entire database file into a fresh, compacted copy. Requires exclusive access and 2× free space.
  • Trigger compaction (RocksDB-based systems) — force a manual compaction if you know a large batch of deletes just landed and you cannot wait for the background job.
  • Right-size fill-factor — PostgreSQL lets you set a per-index fillfactor (default 90). Lowering it pre-reserves space on each page for updates, reducing future page splits and fragmentation.

Key Takeaways

  • Space amplification is the ratio of disk bytes consumed to bytes of live data. A value of 1.0 is impossible; 1.05–1.3 is healthy; values above 2–3× signal a maintenance problem.
  • B-tree engines accumulate internal fragmentation (holes inside pages); LSM engines accumulate version overlap across SSTable levels.
  • Regular maintenance (VACUUM, compaction, OPTIMIZE TABLE) is not optional on write-heavy workloads — it is what keeps amplification from compounding over time.
  • Space amplification is always a trade-off against write speed: engines that minimize space costs (aggressive in-place updates) tend to pay more in write I/O, and vice versa.
