Chapter 4. Schemas and Data Modeling

One-to-one, one-to-many, many-to-many

Every time two kinds of things in your system are connected — a user and a profile, an order and its line items, a student and their courses — you are describing a relationship. Relational databases express these connections with foreign keys, but before you write a single REFERENCES clause you need to know what kind of relationship you are modelling. There are three patterns, and choosing the wrong one leads to data you cannot query cleanly or constraints you cannot enforce.

One-to-One

A one-to-one relationship means each row in table A corresponds to at most one row in table B, and vice versa. This is the rarest pattern in practice.

A classic example: a users table and a user_profiles table. Every user has exactly one profile, and every profile belongs to exactly one user.

users user_profiles
user_id (PK) ←──────── user_id (PK, FK)
email bio
created_at avatar_url

The profile table holds user_id as both its primary key and a foreign key back to users. That double role enforces the one-to-one constraint: you cannot have two profile rows for the same user because primary keys are unique.

Note: You could add profile columns directly to users. Splitting into a separate table makes sense when the extra columns are large, rarely needed, or belong to a different subsystem. It is a design choice, not a rule.

One-to-Many

One-to-many is by far the most common relationship. One row in table A links to many rows in table B, but each row in B points back to exactly one row in A.

Examples abound: one customer places many orders; one blog post has many comments; one department employs many staff members.

The pattern is straightforward: the many side holds a foreign key column that references the one side.

CREATE TABLE customers (
  customer_id SERIAL PRIMARY KEY,
  name        TEXT NOT NULL
);

CREATE TABLE orders (
  order_id    SERIAL PRIMARY KEY,
  customer_id INTEGER NOT NULL,         -- the foreign key lives on the "many" side
  placed_at   TEXT NOT NULL,
  FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

Notice the foreign key always goes on the many side. Putting it the other way — storing a list of order IDs inside a customer row — is not possible in a plain relational column and is a sign the design needs to change.

Many-to-Many

A many-to-many relationship means rows on both sides can relate to multiple rows on the other side. A student enrols in many courses; a course contains many students. A product belongs to many tags; a tag applies to many products.

You cannot model this directly with a single foreign key column — there is no single column that can hold multiple references. The standard solution is a junction table (also called a bridge table or associative table) that sits between the two main tables and holds one row per connection.

CREATE TABLE students (
  student_id SERIAL PRIMARY KEY,
  name       TEXT NOT NULL
);

CREATE TABLE courses (
  course_id  SERIAL PRIMARY KEY,
  title      TEXT NOT NULL
);

-- The junction table: each row = one enrolment
CREATE TABLE enrolments (
  student_id INTEGER NOT NULL,
  course_id  INTEGER NOT NULL,
  enrolled_on TEXT NOT NULL,
  PRIMARY KEY (student_id, course_id),
  FOREIGN KEY (student_id) REFERENCES students(student_id),
  FOREIGN KEY (course_id)  REFERENCES courses(course_id)
);

The composite primary key (student_id, course_id) prevents duplicate enrolments. The junction table also gives you a natural place to store attributes that belong to the relationship itself — like the date a student enrolled, or a grade.

Choosing the Right Relationship

Relationship Example Where the FK lives
One-to-one user → profile FK on the dependent table (profile)
One-to-many customer → orders FK on the many side (orders)
Many-to-many students ↔ courses Junction table with two FKs

When you are unsure which pattern fits, ask: "Can a row on side A link to more than one row on side B — and can a row on side B link to more than one row on side A?" If only one direction is many, it is one-to-many. If both directions are many, you need a junction table. Getting this right before you build the schema saves painful migrations later.
