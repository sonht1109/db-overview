Nested loop joins

When the database engine needs to combine rows from two tables, it must pick a strategy for finding matching pairs. The nested loop join is the most intuitive of those strategies — and understanding it gives you a solid mental model for reasoning about join performance in general.

How It Works

Picture two tables: an outer table and an inner table. The algorithm is exactly what the name says:

for each row R in the outer table:
    for each row S in the inner table:
        if R and S satisfy the join condition:
            emit (R, S)

That's a nested loop — one loop inside another. Every row in the outer table is compared against every row in the inner table. If the outer table has M rows and the inner table has N rows, the worst case is M × N comparisons.

A Concrete Example

Suppose you have these two tables:

orders (outer)
order_id customer_id
1 42
2 17
3 42
customers (inner)
customer_id name
17 Priya
42 Leon

To join them on orders.customer_id = customers.customer_id, the engine:

  1. Takes order row 1 (customer_id = 42) and scans all customer rows → finds Leon.
  2. Takes order row 2 (customer_id = 17) and scans all customer rows → finds Priya.
  3. Takes order row 3 (customer_id = 42) and scans all customer rows → finds Leon again.

Three outer rows × two inner rows = six comparisons, three matches.

When Nested Loop Joins Are Fast (and When They're Not)

The nested loop join has a wide performance range depending on the data and available indexes.

Best case — index on the inner table. If the inner table has an index on the join column, each outer-row lookup becomes an index seek rather than a full scan. The cost drops from O(M × N) to roughly O(M × log N). This is why databases almost always prefer nested loop joins when one side is small and the other side is indexed.

Worst case — no index, large tables. Without an index, every outer row triggers a full scan of the inner table. For two tables with 100 000 rows each, that's 10 billion comparisons. In practice, the query planner switches to a hash join or merge join for large unindexed joins.

Note: You can observe which join strategy the planner chose by running EXPLAIN (or EXPLAIN in SQLite) before your query. If you see "SCAN TABLE" for the inner side, it is doing an unindexed nested loop.

Key Takeaways

  • A nested loop join iterates the outer table once and, for each row, probes the inner table.
  • With an index on the inner join column, cost is O(M × log N) — efficient even for moderate-sized tables.
  • Without an index, cost is O(M × N) — the planner will typically avoid nested loops for large tables and choose a hash join or sort-merge join instead.
  • Choosing the smaller table as the outer input only helps when the inner table has an index — otherwise M×N equals N×M. The real benefit of swapping the outer/inner roles comes when the inner side can be probed via an index seek, turning a full scan into O(log N) per outer row.
  • Most relational engines choose the join algorithm automatically, but understanding nested loops helps you interpret EXPLAIN output and write index-friendly queries.
