Part I. Foundations

Chapter 1. What a Database System Is

Clients, servers, and drivers

When you run a query, you are rarely talking to the database directly. Almost every database interaction involves three distinct roles: a client that asks questions, a server that stores and processes data, and a driver that makes them speak the same language. Understanding these three pieces explains why databases work the same way across a dozen programming languages — and what actually happens in the milliseconds between you calling a function and getting a result back.

The client-server model

A database server is a long-running process dedicated to managing data. It listens on a network port (PostgreSQL defaults to 5432, MySQL to 3306, SQL Server to 1433), accepts connections, runs queries, and returns results. The server owns the data files on disk and enforces rules — access control, transactions, constraints — on everything that touches them.

A client is anything that connects to that server to send queries and receive results. The client could be:

Client type Example
Interactive shell psql, mysql, sqlite3
GUI tool DBeaver, TablePlus, DataGrip
Application code A Node.js API, a Django app
Another server A reporting service, an ETL job

The key insight: the server does the heavy lifting. Sorting a million rows, joining three tables, enforcing a foreign-key constraint — all of that happens inside the server process. The client just sends SQL text and gets rows back.

Note: SQLite breaks this pattern intentionally. It is an embedded database — the "server" is a library linked directly into your application, and the data file is local. There is no network port or separate process. This makes SQLite ideal for local tools and tests but unsuitable for multiple simultaneous writers.

What a driver does

The client and server need a common wire protocol — a byte-level agreement on how to frame a query, authenticate, stream rows, and signal errors. Writing that protocol by hand for every application would be painful. That is what a driver (sometimes called a connector or adapter) is for.

A driver is a language-specific library that:

  1. Opens and manages a TCP connection to the server.
  2. Handles authentication (username/password, SSL certificates, Kerberos, etc.).
  3. Serialises your query string into the protocol format the server expects.
  4. Deserialises the binary response back into native language types (strings, integers, dates).
  5. Exposes a clean API your application code can call.

Each database has official or community drivers for every popular language:

Database Python driver Node.js driver Java driver
PostgreSQL psycopg2 / asyncpg pg pgjdbc
MySQL mysql-connector-python mysql2 MySQL Connector/J
SQLite built-in sqlite3 module better-sqlite3 xerial/sqlite-jdbc

A higher-level library (an ORM like SQLAlchemy or Prisma) sits on top of a driver — it generates SQL for you, but it still needs the driver to actually talk to the server.

Putting it together: a query's round trip

Here is what happens when an application runs SELECT * FROM orders WHERE status = 'pending':

Application code
    │  calls driver API (e.g., cursor.execute(...))
    ▼
Driver
    │  serialises query → TCP bytes → sends to port 5432
    ▼
Database server (network)
    │  parses SQL, plans query, reads data pages
    │  serialises result rows → TCP bytes → sends back
    ▼
Driver
    │  deserialises bytes → Python list / JS array / Java ResultSet
    ▼
Application code
    │  iterates rows

The round-trip time on a local network is typically under 1 ms. Over the public internet it can be 50–200 ms — which is why database servers are always deployed close to the application servers that query them, and why connection pooling (reusing an already-open TCP connection rather than opening a new one for every query) matters at scale.
