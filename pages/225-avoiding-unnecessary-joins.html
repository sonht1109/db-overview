Chapter 23. Designing for Performance

Avoiding unnecessary joins

Joins are one of the most powerful features of relational databases — and one of the most common sources of performance problems. The issue is rarely that joins are slow in principle; it is that unnecessary joins compound, hide behind ORMs, and make a query plan that should touch one table touch five. Understanding when to join, when to denormalize, and when to pre-compute transforms reactive firefighting into intentional design.

Why Joins Cost What They Cost

When two tables are joined, the database must match rows from each side. The planner chooses an algorithm:

Algorithm Best when Cost
Nested loop Small outer set + indexed inner O(n log m) with index
Hash join Large unsorted tables O(n + m), high memory
Merge join Both sides already sorted O(n + m), low memory

The problem is not the algorithm — it is that each join multiplies the I/O and CPU. Three joins on 1 M-row tables can mean scanning tens of millions of logical rows even with good indexes.

The N+1 Query Problem

The most common ORM-induced join disaster is the N+1 problem: loading a list of N records and then executing one query per record to fetch related data, because the developer forgot to eager-load the association.

# N+1: loads 100 orders, then 100 separate customer queries
orders = Order.all()          # 1 query
for order in orders:
    print(order.customer.name) # N queries

# Fixed: a single join or eager load
orders = Order.includes(:customer).all()  # 1 query with join

The N+1 problem looks trivial in development (small dataset, fast) and catastrophic in production (large dataset, slow + high connection pressure).

When to Denormalize

Normalization is a correctness tool. Denormalization is a performance tool. Use denormalization when:

  1. The join is on the critical path of a high-frequency query.
  2. The denormalized column is read far more than it is updated.
  3. The join crosses a network boundary (microservices, sharded databases).
-- Normalized: must join orders + customers for every page load
SELECT o.id, c.name, c.email
FROM   orders o
JOIN   customers c ON c.id = o.customer_id;

-- Denormalized: customer_name copied into orders at insert time
SELECT id, customer_name, customer_email FROM orders;
-- Now a single-table lookup — no join at runtime

Document the denormalization. Write the update trigger or application logic that keeps the copied column current. Denormalization that is not maintained becomes corrupt data.

Avoiding Joins at Query Time: Materialized Views

A materialized view pre-computes a join and stores the result. Reads hit the materialized result (fast). A refresh job or trigger updates it when source data changes.

-- PostgreSQL: pre-compute the join nightly
CREATE MATERIALIZED VIEW order_summary AS
SELECT o.id, o.total, c.name AS customer_name
FROM   orders o
JOIN   customers c ON c.id = o.customer_id;

REFRESH MATERIALIZED VIEW order_summary;  -- run on schedule

SQLite does not have materialized views but you can simulate them with a regular table refreshed by application code.

When Joins Are Fine

Do not over-apply this advice. Joins are appropriate when:

  • The joined table is small (fits in the buffer pool / hash table).
  • The join is infrequent (batch reports, admin queries).
  • You need consistency — a denormalized copy can drift.
  • The planner can use a good index on the join key.

Key takeaways: Joins are powerful but compound. Audit your ORM queries for N+1 patterns. Denormalize hot join paths when the column is read far more than updated. Use materialized views to pre-compute expensive joins for reporting. Never denormalize without documenting how the copy is kept in sync.
