Part VII. Building Understanding

Chapter 29. Common Myths About Databases

What to believe instead

After eight pages of myth-busting, a natural question surfaces: if SQL isn't inherently slow, indexes don't always help, ACID doesn't preclude scale, and no single database does everything — what should you believe? This closing page distills the nuanced reality into a set of durable mental models for choosing, tuning, and trusting databases in production systems.

From Myths to Mental Models

The myths we've examined share a common flaw: they substitute a simple, memorable rule for a more complex, contextual truth. Database engineering is full of situations where "it depends" is the correct answer — but "it depends" is only useful if you know what it depends on. These mental models make that dependency explicit.

Mental Model 1: Measure Before You Optimize

Every myth in this chapter leads someone to make a significant architectural decision — switching databases, avoiding SQL, adding indexes everywhere, distributing too early — without measuring the actual bottleneck. The invariant that prevents all of these mistakes is the same:

Profile first. Optimize the thing you measured, not the thing you assumed.

  • Run EXPLAIN ANALYZE before concluding SQL is slow.
  • Check pg_stat_user_indexes before adding or removing indexes.
  • Load-test your single node before distributing.
  • Measure write amplification before denormalizing.

The performance issue you imagine and the performance issue that actually exists are frequently different problems.

Mental Model 2: Match the Tool to the Access Pattern

Every database makes architectural trade-offs. Row-stores are fast for point lookups and complex queries; column-stores are fast for aggregations over large datasets; key-value stores are fast for single-key reads; graph databases traverse relationships efficiently. No universal trade-off winner exists.

The decision framework:

1. What is the primary access pattern?
   - Transactional (CRUD, ACID): relational DB
   - Exact-key reads at high throughput: key-value store
   - Aggregations over time ranges: time-series or column-store
   - Full-text search: inverted-index search engine
   - Graph traversal: graph DB

2. Does my existing database handle it acceptably?
   - Yes → don't add another system
   - No → consider extension first, then specialist DB

3. Does the operational cost of a new system justify the benefit?
   - Measure the gap, estimate the cost, decide.

Mental Model 3: Schemas Are Contracts, Not Obstacles

Schemas — whether enforced by the database or by application code — represent the contract between the data layer and every consumer of that data. The myth that schemaless means simpler collapses the distinction between "no database-enforced schema" and "no schema."

The productive framing: who enforces the contract?

  • Database: enforcement is automatic, universal, and reported as errors at write time.
  • Application: enforcement is manual, partial, and fails silently when code is missing or wrong.

Choose where the contract lives based on how critical correctness is and how much you trust all writers to the data. For financial data, inventory levels, and user credentials: database enforcement. For optional user metadata and configuration blobs: application enforcement is acceptable.

Mental Model 4: Consistency Is a Spectrum, Not a Binary

The "ACID vs. eventual consistency" framing suggests a binary choice. The reality is a spectrum:

Level Guarantee Cost Use Case
Serializable All transactions appear to run one at a time Highest Financial ledgers, inventory
Repeatable Read No phantom reads within a transaction High Reports, analytics queries
Read Committed See only committed data (default in most DBs) Medium Most OLTP workloads
Read Uncommitted May see uncommitted data Low Rarely appropriate
Eventual Eventually converges to correct state Lowest Shopping carts, social feeds

Most applications run at Read Committed and never need anything stricter. Weakening to eventual consistency is appropriate for specific workloads (global reads, high-write-throughput non-financial data) — but it should be a deliberate decision, not a default assumption.

Mental Model 5: Distribution Is a Solution, Not a Default

Distributed systems are harder to operate, reason about, and debug than single-node systems. The distributed default should never be "use a distributed database because it sounds more scalable." It should be "use a distributed database because we have measured a bottleneck that a single node cannot address."

The questions to ask before distributing:

  1. Have we optimized the single-node setup? (Indexes, query tuning, connection pooling, read replicas.)
  2. What is the specific bottleneck? (Write throughput? Data volume? Read latency for geographically distant users?)
  3. Does the distribution model match the bottleneck? (Read replicas for read latency; sharding for write throughput; global distribution for geographic latency.)

What to Believe Instead: A Summary

Myth What to believe instead
SQL is slow SQL speed is determined by indexes and schema design; the query planner does the optimization
NoSQL means no schema Every application has a schema; the question is who enforces it
Indexes always help Indexes are a read/write trade-off; measure selectivity and write load before adding
ACID means no scale Single-node ACID handles enormous workloads; distributed ACID is real and deployable
Distributed means faster Distribution adds coordination cost; profile before distributing
Flexible schema means easy design Deferred design decisions become expensive application-layer complexity
Denormalization always helps Denormalization trades consistency risk for read speed; measure before duplicating
One database can do everything Different access patterns have different optimal storage structures

Key Takeaways

  • Measure before you optimize — the bottleneck you imagine and the one you have are usually different.
  • Choose the database that fits the access pattern, not the one that sounds most scalable.
  • Schema enforcement is a question of where, not whether — database or application code, both have a cost.
  • Consistency guarantees exist on a spectrum; most applications work well at Read Committed.
  • Distribution is a measured response to a specific bottleneck, not a default architectural choice.
  • The best database decision you'll ever make is the one you can justify with a measurement.
