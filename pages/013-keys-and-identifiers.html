Chapter 2. How Data Is Organized

Keys and identifiers

Every table needs a way to tell rows apart. Without that, you cannot reliably update a single record, link two tables together, or enforce that the same thing is not stored twice. Keys solve this — they are the identifiers that give every row a unique, stable address.

Primary Keys: One Row, One Identity

A primary key is a column (or combination of columns) whose value uniquely identifies each row in a table. The database enforces two rules automatically:

  • Uniqueness — no two rows may share the same primary key value.
  • Not null — a primary key must always have a value; it cannot be left blank.
CREATE TABLE customers (
    customer_id INTEGER PRIMARY KEY,
    name        TEXT NOT NULL,
    email       TEXT NOT NULL
);

Here customer_id is the primary key. If you try to insert two rows with customer_id = 7, the database rejects the second one immediately — no application-level check needed.

Natural vs. Surrogate Keys

There are two schools of thought on what to use as a primary key:

Type What it is Example Trade-off
Natural key A real-world value that is already unique Email address, ISBN, passport number Meaningful, but can change or be longer than needed
Surrogate key A system-generated value with no business meaning Auto-incrementing integer, UUID Stable and compact, but opaque

Most production tables use surrogate keys — typically an auto-incrementing integer (INTEGER PRIMARY KEY in SQLite auto-increments) or a UUID — because natural identifiers change more than people expect. An email address seems permanent until a customer asks to change it.

Note: In SQLite, declaring a column as INTEGER PRIMARY KEY automatically makes it an alias for the internal rowid, giving you auto-increment behavior with no extra syntax.

Foreign Keys: Linking Tables Together

A foreign key is a column whose value must match an existing primary key in another table. It is how relationships are expressed in a relational database.

CREATE TABLE orders (
    order_id    INTEGER PRIMARY KEY,
    customer_id INTEGER NOT NULL,
    placed_at   TEXT NOT NULL,
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

The FOREIGN KEY constraint tells the database: "every customer_id in orders must exist in customers." Trying to insert an order for a customer that does not exist fails. Trying to delete a customer who still has orders also fails — unless you have configured cascading rules.

This constraint is what prevents orphan records — orders pointing at customers who no longer exist.

Composite Keys and Unique Constraints

Sometimes no single column uniquely identifies a row, but a combination of columns does. A composite primary key uses two or more columns together.

CREATE TABLE course_enrollments (
    student_id INTEGER NOT NULL,
    course_id  INTEGER NOT NULL,
    enrolled_at TEXT NOT NULL,
    PRIMARY KEY (student_id, course_id)
);

A student can enroll in many courses, and a course can have many students — but the pair (student_id, course_id) must be unique. Attempting to enroll the same student in the same course twice is rejected.

A UNIQUE constraint is a softer version: it enforces uniqueness without making the column the primary key. This is useful for things like email addresses that should not repeat but are not the main identifier.

CREATE TABLE users (
    user_id INTEGER PRIMARY KEY,
    email   TEXT NOT NULL UNIQUE
);

Why Keys Matter Beyond Correctness

Keys do more than prevent bad data. They are the foundation of two other critical database features:

  • Indexes — the database automatically creates an index on the primary key, making lookups by that column fast even across millions of rows. Foreign key columns are good candidates for manual indexes too.
  • Joins — every JOIN in SQL works by matching values across tables. Without a reliable primary/foreign key relationship, joins produce ambiguous or incorrect results.

A well-chosen key strategy is one of the earliest and most consequential design decisions you make when building a schema. Get it right early — changing primary keys in a live system is painful.
