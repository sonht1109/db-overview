Part V. Database Families

Chapter 18. Key-Value Databases

Range scans and ordered keys

Pure hash-based key-value stores are magnificent for point lookups — one key, one value, done. But the moment you ask "give me all events from the last hour" or "list every user whose ID falls between 1000 and 2000," a hash table has nothing useful to offer. That is where ordered key-value stores step in: by keeping keys sorted, they make range queries a first-class operation.

Why Order Matters for Ranges

In a hash table, adjacent keys in the hash space have nothing to do with adjacent keys in the logical space. The hash of "user:1000" might land in slot 7, while "user:1001" lands in slot 412. To find all keys between two values you must scan every slot — O(n) work regardless of how narrow the range is.

An ordered store keeps keys in lexicographic (sorted) order on disk and in memory. Keys that sort near each other are stored physically near each other. A range query then becomes a single seek to the start key followed by a sequential read to the end key — far cheaper than a full scan.

Key Design: Putting Order to Work

Because the sort is lexicographic (byte-by-byte, like dictionary order), you can engineer ranges directly into the structure of your keys. This is the single most important design skill in ordered key-value stores.

Composite keys and natural ranges

Consider an event log for a multi-tenant application. By composing the key as:

<tenant_id>:<ISO-8601-timestamp>:<event_id>

...you get three useful range queries for free:

Query goal Key prefix / range
All events for tenant acme SCAN "acme:" … "acme:~"
Events for acme in March 2024 SCAN "acme:2024-03-" … "acme:2024-03-~"
Single event by exact ID GET "acme:2024-03-10T14:22:01Z:evt_991"

The key insight: ISO 8601 timestamps sort correctly as plain strings ("2024-03" < "2024-04" lexicographically), so no special date logic is needed. Numeric IDs need zero-padding for the same reason ("007" < "010" whereas "7" > "10").

Note: RocksDB, LevelDB, and FoundationDB — three widely-used ordered key-value stores — all expose this exact SCAN(start, end) primitive. Redis sorted sets offer a similar capability through ZRANGEBYLEX.

The prefix scan pattern

A prefix scan retrieves every key that starts with a given string. Implementations typically express the open end of the range by appending the highest sentinel character after the prefix:

start = "acme:2024-03-"
end   = "acme:2024-03~"   -- '~' (ASCII 126) sits above all printable alphanum chars

This pattern appears everywhere — from namespace isolation in multi-tenant stores to per-user data shards in distributed systems.

PostgreSQL Range Scans on B-tree Indexes

PostgreSQL uses B-tree indexes as its default index type, and they are the workhorse for range queries. Key behaviors to know:

  • PostgreSQL B-trees store index entries in sorted key order within each leaf page, and leaf pages are linked together in a doubly-linked list. A range scan performs a single seek to the start key (via B-tree traversal, O(log n)), then follows the leaf-page chain forward — reading multiple rows from each page — until the end key is reached. This is O(log n + k) where k is the result size.
  • The Index Scan node reads matching index entries and fetches heap tuples. If the table is clustered on the index key (CLUSTER command) or correlation is high, the heap fetches benefit from sequential I/O. With low correlation, a Bitmap Index Scan may be chosen instead — it builds a bitmap of matching pages and fetches them in physical order to reduce random I/O.
  • Covering indexes (INCLUDE clause, PostgreSQL 11+) allow an Index Only Scan that never visits the heap at all, dramatically improving range scan throughput — especially on wide tables.
  • For time-series or append-only workloads, BRIN indexes (Block Range INdex) can be far more compact than B-trees while still supporting range scans efficiently, because naturally ordered data already clusters nearby values in the same blocks.

Note: Use EXPLAIN (ANALYZE, BUFFERS) to see whether a query is using an Index Scan, Index Only Scan, or Bitmap Scan — and how many buffer hits it costs. The difference between an Index Scan (heap fetches per row) and an Index Only Scan (no heap fetches) can be 5–50× in throughput for large ranges.

Maintaining sort order is not free. Ordered stores typically use a B-tree or an LSM-tree (log-structured merge-tree), both of which incur more write overhead than a simple hash table. Random inserts in a B-tree require page splits; LSM-trees batch writes but run background compaction to merge sorted runs.

Property Hash KV (e.g. Redis) Ordered KV (e.g. RocksDB)
Point lookup O(1) average O(log n)
Range scan O(n) — full scan O(log n + k) where k = result size
Write throughput Very high High (LSM) / moderate (B-tree)
Key ordering None Lexicographic
Typical use case Caching, sessions Event logs, time-series, sorted leaderboards

Choose an ordered key-value store when your queries are naturally range-shaped — time windows, alphabetical sweeps, prefix lookups — and you are willing to encode that shape into your key design. Choose a hash store when you always know the exact key and need the lowest possible latency for point reads.

Key Takeaways

  • Hash tables scatter keys randomly — range queries require a full scan.
  • Ordered stores keep keys in lexicographic order so range queries read a contiguous segment in O(log n + k).
  • Key design is everything: timestamps, zero-padded numbers, and composite prefixes let you engineer range access patterns up front.
  • ISO 8601 timestamps and zero-padded integers are self-ordering as plain strings — use them freely.
  • The prefix-scan sentinel pattern (prefix~) is a widely used idiom for expressing open-ended ranges.
