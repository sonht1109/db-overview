Part IV. Distributed Systems

Chapter 15. Replication

Replication lag

In an ideal world, every replica would mirror the primary's data the moment a write is committed. In practice there is always a delay — replication lag — the gap in time between when a write lands on the primary and when it becomes visible on a replica. Understanding lag is essential because your application may read from replicas for performance reasons, and stale reads can silently break correctness guarantees.

Why Lag Exists

Replication is an asynchronous pipeline. After the primary writes and flushes a change to its own log (e.g., the WAL in PostgreSQL or the binlog in MySQL), it ships that log entry to each replica. The replica must then receive, write, and apply the change before it becomes queryable. Every step in that pipeline adds time:

Pipeline stage Typical source of delay
Primary → network Bandwidth saturation, TCP overhead, cross-region latency
Network → replica disk Write I/O on the replica host
Apply (replay) Replica CPU replaying writes serially (single-threaded replay is common)
Long transactions The replica cannot advance past an open transaction on the primary

Under low load, lag may be milliseconds. Under heavy write bursts — bulk imports, large transactions, schema changes — it can balloon to seconds or minutes.

Consequences: Anomalies You Will See

Lag is not just a performance footnote — it changes what reads return.

Stale reads — A user writes a row, then immediately queries a replica that hasn't applied the write yet. The row appears missing. This is the most common complaint in read-replica architectures.

Read-your-own-writes violation — Your application writes to the primary but reads from a lagged replica. The user who just submitted a form sees the old data when they're redirected to the confirmation page.

Monotonic reads violation — A user refreshes twice. The first request hits Replica A (up to date), the second hits Replica B (further behind). The data appears to go backwards.

Note: These anomalies are the reason databases define consistency levels — a formal vocabulary for trading off freshness against performance (more in the next chapter on consistency models).

Measuring Lag

Most databases expose lag metrics directly:

-- PostgreSQL: check replica lag from the primary
SELECT
    client_addr,
    state,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication;

-- MySQL / MariaDB: check lag on the replica itself
SHOW REPLICA STATUS\G
-- Look at: Seconds_Behind_Source

Monitoring tools like Prometheus + pg_exporter or Datadog can alert when lag exceeds a threshold (e.g., > 30 seconds).

Mitigating Lag

There is no single fix — the right approach depends on your workload:

Strategy How it helps Trade-off
Synchronous replication Zero lag — primary waits for replica ACK Writes are slower; replica failure blocks primary
Read from primary for critical reads Always fresh Loses the scale-out benefit of replicas
Sticky sessions / read-after-write routing Route a user's reads to primary just after they write Adds routing complexity
Parallel replay Replicas apply changes on multiple threads Reduces CPU bottleneck; not all engines support it fully
Smaller transactions Shorter apply time per event Requires application redesign

Note: PostgreSQL's synchronous_commit = remote_apply setting guarantees a replica has applied (not just received) a WAL record before the primary acknowledges the commit. This eliminates lag for those writes at the cost of higher commit latency.
