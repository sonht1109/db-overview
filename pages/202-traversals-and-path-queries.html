Part V. Database Families

Chapter 21. Graph Databases

Traversals and path queries

If nodes and edges are the nouns and verbs of a graph, traversal is the grammar that makes them useful. A traversal starts at one or more nodes and follows edges outward — hop by hop — collecting or filtering nodes along the way. This is fundamentally different from a relational join: instead of comparing columns across two flat tables, the database follows physical pointers embedded in each node's record, expanding outward through the network.

How Graph Traversal Works

Every traversal begins with a start set — one or more seed nodes, usually identified by a property filter (e.g., name = "Alice"). From there the engine expands outward by following edges. Two classic strategies:

  • Breadth-First Search (BFS): Explore all neighbors at depth 1 before moving to depth 2. Finds the shortest path and all nodes reachable within a given number of hops. Memory-intensive because it must track the entire frontier.
  • Depth-First Search (DFS): Follow one path as deep as possible before backtracking. Lower memory footprint than BFS; useful for finding any path or exhausting all paths.

Most graph query languages let you specify path patterns without choosing the algorithm explicitly — the query planner picks the appropriate strategy based on the query shape (shortest path → BFS; all paths → DFS or iterative deepening).

Cypher Path Patterns

In Cypher (Neo4j's query language), you describe paths declaratively:

-- One-hop: direct friends of Alice
MATCH (a:Person {name: "Alice"})-[:KNOWS]->(friend)
RETURN friend.name

-- Two hops: friends-of-friends
MATCH (a:Person {name: "Alice"})-[:KNOWS*2]->(fof)
RETURN fof.name

-- Variable-length: all nodes reachable within 1–3 hops
MATCH (a:Person {name: "Alice"})-[:KNOWS*1..3]->(reach)
RETURN DISTINCT reach.name

-- Shortest path between Alice and Carol
MATCH p = shortestPath(
  (a:Person {name: "Alice"})-[:KNOWS*]-(c:Person {name: "Carol"})
)
RETURN length(p), [n IN nodes(p) | n.name] AS path

The * notation means "any number of hops"; *1..3 bounds it. shortestPath() activates a BFS internally.

Index-Free Adjacency

The reason graph traversal is fast in a native graph database comes down to index-free adjacency: each node record contains a direct pointer (or a linked list of pointers) to its adjacent edges. Following one hop is an O(1) memory dereference — the engine reads the pointer and jumps to the edge record, then reads the pointer to the target node.

Contrast this with a relational join. Even with an index, each hop requires:

  1. A B-tree index lookup on the foreign key value
  2. A row fetch from the joined table
  3. Repeat for each matching row

The index lookup is O(log N) in the size of the indexed table — and for multi-hop queries it compounds: three hops means three nested O(log N) lookups. In a graph with millions of nodes the difference between pointer-traversal and index-lookup per hop is orders of magnitude.

Types of Path Queries

Query type Description Example use case
Single-hop Direct neighbors "Who does Alice follow?"
Fixed-depth Exactly N hops "Friends-of-friends (depth 2)"
Variable-length Between min and max hops "Nodes within 3 hops of Alice"
Shortest path Minimum-hop route between two nodes "Degrees of separation between Alice and Frank"
All paths Every route, possibly bounded "All supply chain routes from factory to store"
Weighted shortest path Dijkstra / A* over edge weights "Cheapest flight route"

Note: Unbounded all-paths queries (MATCH p=(a)-[*]->(b)) can be exponential in the number of cycles in the graph. Always bound variable-length traversals in production queries.

Multi-Hop Traversal: The Relational Challenge

Relational databases can answer multi-hop queries using recursive CTEs (Common Table Expressions). The SQL standard introduced WITH RECURSIVE precisely for this use case. Here is a three-hop friend-of-friend query:

WITH RECURSIVE reachable(id, depth) AS (
  -- Base case: start from Alice (id=1)
  SELECT end_id, 1
  FROM   edges
  WHERE  start_id = 1 AND type = 'KNOWS'

  UNION ALL

  -- Recursive case: follow one more hop
  SELECT e.end_id, r.depth + 1
  FROM   reachable r
  JOIN   edges e ON e.start_id = r.id AND e.type = 'KNOWS'
  WHERE  r.depth < 3
)
SELECT DISTINCT id FROM reachable;

This works, but it has a critical scalability problem: each recursive step is a full join over the entire edges table. The query planner re-evaluates the join predicate for every row at every depth level. For a graph with millions of edges, a 5-hop traversal requires five sequential full-table joins. The cost grows as O(E × depth) where E is the number of edges — compared to O(neighbors × depth) for a native graph database using index-free adjacency.

Why Recursive CTEs Get Expensive

Three compounding problems make recursive CTEs unscalable for deep graph traversal:

1. Full-table join per level. Each recursive step joins against every edge in the edges table, then filters to relevant rows. A native graph store follows only the edges stored on the current node — irrelevant edges are never touched.

2. No native visited-node tracking. SQL has no built-in cycle detection. You must maintain a path string or visited-set array using non-standard extensions (ARRAY in PostgreSQL, string hacks in SQLite). This adds CPU and memory overhead proportional to path length.

3. Materialization between steps. Most SQL engines materialize the intermediate result of each recursive step before moving to the next. For wide graphs, this creates large temporary tables that must be written to disk.

A native graph engine — Neo4j, TigerGraph, or Amazon Neptune — maintains the traversal frontier in memory as a set of node pointers, expanding it level by level without any join or materialization. For 6-degree-of-separation queries over a 100M-node social graph, a native graph store answers in seconds; a recursive CTE may time out.

Shortest Path

Shortest-path queries are the canonical graph workload. In Cypher:

MATCH p = shortestPath(
  (a:Person {name: "Alice"})-[:KNOWS*]-(f:Person {name: "Frank"})
)
RETURN length(p)           AS hops,
       [n IN nodes(p) | n.name] AS path

The engine runs Dijkstra's algorithm (or BFS for unweighted graphs) using index-free adjacency. It terminates as soon as it finds the destination — it doesn't need to explore the entire graph.

The relational equivalent requires running the recursive CTE to completion across all reachable nodes, then filtering — you cannot terminate early because SQL has no mechanism to stop recursion when a target is found.

Key Takeaways

  • Graph traversal follows edges hop by hop, using BFS for shortest paths and DFS for all-path enumeration.
  • Index-free adjacency gives native graph databases O(1) per-hop cost: each node stores direct pointers to its adjacent edges, so traversal never touches a global index.
  • Recursive CTEs let relational databases answer multi-hop queries, but each recursive step is a full join over the edge table — cost scales with total edge count, not just the local neighborhood.
  • Shortest-path queries in native graph engines terminate early (Dijkstra / BFS); recursive CTE equivalents must exhaust all reachable nodes first.
  • For queries that are "shallow and wide" (1–2 hops over many nodes), relational joins are often competitive. For "deep and targeted" queries (5+ hops, shortest paths, cycle detection), native graph traversal wins decisively.
