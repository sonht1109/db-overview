Part V. Database Families

Chapter 21. Graph Databases

Performance characteristics

Understanding why graph databases are fast for some queries — and slow for others — requires looking at how they physically store and traverse data. This page digs into the storage layout, complexity analysis, and real-world trade-offs that determine when a graph engine outperforms a relational one and when it does not.

Index-Free Adjacency: The Core Performance Mechanism

The defining feature of native graph storage is index-free adjacency: each node record holds a direct pointer to its first relationship record, and each relationship record chains to the next relationship for each of its two endpoints. No global index is consulted during traversal.

In Neo4j's original fixed-size record format (up to Neo4j 4.x), every node record is exactly 15 bytes and every relationship record is exactly 34 bytes. Because records are fixed-size, random access by ID is arithmetic: offset = id × record_size. Both node and relationship lookups are O(1) file seeks.

Version note: Neo4j 5.x+ introduced a more flexible record format with different internal sizing. While the principle of index-free adjacency and pointer-based traversal remains, the exact byte sizes are no longer fixed at 15/34 bytes. The O(1) pointer-following behaviour, however, is unchanged.

Complexity Analysis by Query Type

Query type Relational (indexed) Graph DB
Point lookup by ID O(log N) O(1) — arithmetic offset
1-hop neighbour O(log N) index join O(degree) — follow pointer chain
k-hop traversal O(k × log N × b^k) O(b^k) — b = avg branching factor
Shortest path O(N log N) Dijkstra via CTE O(V + E) BFS on adjacency lists
Full graph scan / global aggregation O(N) sequential scan O(N) — not faster; same cost
COUNT(*) with filter O(N) or O(index scan) O(N) — graph DBs have no advantage

Key insight: Graph databases excel at local traversal where you start from known nodes and follow edges. They offer no advantage — and sometimes a disadvantage — for global queries (aggregations, full scans, analytics) because those require visiting every node or edge anyway.

Why O(b^k) Not O(k × log N × b^k)?

In a relational join, each hop requires a B-tree index lookup to find the next set of rows. The B-tree lookup costs O(log N) where N is the total number of rows in the edges table. With 100 million edges and branching factor 10:

  • Relational 3-hop: 10 + 100 + 1 000 rows fetched, each requiring ~26 B-tree comparisons = ~29 000 operations
  • Graph 3-hop: 10 + 100 + 1 000 pointer dereferences = ~1 110 operations

The ratio grows with N. On a graph of 10 billion edges the B-tree is 33 levels deep; the graph traversal pointer cost stays the same.

Storage Layout and Memory Behaviour

Adjacency List Embedding

Unlike a relational row (which contains all column data), a graph node record is lean — it stores only metadata and pointers. Properties are stored in a separate property store, chained off the node record. This means:

  • Traversal (following edges) touches only the small node/relationship records — high cache efficiency for traversal
  • Property access requires a second pointer follow — slightly more expensive than relational column access

Write Performance

Adding an edge in a graph DB requires:

  1. Write the new relationship record (fixed size, append-friendly)
  2. Update the first_rel pointer in the two endpoint nodes
  3. Update the next_rel chain in the previously-first relationship (if any)

This is 3–4 random writes regardless of graph size. Compare to a relational insert into an indexed junction table: the insert + B-tree rebalance cost grows as the table fills. Graph edge insertion stays roughly O(1).

Removing an edge is symmetric: unlink the record from both endpoint chains, mark the slot free.

Memory Usage

The pointer-heavy structure consumes more memory per logical entity than a row in a flat table:

Structure Approx. size
Relational edge row (two FK integers + index entry) ~30–50 bytes
Neo4j relationship record ~34 bytes record + property store
In-memory graph node (JVM object overhead) ~200–400 bytes

For very dense graphs where RAM is the bottleneck, purpose-built compressed formats (CSR — Compressed Sparse Row) used by graph analytics engines (GraphX, Giraph) can be more efficient than a transactional graph DB.

Scaling Behaviour

How Performance Scales with Graph Size

For local traversal queries (k-hop, shortest path starting from a known node):

  • Graph DB: Performance is determined by the local neighbourhood size, not total graph size. Adding 10× more nodes and edges that are unreachable from the query start node has zero impact.
  • Relational DB: The B-tree index depth grows as O(log N) of the total table size. Doubling the edges table makes every lookup slightly slower, even for isolated queries.

This is the property that makes graph databases scale well for social-network-style queries even as the overall graph grows.

Where Graph DBs Do Not Scale Well

  • Global aggregations — "count all triangles," "compute PageRank for every node" — must visit the whole graph. Graph DBs have no special advantage here; dedicated graph analytics systems (Pregel, GraphX) or columnar engines are more efficient.
  • High-write, high-concurrency workloads — updating the doubly-linked relationship chains requires careful locking. Wide transactions that modify many relationship chains can create contention hot-spots around heavily-connected nodes ("supernode problem").
  • Very large properties — if nodes carry large blobs (images, full text), the property store becomes the bottleneck, negating the traversal advantage.

The Supernode Problem

A "supernode" is a node with very high degree (e.g., a celebrity with 50 million followers). Traversing through that node means iterating a 50-million-entry linked list to find any specific neighbour. Graph DBs handle this poorly without special indexing on relationship types. Relational DBs with a composite index (from_user, to_user) can seek directly to a specific neighbour in O(log N).

Summary Performance Table

Scenario Best tool Why
1–2 hop lookup, small graph Either Index join vs pointer — negligible diff
3+ hop traversal, medium graph Graph DB O(b^k) vs O(k · log N · b^k)
Shortest path, variable depth Graph DB BFS on adjacency pointers
Full-graph aggregation Column store / analytics Graph DBs have no structural advantage
High write throughput, simple schema Relational Simpler locking, well-optimised bulk load
Supernode traversal (degree > millions) Relational with composite index Direct seek vs linked-list scan
Schema-flexible relationship types Graph DB No migration needed to add edge labels

Key Takeaways

  • Index-free adjacency stores relationships as physical pointers, making each hop O(1) rather than O(log N) — the fundamental performance difference.
  • Local traversal time in a graph DB depends on the neighbourhood visited, not total graph size — this is the scaling property that matters most.
  • Write performance for adding edges is roughly O(1) in a graph DB (3–4 pointer updates); relational junction tables carry B-tree rebalancing overhead as they fill.
  • Global queries (aggregations, full scans) are not faster in a graph DB — use columnar or analytics engines for those.
  • The supernode problem is a real weakness: very high-degree nodes degrade traversal performance and create write contention.
  • Choose a graph DB when your workload is dominated by multi-hop traversal of connected data at moderate depth — not when you need global analytics or simple flat lookups.
